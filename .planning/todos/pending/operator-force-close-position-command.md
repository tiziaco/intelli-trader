# Operator "force-close position" command — targeted manual close (consumer of ForceCloseEvent)

**Origin:** Brainstorming session 2026-07-13 that produced
[[docs/superpowers/specs/2026-07-13-force-close-event-design]] (the `ForceCloseEvent` primitive +
honest strategy attribution). The user flagged an intent to add an operator-issued **force-close**
capability "soon, manually generated by a human." That work is deliberately **out of scope** of the
primitive spec — this todo captures the targeted single-position case as a thin consumer to build
**on top of** the primitive once it lands.

## What this is (and is NOT)

A human/operator command to force-close **one specific open position** on demand:

- Operator selects a `(portfolio_id, ticker)` target → the system emits a **single**
  `ForceCloseEvent(origin=ForceCloseOrigin.OPERATOR)`.
- Side + size are derived at handle-time from the live position (spec decision **FC-05**) — the
  command does not carry them.
- **No fan-out, no halt.** This is the single-position case.

It is **distinct from** the emergency-shutdown command
([[operator-emergency-shutdown-command]] — flatten ALL positions + latch `HALTED`). This targeted
command is the **simpler precursor**: same primitive, one event instead of a fan-out, no lifecycle
freeze.

## Why it's a clean fit on the primitive

Once the spec's `ForceCloseEvent` + `OrderHandler.on_force_close` land, this command needs almost no
new order machinery — it just needs an **ingress** that constructs one `ForceCloseEvent`. The honest
attribution (no `strategy_id`, `OrderTriggerSource.OPERATOR_FORCE_CLOSE` on the order state change)
comes for free from the primitive. Bypasses admission adjudication but still gets a real venue fill
(spec **FC-04**), and is a clean no-op if the target is already flat at handle-time (**FC-05**).

## Open questions to settle before building

1. **Ingress shape** — reuse the D-10 `STRATEGY_COMMAND` external-ingress path (fail-closed
   default-deny admits externally-originated `SIGNAL` + `STRATEGY_COMMAND`) vs. a dedicated
   `LiveTradingSystem` method (e.g. `force_close_position(portfolio_id, ticker, fraction=1)`).
   Thread-safety: connector-loop callbacks only flip flags; blocking venue I/O runs on the engine
   thread.
2. **Target addressing & validation** — `(portfolio_id, ticker)`; validate the portfolio is active
   and the position exists (else surface a clear "no such open position" result, not a silent drop).
3. **Operator-visible result / ack** — accepted / no-op-if-flat / venue-rejected (e.g. venue
   min-notional on a dust position). What does the operator see?
4. **Authorization / guard** — who may issue it; any confirmation step for a destructive action.
5. **Full vs partial exit** — full-only (`exit_fraction=1`) or allow `exit_fraction ∈ (0, 1]` for a
   partial reduce.

## When to schedule

After the `ForceCloseEvent` primitive lands and locks (event + route + `OrderHandler.on_force_close`
+ optional `strategy_id` migration + `OrderTriggerSource` members). Natural fit alongside any live
operator control-plane work.
