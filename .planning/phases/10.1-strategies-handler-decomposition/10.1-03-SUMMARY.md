## Roster-state isolation proof (CRITICAL correction 1)

The dangerous repair — giving `StrategyLifecycleManager` its own `_pending_removals` —
was **not** taken. Required proof:

```
$ grep -nE 'self\._pending_removals|self\.strategies' itrader/strategy_handler/lifecycle/manager.py
(no output)
```

Empty. The manager owns no roster state; every access routes through `self._managed`.

An intermediate run of this grep returned **3 hits, all docstring prose** (e.g. "the
instance STAYS in ``self.strategies``") — the same docstring-blindness Wave 2 documented
for its own gate. I confirmed via AST that there were never any code references, then
repointed those three prose fragments to "the MANAGED ROSTER" so the docstrings are
accurate *and* the literal grep passes:

```
AST code hits for self.strategies / self._pending_removals: NONE
```

**The three sites handed to this wave, and how each was resolved:**

| Site | Original | Now |
|---|---|---|
| D-02 duplicate-name reject (`_add_strategy_verb`) | `for existing in self.strategies` | `for existing in self._managed.strategies` |
| `on_fill` fast-exit | `if not self._pending_removals:` | `if not self._managed.has_pending():` |
| `on_fill` iteration | `for name in list(self._pending_removals):` | `for name in self._managed.pending_names():` |

`has_pending()` and `pending_names()` are **new accessors on `ManagedStrategies`**, not a
copied set. `pending_names()` returns `list(self._pending_removals)` — reproducing the
original body's explicit snapshot, which exists because the loop calls `discard_pending`
as it completes removals and iterating the live set would raise. Both carry a docstring
recording *why* a second set would be catastrophic (membership never re-derives → the
D-11 force-close poll never fires → `remove` hangs forever, silently).
