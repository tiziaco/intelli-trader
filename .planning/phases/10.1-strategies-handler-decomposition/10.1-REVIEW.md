# Phase 10.1: Code Review Report

**Reviewed:** 2026-07-20T10:56:06Z
**Depth:** standard
**Files Reviewed:** 13
**Status:** issues_found

## Summary

The decomposition itself is structurally sound. I verified the three high-risk claims
independently and they hold:

- **Single roster owner.** `StrategyLifecycleManager` declares no `strategies` list and no
  `_pending_removals` set; every access routes through `self._managed`
  (`manager.py:372, 441, 509, 517, 565, 569, 570, 597-606, 788, 910`). No second source of truth.
- **Flag delegation is real.** `_allow_short_selling` / `_enable_margin` are read/write
  properties forwarding to `ManagedStrategies` (`strategies_handler.py:199-215`); there is no
  shadow attribute. Grep across `itrader/` + `tests/` found **zero** external assignment sites
  to `.strategies`, `.min_timeframe`, `._pending_removals`, or `strategies_handler._universe`,
  so the read-only-property conversion breaks no caller.
- **Import hoisting is genuinely inert.** I ran the check in a clean interpreter:
  importing `itrader.strategy_handler.strategies_handler` (which now transitively pulls
  `lifecycle.manager` → `registry.*`, `core.policy_codec`, `feed.cache_registration`) leaks
  **zero** `sqlalchemy` / `psycopg2` / `alembic` / `ccxt` modules. GATE-01 holds. Indentation is
  clean per file (manager/managed/handler are 100% tabs; the storage factory is 100% spaces).
- 342 tests green across `tests/unit/strategy`, `test_compose_wiring.py`, `test_okx_inertness.py`.

The defects are elsewhere. The one that matters is an **exception-tuple asymmetry in the
`add` verb** that lets an untrusted external payload raise out of a documented never-raise
boundary and, through the live failure-rate tripwire, latch a HALT. The remaining findings
cluster around the seams the split created: an un-encapsulated roster mutator, `Any`-typed
dependencies that erase real checking under `mypy --strict`, a silent-degradation arm in the
new factory, and doc pointers the rename invalidated.

## Critical Issues

### CR-01: `add` verb propagates bare `ValueError` into the live queue — untrusted-payload HALT vector

**File:** `itrader/strategy_handler/lifecycle/manager.py:391-405`

**Issue:** `_add_strategy_verb` wraps `build_strategy(...)` in a catch tuple that omits
`ValueError`:

```python
except (
    UnknownStrategyTypeError,
    StrategyConfigError,
    UnknownParamError,
    MissingParamError,
) as exc:
```

`build_strategy` deliberately propagates (`registry/rehydrate.py:215-221` — "Errors PROPAGATE
deliberately"), and its `cls(**params)` runs `_apply_params` → `validate()`, **both of which
raise bare `ValueError`**:

- `base.py:292` — `raise ValueError("tickers must be a non-empty list[str] ...")`
- `strategies/SMA_MACD_strategy.py:42` — `raise ValueError("short_window must be < long_window")`

Neither is a subclass of the four caught types (`UnknownParamError`/`MissingParamError` derive
from `ValidationError`/`ITraderError`, not `ValueError`). So a payload like
`{"strategy_type": "SMA_MACD", "tickers": []}` or `{"short_window": 50, "long_window": 10}`
decodes cleanly and then raises straight out of `_add_strategy_verb` → `on_strategy_command` →
the live `EventHandler`.

This is **the asymmetry, not a design choice**: the other two construction/apply sites in the
same file *do* include `ValueError`, with an explicit comment naming exactly this hazard —
`manager.py:764-773` ("ValueError covers validate() + the _apply_params tickers/enum guards")
and `manager.py:824-829`. Only the `add` path was left behind.

**Failure mode.** `STRATEGY_COMMAND` is admitted from external ingress (`LiveTradingSystem.add_event`,
D-10). The raise violates the method's own contract — `manager.py:318-320`: *"Every rejection
below is a LOUD no-op (`logger.warning` + return) that registers and persists NOTHING"*, and
`manager.py:885-887`: *"NEVER a raise into the queue"*. Worse, in live mode the uncaught handler
exception reaches `ErrorPolicy.on_handler_error` → `record_failure` → the failure-rate tripwire
(`error_policy.py:243-259`), which calls `self._halt(reason.value)`. Per CLAUDE.md, `HALTED` has
**no legal exit except operator `reset_halt()`**. A handful of malformed `add` payloads — routine
bad operator input, not an attack — can therefore latch live trading into HALT. Backtest is
unaffected (STRATEGY_COMMAND routes to an empty list there).

**Fix:** add `ValueError` to the tuple, matching the two sibling sites verbatim.

```python
		try:
			strategy = build_strategy(rec, catalog=self.strategy_catalog)
		except (
			UnknownStrategyTypeError,
			StrategyConfigError,
			UnknownParamError,
			MissingParamError,
			# ValueError covers validate()'s cross-field rules and the _apply_params
			# tickers/enum guards — both reach here through build_strategy's deliberate
			# propagate. Matches the _reconfigure_strategy_verb tuples below; without it
			# an external add payload raises into the queue and feeds the failure-rate
			# tripwire (never-raise contract, D-10).
			ValueError,
		) as exc:
```

## Warnings

### WR-01: `ManagedStrategies.remove()` is a public mutator that leaves derived state stale

**File:** `itrader/strategy_handler/managed_strategies.py:123-131`, `itrader/strategy_handler/lifecycle/manager.py:563-570`

**Issue:** Before the split, the roster drop was an inline two-liner inside
`_try_complete_removal`, physically adjacent to the `_pending_removals.discard` and
`_recompute_min_timeframe()` tail that restores the invariant. The extraction promoted the drop
to a standalone public method on `ManagedStrategies` while leaving the other two halves of the
completion sequence on the *other* object:

```python
self._managed.remove(strategy)          # roster mutated
if self.registry_store is not None:
    self.registry_store.delete(...)     # <- can raise
self._managed.discard_pending(...)      # never runs
self._managed.recompute_min_timeframe() # never runs
```

Two concrete consequences. (a) A store fault at `delete` leaves the strategy dropped from the
roster but still in `_pending_removals` and with `min_timeframe` stale — the object is gone, yet
`get_universe()` still filters on a name nothing will ever discard, and `on_fill` re-enters
`_try_complete_removal` on a strategy `by_name()` can no longer resolve, taking the
"already dropped — clear it defensively" arm each cycle. The system self-heals in that specific
path, but only by accident of the defensive branch. (b) `remove()` now advertises itself as a
safe standalone API (its docstring documents only the same-object invariant) while silently
violating the recompute invariant that `recompute_min_timeframe`'s own docstring calls out.

**Fix:** either fold `discard_pending` + `recompute_min_timeframe` into `ManagedStrategies.remove()`
so the invariant is encapsulated with the state that owns it, or document on `remove()` that the
caller must run the recompute/discard tail. Ordering the store `delete` *before* the roster
mutation would also make the sequence fail-safe.

### WR-02: `StrategyRegistryStorageFactory` degrades silently on the unwired-spine arm

**File:** `itrader/strategy_handler/storage/registry_storage_factory.py:77-79`

**Issue:** The factory has two ways to return `None` on the `'live'` arm, and only one of them
is observable:

```python
elif environment == 'live':
    if sql_engine is None:
        return None          # <- no log at all
    ...
    if not _sa_inspect(...).has_table("strategy_registry"):
        logger.warning(...)  # <- D-21 path is loud
        return None
```

A `None` registry store makes **every** persist arm in the control plane a clean no-op
(`_persist_strategy` returns immediately at `manager.py:225-226`, the child-row writes are gated
at `manager.py:456` and `1087`, and `build_live_system` skips rehydrate entirely at
`live_trading_system.py:1624`). So `environment='live'` with no spine yields a live engine where
every `enable`/`disable`/`subscribe`/`add`/`reconfigure` applies to memory and vanishes on
restart — with **zero** diagnostic output. The D-21 arm was correctly kept loud precisely because
that outcome needs to be visible; this arm gets the same outcome for free.

Today's wiring makes it unreachable (`build_live_system` sets `environment='backtest'` when the
credential probe fails, `live_trading_system.py:1351`), but the factory is now the single owner
of this decision and is a public seam — the invariant should live in the factory, not in one
caller's coincidental gating.

**Fix:**

```python
        if sql_engine is None:
            get_itrader_logger().bind(
                component="StrategyRegistryStorageFactory").warning(
                "environment='live' but no SQL spine was wired — the strategy registry is "
                "DISABLED: every STRATEGY_COMMAND verb will apply in memory only and be "
                "lost on restart")
            return None
```

### WR-03: `logger: Any` erases the typed logger across 1400 lines of moved code

**File:** `itrader/strategy_handler/managed_strategies.py:77`, `itrader/strategy_handler/lifecycle/manager.py:158,190`

**Issue:** Both new collaborators declare `logger: Any` and store the result of `logger.bind(...)`,
which is also `Any`. `get_itrader_logger()` has a real return type — `ITraderStructLogger`
(`itrader/logger.py:321`) — and the handler's own `self.logger` is correctly typed by inference.
By accepting `Any` at the constructor boundary, the split converted a checked dependency into an
unchecked one: all ~30 `self.logger.warning(...)` / `.error(...)` call sites in `manager.py` and
the `add_strategy` info log are now invisible to `mypy --strict`. A typo in a method name or a
wrong argument arity there typechecks clean and fails at runtime — on the live control plane,
inside a `warning` call whose whole purpose is to report a rejection.

This is exactly the kind of `Any` escape hatch introduced to make a split typecheck. Unlike the
three live deps, there is no GATE-01 justification: `itrader.logger` pulls no SQL and is already
on the handler's import graph.

**Fix:** import the concrete type and annotate it.

```python
from itrader.logger import ITraderStructLogger
...
		logger: ITraderStructLogger,
```

### WR-04: `portfolio_read_model: Optional[Any]` masks a real `PortfolioId | int` type mismatch

**File:** `itrader/strategy_handler/lifecycle/manager.py:157,189,544-550`

**Issue:** The three-`Any` block is justified in the docstrings as GATE-01 inertness ("so the SQL
stack stays off this module's annotations"). That rationale is sound for `registry_store` and
`strategy_catalog`. It does **not** apply to `portfolio_read_model`: `PortfolioReadModel` is a
`Protocol` in `itrader/core/portfolio_read_model.py:84`, and `core/` depends on nothing inside
`itrader` (CLAUDE.md) — it pulls no SQL and could be imported under `TYPE_CHECKING` at zero
runtime cost.

The `Any` is not cosmetic. It hides a live type disagreement in `_strategy_is_flat`:

```python
for portfolio_id in strategy.subscribed_portfolios:   # list[PortfolioId | int]
    for ticker in strategy.tickers:
        if read_model.get_position(portfolio_id, ticker) is not None:
```

`PortfolioReadModel.get_position` is declared `get_position(self, portfolio_id: PortfolioId, ticker: str)`
(`core/portfolio_read_model.py:112`) — it does **not** accept the `int` arm of the union. Under a
real annotation `mypy --strict` would flag this; under `Any` it is invisible. This is the same
class of defect `_portfolio_id_from`'s own docstring (`manager.py:284-292`) warns about at length
— "a bare `str` sails through that cast unchallenged" — reproduced one method below it.

**Fix:**

```python
if TYPE_CHECKING:
    from itrader.core.portfolio_read_model import PortfolioReadModel
...
		portfolio_read_model: "Optional[PortfolioReadModel]",
```
then resolve whichever side of the `PortfolioId | int` union is actually wrong.

### WR-06: `min_timeframe` docstrings state a consumer that does not exist

**File:** `itrader/strategy_handler/managed_strategies.py:233-238`, `itrader/strategy_handler/strategies_handler.py:699-705`

**Issue:** Both `add_strategy` docstrings (the moved copy and the delegator) assert:

> "At the same time, calculate the minimum timeframe among the different strategies to be traded.
> **This timeframe will be used from the price handler to download historical prices**"

A grep for `min_timeframe` across `itrader/` finds **no production reader** of
`StrategiesHandler.min_timeframe` / `ManagedStrategies.min_timeframe` — the only non-strategy hit
is `screeners_handler.py:113`, which maintains its own unrelated attribute. The price handler
does not consume it; the live feed sizes its ring from `required_history_depth`
(`cache_registration.py`), not from this value. The single reader is
`tests/unit/strategy/test_strategy_command_verbs.py:997`.

The phase reproduced this false claim verbatim into a new module and additionally built a
delegating property (`strategies_handler.py:181-184`) plus a public `recompute_min_timeframe()`
around it, giving dead derived state the appearance of a maintained invariant. This is the kind of
stale decision-anchored comment the phase brief asks to flag.

**Fix:** correct the docstrings to state that `min_timeframe` is currently a derived read-model
value with no production consumer, or delete the field and its three write sites if it is genuinely
unused. Note it is also *not* recomputed after `reconfigure` mutates `strategy.timeframe`
(`manager.py:823` — `timeframe` is absent from both `_RECONFIGURE_IMMUTABLE` and
`_RECONFIGURE_VERB_ONLY`), so the value is already stale on that path — harmless only because
nothing reads it.

## Info

### IN-01: Handler reaches into collaborator privates despite a public API existing

**File:** `itrader/strategy_handler/strategies_handler.py:189,202,206,211,215`

**Issue:** Five accessors read/write `self._managed._pending_removals`,
`self._managed._allow_short_selling`, `self._managed._enable_margin` — private names on another
object. `ManagedStrategies` already exposes public equivalents (`is_pending`, `mark_pending`,
`discard_pending`, `has_pending`, `direction_admissible`). The `_`-prefixed handler property names
are constrained by the 11 test files that flip them, but the *collaborator* side need not stay
private.

**Fix:** rename the two flags to public `allow_short_selling` / `enable_margin` on
`ManagedStrategies` and have the handler's `_`-prefixed properties forward to those, so the
private-reach disappears without touching a single test.

**Issue:** `#self.portfolios: dict = {}` survives in the middle of the rewritten constructor.

**Fix:** delete it.

### IN-03: Four constructor parameters are undocumented

**File:** `itrader/strategy_handler/strategies_handler.py:45-96`

**Issue:** The `__init__` docstring documents `global_queue`, `feed`, `signal_store`,
`allow_short_selling`, `enable_margin`, `registry_store` — but not `environment`, `sql_engine`,
`strategy_catalog`, or `portfolio_read_model`. Two of those (`strategy_catalog`,
`portfolio_read_model`) are new public parameters this phase added and are security-relevant
(`strategy_catalog` is the D-10 allowlist). The rationale exists — but only in a comment block
30 lines below (`:217-237`), not in the docstring a caller reads.

**Fix:** add the four `Parameters` entries, citing the D-10/D-11 roles for the two seams.

### IN-04: `_universe` is duplicated across handler and manager, synced by one method

**File:** `itrader/strategy_handler/strategies_handler.py:162,288`, `itrader/strategy_handler/lifecycle/manager.py:193,202`

**Issue:** Two independent mutable fields hold the same object, kept consistent only by
`StrategiesHandler.set_universe` writing both. It is correct today (no other assignment site
exists — verified by grep), and the docstrings call it "two references to ONE object". But the
manager already holds the handler's other injected deps by single-ownership + read-through
properties; `_universe` is the one dep that got copied instead. A future `set_universe` override
or a direct `manager._universe` write would desync `_request_rewarm` (which would then
short-circuit at `manager.py:265-266` and silently never re-warm).

**Fix:** for consistency with the three live deps, give the handler a read-through property over
`self._lifecycle._universe` rather than its own field.

### IN-05: Redundant members in `except` tuples

**File:** `itrader/strategy_handler/lifecycle/manager.py:764-770`, `itrader/strategy_handler/lifecycle/manager.py:824-829`

**Issue:** `StrategyConfigError` (`registry/config_codec.py:113`) and `UnknownStrategyTypeError`
(`registry/catalog.py:44`) both subclass `ValueError`, which is already in both tuples. Harmless,
but it obscures which exceptions are genuinely distinct — and the fact that the *first* tuple
(`:393`, see CR-01) is missing `ValueError` is much harder to spot when the other two list
subsumed types alongside it.

**Fix:** list `ValueError` plus only the non-`ValueError` arms (`UnknownParamError`,
`MissingParamError`), with a comment naming what `ValueError` subsumes.

### IN-06: Stale inertness claim in a neighbouring barrel, acknowledged but not fixed

**File:** `itrader/strategy_handler/lifecycle/__init__.py:10-12`

**Issue:** The new barrel documents that `registry/__init__.py`'s SQL-inertness claim is
"(now-stale)" — correctly, since 10.1-03 proved the registry imports leak no SQL. It then leaves
the stale docstring in place in the other file. A reader who opens `registry/__init__.py` directly
gets the false claim with no pointer to the correction.

**Fix:** update `registry/__init__.py`'s docstring in the same spirit, rather than annotating its
staleness from a different module.
