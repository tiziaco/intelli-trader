## Summary

The locked spec (`docs/superpowers/specs/2026-07-18-strategies-handler-decomposition-design.md`)
is **structurally sound and its design decisions hold**. The three-unit split, the naming, the
sequencing, and the risk analysis all survive contact with the code.

However, **five factual claims in the spec are wrong or materially incomplete**, and two of them
would produce a broken implementation if a planner cited them verbatim:

1. **BLOCKING — the live deps are injected POST-CONSTRUCTION, not through the constructor.**
   `live_trading_system.py:1630/1641/1642` assign `registry_store` / `strategy_catalog` /
   `portfolio_read_model` onto an already-built handler. At `__init__` time they are `None` on
   **both** paths. The spec's "when live deps are present, construct `self._lifecycle`" in
   `__init__` would therefore yield `self._lifecycle is None` **always**, including live — every
   verb would then hit the `assert` and the live control plane would be dead. This is the single
   most important finding in this document.
2. **BLOCKING — `set_universe` IS called on the backtest path.** `universe_wiring.py:109`
   (reached from `backtest_runner._initialise_backtest_session`) calls
   `engine.strategies_handler.set_universe(universe)`. `self._universe` is **not** `None` in
   backtest. The handler's own docstring at `strategies_handler.py:182-190` claiming otherwise is
   **stale**. A `set_universe` delegator carrying `assert self._lifecycle is not None` would abort
   every backtest run.
3. **The five "load-bearing" GATE-01 lazy imports are NOT load-bearing at HEAD.** A clean-interpreter
   probe importing all five targets leaks **zero** forbidden modules and **zero** SQLAlchemy. The
   inertness rationale in the code comments is stale. This does not change the plan — moving them
   is still correct — but it removes the risk the spec attributes to Wave 2, and DECOMP-02's
   rationale needs restating.
4. **`order_handler/lifecycle/` IS barrelled** (its own `__init__.py` re-exports `LifecycleManager`).
   The spec's "never barrelled" requirement needs disambiguation against the precedent it cites.
5. **The "312 test field-references" figure is wrong.** The measured count is **185** occurrences
   (113 handler-scoped). The 59 `calculate_signals` figure is correct as a *line* count.

Everything else in the spec verified clean, including the 1648 line count, the 13-method move
list (complete — no method is missed), `full_event_handler.py:95` as the sole route site, and
`compose.py:227` as the sole production construction site.

**Primary recommendation:** Plan the three waves exactly as locked, but replace the spec's
`__init__`-time lifecycle construction with a **lazy/late construction seam** (see
[Finding D-1](#d-1-blocking--live-deps-are-post-construction-injected)), and drop the `assert`
from `set_universe` (see [Finding D-2](#d-2-blocking--set_universe-runs-on-the-backtest-path)).

## Detailed Findings

### D-1 (BLOCKING) — Live deps are post-construction injected

The spec, §"Unit 3", says:

> Constructs `self._managed = ManagedStrategies(...)` and, when live deps are present,
> `self._lifecycle = StrategyLifecycleManager(self._managed, ...)`. On the backtest path the
> live deps are absent, so `self._lifecycle` is `None` (never constructed).

**This premise is false.** The backtest construction site passes no live deps:

```python
# itrader/trading_system/compose.py:227-232  (TABS)
strategies_handler = StrategiesHandler(
	ctx.bus, feed,
	allow_short_selling=trading_rules.allow_short_selling,
	enable_margin=trading_rules.enable_margin,
	environment=ctx.environment,
	sql_engine=ctx.sql_engine)
```

`compose_engine` is the **shared, mode-agnostic** composition seam — the live path calls the same
function. The live deps arrive **afterwards**, by attribute assignment on the built handler:

```python
# itrader/trading_system/live_trading_system.py:1630, 1641-1642  (4-SPACE)
engine.strategies_handler.registry_store = strategy_registry_store
engine.strategies_handler.strategy_catalog = strategy_catalog
engine.strategies_handler.portfolio_read_model = portfolio_handler
```

Note these assignments sit **inside** an `else:` arm gated on
`_sa_inspect(...).has_table("strategy_registry")` (line 1615) — so even on the live path they can
be skipped entirely when the Alembic chain has not been applied (the D-21 first-start state).

**Consequences for the plan:**

- A `__init__`-time `if registry_store is not None: self._lifecycle = ...` yields `None` on every
  path. The live control plane would be entirely dead and the `assert` would fire on the first verb.
- The three attributes are **public and externally mutable**. `StrategyLifecycleManager` cannot
  capture them by value at construction without going stale.

**Recommended contract for the planner (Wave 2):**

Construct the lifecycle manager **lazily, on first use**, from the handler's current attribute
values — or, equivalently, inject the *handler* reference for those three fields. Concretely:

```python
def _ensure_lifecycle(self) -> "StrategyLifecycleManager":
	"""Late-construct the live-only control plane (D-1: live deps are
	POST-CONSTRUCTION injected at live_trading_system.py:1630/1641/1642,
	so __init__ cannot see them). Import is function-local ONLY to keep the
	module off the backtest import graph — the same discipline the v1.7 live
	stack uses. Never reached on the backtest path (STRATEGY_COMMAND and the
	strategies FILL route are live-only, route_registrar.py:107/127)."""
	if self._lifecycle is None:
		from itrader.strategy_handler.lifecycle.manager import StrategyLifecycleManager
		self._lifecycle = StrategyLifecycleManager(
			managed=self._managed,
			global_queue=self.global_queue,
			feed=self.feed,
			registry_store=self.registry_store,
			strategy_catalog=self.strategy_catalog,
			portfolio_read_model=self.portfolio_read_model,
			logger=self.logger,
		)
		self._lifecycle.set_universe(self._universe)
	return self._lifecycle
```

⚠ **This re-introduces exactly one function-local import** — the `StrategyLifecycleManager` import
itself. That is the *correct* place for it (it is the GATE-01 seam, and it mirrors
`LiveTradingSystem.__init__`'s lazy live-stack imports). It does **not** undermine DECOMP-02: the
five registry/codec/catalog imports still collapse to module top **inside the lifecycle module**.
The planner should state this explicitly so the DECOMP-02 verification does not read the surviving
import as a failure.

**Alternative the planner may prefer** (equally valid, note both): keep the three live deps as
`@property` reads that forward to the handler, so `StrategyLifecycleManager` holds a narrow
`_deps` reference rather than three snapshotted values. This avoids the staleness question
entirely. Either is acceptable; the lazy construction is simpler and matches the precedent.

### D-2 (BLOCKING) — `set_universe` runs on the backtest path

The handler docstring says:

```python
# itrader/strategy_handler/strategies_handler.py:182-190
def set_universe(self, universe: "Universe") -> None:
	"""Wire the live dynamic universe for the WR-02 readiness gate (D-01).

	Live-only seam (mirrors the inert-by-default pattern): the backtest
	composition root never calls this, so ``self._universe`` stays ``None`` …
```

**This docstring is stale.** The backtest reaches it:

```python
# itrader/trading_system/universe_wiring.py:109  (TABS)
engine.strategies_handler.set_universe(universe)
```

`wire_universe(engine)` is called from `backtest_runner._initialise_backtest_session`
(`backtest_runner.py:68` — its own comment says *"wire_universe ADDS strategies_handler.set_universe
to this path; it is oracle-inert BY CONSTRUCTION"*). So in backtest, `self._universe` holds a real
`Universe` whose members are all `Readiness.READY` — the gate never skips, which is why the oracle
is unaffected.

**Consequences for the plan:**

- `set_universe` must **not** carry `assert self._lifecycle is not None`. Every backtest run would abort.
- The correct forwarding contract:

```python
def set_universe(self, universe: "Universe") -> None:
	self._universe = universe
	# D-2: reached on the BACKTEST path too (universe_wiring.py:109) — the
	# lifecycle manager is absent there, so forward only when it exists.
	# _ensure_lifecycle also replays the current _universe at construction,
	# covering the set_universe-then-first-verb ordering.
	if self._lifecycle is not None:
		self._lifecycle.set_universe(universe)
```

- Because `_ensure_lifecycle` may run **after** `set_universe`, the late construction must replay
  `self._universe` into the new manager (shown in [D-1](#d-1-blocking--live-deps-are-post-construction-injected)).
  Missing this would leave `_request_rewarm` a silent no-op — the exact class of
  silent-permanent-no-warm defect this codebase treats as a correctness bug.
- **The stale docstring should be corrected in Wave 1** (it is load-bearing misinformation that has
  already misled one spec). Track it as an explicit plan task.

### D-3 — The 1041-1042 imports are NOT already at module top

The spec claims the `ErrorSeverity` / `ErrorEvent` pair is "not [load-bearing] — both modules are
already imported at module top and pull no SQL."

**The second half is true; the first half is false.** Module top imports are:

```python
from itrader.core.enums import OrderType                      # line 5  — NOT ErrorSeverity
from itrader.events_handler.events import (                    # lines 16-22
	BarEvent, BarsLoaded, SignalEvent, StrategyCommandEvent, UniversePollEvent,
)                                                              # — NOT ErrorEvent
```

Neither name is at module top. The *modules* are on the import graph (so the imports are cheap and
carry no inertness weight), but the **names** are genuinely function-local.

**Consequence:** this is not a delete-only change. When `_emit_reconfigure_apply_failure` moves to
the lifecycle module, `ErrorSeverity` and `ErrorEvent` must be **added to that module's top
imports**. Do not plan a bare deletion.

### D-5 — The precedent IS barrelled; disambiguate "never barrelled"

```python
# itrader/order_handler/lifecycle/__init__.py  (no indented lines; 4-space file per convention)
"""
Lifecycle subdomain package.

Re-exports the LifecycleManager … so consumer import paths stay short after the
order-manager decomposition (pure move, D-12/D-13). It is NOT added to the
order_handler top barrel (D-12): it is an OrderManager implementation detail.
"""

from .lifecycle_manager import LifecycleManager

__all__ = ["LifecycleManager"]
```

All four collaborator subdirs (`admission/`, `brackets/`, `lifecycle/`, `reconcile/`) follow this
identical pattern: **subpackage `__init__.py` re-exports; top-level barrel does not.**

`itrader/strategy_handler/__init__.py` is currently **empty** (zero bytes), so there is no top
barrel to pollute. `itrader/strategy_handler/registry/__init__.py` follows the same
subpackage-barrel pattern with an explicit "NOT added to the `strategy_handler` top barrel" note.

**Interpretation the planner should adopt:** "never barrelled" means **never re-exported from
`itrader/strategy_handler/__init__.py`**. A `strategy_handler/lifecycle/__init__.py` that re-exports
`StrategyLifecycleManager` is *consistent with the cited precedent* and is inertness-safe, because
nothing on the backtest graph imports `itrader.strategy_handler.lifecycle` at all. The planner
should mirror the precedent (including the docstring style with its decision-tag citation) and say
so explicitly, rather than inventing a barrel-less variant that breaks the house pattern.

## Common Pitfalls

### Pitfall 1: Constructing the lifecycle manager in `__init__`
**What goes wrong:** `self._lifecycle` is `None` on every path; every live verb hits the `assert`.
**Why:** live deps are post-construction injected (D-1).
**Avoid:** late construction via `_ensure_lifecycle()`.
**Warning sign:** `tests/unit/strategy/test_strategy_command_verbs.py` fails with `AssertionError`
— note those tests construct the handler *then* set `registry_store`, mirroring live exactly.

### Pitfall 2: `assert self._lifecycle is not None` inside `set_universe`
**What goes wrong:** every backtest run aborts.
**Why:** `universe_wiring.py:109` calls it on the backtest path (D-2).
**Warning sign:** the oracle test fails immediately, not subtly.

### Pitfall 3: Deleting the 1041-1042 imports without re-adding them
**What goes wrong:** `NameError: ErrorSeverity` on the reconfigure-apply-failure path.
**Why:** neither name is at module top (D-3). `mypy` **will** catch it (strict scope); the verb
suite may not, if no test drives the apply-failure branch. Check
`tests/unit/strategy/test_reconfigure_atomic.py` coverage before relying on the suite alone.

### Pitfall 4: Leaving the module-level verb constants behind
**What goes wrong:** `NameError` at first verb dispatch.
**Why:** `_PAIR_REFUSED_VERBS` (38), `_POLL_FOLLOW_ON_VERBS` (45), `_RECONFIGURE_IMMUTABLE` (66),
`_RECONFIGURE_VERB_ONLY` (69) are read only by moving methods. The spec omits them entirely.

### Pitfall 5: `_universe` not replayed into a late-constructed lifecycle manager
**What goes wrong:** `_request_rewarm` silently no-ops → strategies stay dark forever after
enable/reconfigure. **Silent**, no exception.
**Avoid:** `_ensure_lifecycle` must call `self._lifecycle.set_universe(self._universe)`.

### Pitfall 6: Trusting the stale docstrings
`strategies_handler.py:182-190` (set_universe "live-only"), the five lazy-import GATE-01 comments,
and `registry/__init__.py`'s "would pull SQL" claim are all **stale**. This codebase's docstrings
carry decision tags and read as authoritative — they are not, here. **Code wins.**

### Pitfall 7: Using `make test` as the gate
`make test` exports `ITRADER_DISABLE_LOGS=true`, which breaks `caplog` assertions, and aborts in
worktrees on a missing `.env`. **Use `poetry run pytest`** (what every result in this document used).
