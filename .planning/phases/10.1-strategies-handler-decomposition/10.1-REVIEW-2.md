## Warnings

### WR2-01: `add` verb silently discards a malformed `portfolio_id` — no warning, no subscription

**File:** `itrader/strategy_handler/lifecycle/manager.py:485-491` (with `:282-314`)

**Issue.** `_portfolio_id_from` collapses four distinct outcomes into a single `None`:
config-not-a-dict, key absent, value not a non-empty `str`, and **UUID parse failure**. The
two light verbs handle that ambiguity by warning unconditionally:

```python
portfolio_id = self._portfolio_id_from(event)
if portfolio_id is None:
    self.logger.warning(
        'subscribe_portfolio for strategy %s carries no valid '
        'config["portfolio_id"] — ignored', event.strategy_name)
    return
```

The `add` verb does not:

```python
portfolio_id = self._portfolio_id_from(event)
if portfolio_id is not None:
    strategy.subscribe_portfolio(portfolio_id)
```

The comment above it justifies this as *"Absent -> the strategy computes but fans out to
nobody (a legal state, D-09)"* — which is true for **absent** and false for **malformed**.
There is no branch that can tell them apart.

**Why this is new.** The `owe` remediation removed `_portfolio_id_from`'s `int` fallback
(`return int(raw)`), which is correct — but it **widened the set of payloads that reach this
silent arm**. Before the change, `{"portfolio_id": "7"}` parsed to `7` and produced a
subscription; now it, and every other non-UUID string, lands in the unreported `None` arm.

**Concrete failure scenario.** An operator POSTs
`{"verb":"add","strategy_name":"sma_1","config":{"strategy_type":"SMA_MACD","portfolio_id":"7","tickers":["BTCUSD"],...}}`
against a live engine. Result: the strategy is constructed, registered in the roster, and
**persisted to `strategy_registry`** — but zero `strategy_portfolio_subscriptions` rows are
written and `subscribed_portfolios` is `[]`. The command returns clean, the log is silent,
and the `UniversePollEvent` fires so the symbol warms normally. The strategy then runs
forever computing signals and fanning them to nobody: `_emit_intent`'s
`for portfolio_id in strategy.subscribed_portfolios` iterates an empty list, so **no
`SignalEvent` is ever enqueued** and no order is ever placed. This is precisely the
"healthy-looking engine that trades nothing" outcome D-19 rates as worse than failing loudly.
The same payload sent as `subscribe_portfolio` *does* warn — so the diagnosis depends on
which verb the operator happened to use.

**Fix.** Distinguish absent from malformed, and warn on the latter on the `add` path too.
Minimal form:

```python
		# WR2-01: a MALFORMED portfolio_id is operator error, not the legal
		# "no subscription requested" state — warn so an add that fans out to
		# nobody is never silent. Absence stays a clean legal no-op (D-09).
		raw_pid = event.config.get("portfolio_id") if isinstance(event.config, dict) else None
		portfolio_id = self._portfolio_id_from(event)
		if portfolio_id is not None:
			strategy.subscribe_portfolio(portfolio_id)
		elif raw_pid is not None:
			self.logger.warning(
				'add for strategy %s carries an unparseable config["portfolio_id"] — the '
				'strategy was registered but is subscribed to NO portfolio and will emit '
				'no signals; re-issue subscribe_portfolio with a UUID',
				event.strategy_name)
```

A cleaner structural fix is to have `_portfolio_id_from` return a three-state result
(`ABSENT` / `MALFORMED` / handle) so all three call sites branch on the same distinction.

### WR2-02: rehydrate's quarantine tuple is narrower than the sibling admission catch sites — a bare-`ValueError` row crashes boot instead of quarantining

**File:** `itrader/strategy_handler/registry/rehydrate.py:94-115`, `:325-379`

**Issue.** The `ljn` remediation rewrote `_QUARANTINABLE` from a five-member explicit tuple
to `(StrategyAdmissionError, UnwarmableTimeframeError)`. That rewrite is **set-preserving**
for the five previously-listed types (I checked each reparent:
`UnknownStrategyTypeError` → `StrategyAdmissionError` at `catalog.py:46`,
`StrategyConfigError` → `StrategyAdmissionError` at `config_codec.py:114`,
`UnknownParamError`/`MissingParamError` → `(ValidationError, StrategyAdmissionError)` at
`strategy.py:52,69`), so it is not a regression.

The problem is the **asymmetry it makes visible and then pins**. The same `ljn` task widened
both control-plane admission sites to catch **bare `ValueError`**, with an explicit rationale:

```python
# ValueError stays as the residue: validate() and the _apply_params tickers/enum guards
# raise BARE ValueError (base.py, SMA_MACD_strategy.py), and a third-party strategy's
# validate() override can never be brought into our hierarchy.
```

`rehydrate_strategies` processes payloads of the **same shape through the same
`build_strategy` → `cls(**params)` → `_apply_params` → `validate()` chain**, but its catch
does *not* include that residue — and the surrounding comment now instructs future
maintainers **"Do NOT widen this tuple."** So the exact exception class the sibling sites
were just taught to expect from `validate()` is the one class rehydrate will not quarantine.

**Concrete failure scenario.** Ship a new version of a registered strategy whose `validate()`
gains a cross-field rule — e.g. `SMA_MACD.validate()` at `SMA_MACD_strategy.py:42` already
raises a bare `ValueError("short_window must be < long_window")`, and adding any comparable
rule (a max-window cap, a tickers/timeframe compatibility check) is routine class evolution.
One legacy row in `strategy_registry` violates the new rule. At the next live boot,
`build_strategy` raises a bare `ValueError` inside the per-row `try`; `_QUARANTINABLE` does
not match it; it propagates out of `rehydrate_strategies` → `build_live_system` and **the
whole engine fails to boot**, taking every healthy sibling strategy with it — the precise
"a bad ROW becomes a bad SYSTEM" outcome the D-19 arm split exists to prevent. Note the
module already treats the structurally identical case (`UnknownParamError` /
`MissingParamError` — "param drift: the class gained a required param") as quarantinable;
cross-field drift is the same event with a different exception base.

This is a robustness gap rather than a live defect — no *currently persisted* row can be
invalid, because `add` and `reconfigure` both validate before writing. But that is exactly
the argument that would also have dismissed the already-listed param-drift arms.

**Fix.** Either add the residue and say why, keeping the narrowness argument intact
(`RehydrateInfrastructureError` roots at `RuntimeError`, so it is still not swallowed):

```python
_QUARANTINABLE: tuple[type[Exception], ...] = (
	StrategyAdmissionError,    # unknown strategy_type / undeserializable blob / param drift
	UnwarmableTimeframeError,  # payload x environment: finer-than-base / non-multiple tf
	ValueError,                # WR2-02: the residue the add + reconfigure admission sites
	                           # already catch — validate()'s cross-field rules and the
	                           # _apply_params tickers/enum guards raise BARE ValueError, and
	                           # a third-party validate() override can never join our
	                           # hierarchy. Cross-field drift is param drift; a stale ROW must
	                           # not become a boot outage. Still narrow: infrastructure roots
	                           # at RuntimeError (RehydrateInfrastructureError) and store
	                           # faults propagate unchanged.
)
```

— or, if the narrowness is deliberate, record the decision explicitly in the comment ("a
`validate()` failure at rehydrate is treated as a SYSTEM problem because …") so the
divergence from the two admission sites is a stated choice rather than an accident of which
task touched which file.

### IN2-02: the new `except` tuples are still subsumed (IN-05 re-expressed, not closed)

**File:** `itrader/strategy_handler/lifecycle/manager.py:397`, `:809`, `:866`

**Issue.** `StrategyAdmissionError` is declared `(ITraderError, ValueError)`
(`core/exceptions/strategy.py:23`), so in `except (StrategyAdmissionError, ValueError)` the
first member is entirely subsumed by the second — the same shape IN-05 flagged. The listing
is defensible as self-documentation (it names the intended domain, and the docstring makes
the `ValueError` base explicitly non-removable), but IN-05's underlying complaint —
"it obscures which exceptions are genuinely distinct" — is unaddressed, and a reader
comparing these three sites to `_QUARANTINABLE` (which lists `StrategyAdmissionError`
*without* `ValueError` and means something genuinely different by it — see WR2-02) has no
signal that the two spellings are not interchangeable.

**Fix.** Either drop the redundant member and let the comment carry the intent, or keep it
and add one clause naming the redundancy as deliberate: *"`StrategyAdmissionError` is listed
for intent even though `ValueError` already subsumes it — the two are NOT interchangeable at
`rehydrate._QUARANTINABLE`, which means the narrow one."*

### IN2-03: `IN-04`'s fix re-creates the `IN-01` private-reach pattern

**File:** `itrader/strategy_handler/strategies_handler.py:292`

**Issue.** The new read-through property returns `self._lifecycle._universe` — a private
attribute on another object. This is the same reach `IN-01` asked to remove for
`_pending_removals` / `_allow_short_selling` / `_enable_margin`, and the fix applied there
(make the collaborator-side name public, keep the handler-side name private) applies verbatim
here. `StrategyLifecycleManager` exposes no public accessor for `_universe`.

**Fix.** Add a public read accessor on the manager (a `universe` property over `_universe`,
mirroring `ManagedStrategies.pending_removals`) and have the handler property forward to it.
No test touches either name, so this is a zero-risk edit.
