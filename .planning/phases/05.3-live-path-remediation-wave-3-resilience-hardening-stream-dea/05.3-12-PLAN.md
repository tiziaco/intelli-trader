<objective>
Close the two remaining Phase-05.3 code-review warnings that were scoped out of the D-24..D-27
gap-remediation wave: **WR-03** (resume-while-blind-to-fills) per **D-28** and **WR-05**
(recursive gap-backfill on an under-returning page) per **D-29**. Each ships a production-premise
test per **D-27**. WR-04 is deliberately DEFERRED (single-symbol wiring today; not in this plan).

Purpose:
- **D-28 (WR-03):** the reversible pause state is a single bool + single `_pending_stream_resume`
  Event, and BOTH venue arms (exchange fills+orders, data-provider candles) share one
  `_on_venue_stream_down`/`_on_venue_stream_up` callback pair. `_on_venue_stream_up` sets the resume
  Event on ANY single stream reconnect, so `_maybe_resume_after_reconnect` resumes NEW order
  submission while another stream is still down (fill stream down → engine trades blind to fills).
  Fix WITHOUT adding engine-side aggregation: each arm already owns a `_streams_down` set — expose a
  public `is_streaming_healthy()` on each, and gate `resume_submission()` behind a compound
  `_all_venue_streams_healthy()` read. No new engine state, no namespaced names. (`LiveTradingSystem`
  is slated for a full post-Phase-6 refactor; keep its footprint to one helper + one guard.)
- **D-29 (WR-05):** `_replay_and_deliver` clamps the replayed interior only at the UPPER end and
  assumes the venue page begins exactly at `first_missing`. An under-returning page (starts later /
  omits the earliest bar) makes the first `update(cb)` re-enter the gap branch and spawn ANOTHER
  loop-native backfill recursively, mid-replay. Fix: low-end clamp + fail-loud on a non-contiguous
  first bar (typed data-integrity error → provider supervisor → connector halt) + a `_replaying_backfill`
  re-entrancy guard that refuses a nested spawn.
Output: two arm predicates + a compound resume gate; feed fail-loud + re-entrancy guard; a
resume-gate integration test + a backfill-contiguity unit test; oracle byte-exact; OKX stack
hot-path-inert; mypy strict clean.
</objective>
