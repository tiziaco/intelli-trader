### SimulatedExchange cash-restore gap (investigation → D-23)
- **Prompt:** owner recalled the 05.2 reviewer flagging that a real account reconciles cash from the venue, but the SimulatedExchange can't — asked for the current state.
- **Investigation findings:**
  - Live/sandbox (`VenueAccount`): cash truth = venue REST `snapshot()` + push stream; `VenueReconciler.reconcile()` + `_compare_symbol_drift` heal drift. `restore_cash` is a no-op.
  - Paper (`SimulatedExchange`/`SimulatedCashAccount`): cash = computed ledger only; no venue to reconcile against.
  - **Gap:** `portfolio_handler.rehydrate()` is gated to `if self.exchange == 'okx'` (`live_trading_system.py:1337/1351/1360`). Consequences: (1) `SimulatedCashAccount.restore_cash` (`simulated.py:167`) is currently DEAD — unreachable on both paths (okx swaps to `VenueAccount` before rehydrate; paper never calls `start()`); (2) durable portfolio storage is wired on DB-presence (line 299), decoupled from mode, so a persistent paper/sim account through `start()` would restart with construction-time cash and no venue to correct it.
  - **Status:** latent, not active — shipped paper flow uses `run_paper_replay()` (single-shot, no `start()`). Becomes real once a persistent/restartable paper account is driven through `start()` (e.g. the FastAPI app layer).
- **Conceptual clarifications** (owner mental-model check): the DB read is RESTORE (engine's own persisted memory), not RECONCILE (independent authority) — Simulated can remember but never self-correct; cash is NOT derivable from positions (recompute source = the durable transaction ledger, not positions); D-19 atomicity matters MORE here (torn restore is permanent — no venue heals it).
- **Options:** (a) restore the persisted cash scalar (activate existing `restore_cash`, lean on D-19) / (b) recompute cash+positions by replaying the durable `transactions` ledger (event-sourcing).
- **Selected:** **Option (a).** Ungate rehydrate; keep venue reconcile/snapshot/link okx-only. Recorded as **D-23**. Option (b) deferred.

# Gap-Remediation Session — 2026-07-05 (evening)

**Trigger:** `/gsd-discuss-phase 05.3 --gaps`. Verification returned `gaps_found` (20/23 must-haves; `05.3-VERIFICATION.md`). Three D-NN decisions have a confirmed code-vs-production gap. Owner asked for the architecturally-correct fix and recommendation.

## The three gaps

All share one pattern: the code unit is correct-in-isolation and green on its unit test, but composition-root wiring (or a cross-module premise) breaks the end-to-end behavior — the class goal-backward verification exists to catch.

1. **D-15 / CR-01 (blocker):** `drop_pending` premise ("venue `free` already excludes the hold") is false for the wired `market_type='spot'` leaf. `_write_balance_stream` is positions-only, `_venue_balance` re-baselined only by `snapshot()` (D-01 single-channel cash) → dropping the overlay on ack over-states buying power (the overlay was the sole hold-tracker).
2. **D-12 / WR-01:** `catch_up_missed_fills` fully built + unit-tested, zero production call sites — `_maybe_resume_after_reconnect` never calls it.
3. **D-11 / WR-02 (partial):** `_on_task_done` observes `task.exception()` but `OkxConnector.set_halt_signal` is never wired at the composition root → escalation guard always `None`.

## Area discussed — D-15 fix (the only real architectural decision)

- **Framing:** the `_pending` overlay's correct lifetime is a function of the cash channel. Gap between "cash committed" and "cash in `_venue_balance`" closes at ack ONLY for a stream-refreshed (margin) account; on the single-channel spot leaf it never closes until terminal fill/cancel.
- **Options presented:**
  1. **(a) Gate the drop for spot** — `drop_pending` no-op when `market_type=='spot'`; overlay held until terminal release (existing path already pops it). Respects D-01.
  2. **(b) Net venue `used`/`free` into `_venue_balance`** — makes D-15's premise universally true, but reintroduces the dual-channel cash authority D-01 deliberately killed.
- **Recommendation given & accepted:** **option (a)** — minimal change, respects D-01, reverts the wired path to correct behavior; D-15's drop-on-ack stays dormant-valid for a future margin account. On spot there was never a double-count for D-15 to remove (V17-13's double-count exists only on a stream-refreshed account). → **D-24.**

## Areas discussed — the two mechanical fixes

- **D-12:** wire `catch_up_missed_fills()` into `_maybe_resume_after_reconnect` before `snapshot()`. **Resume path only**; plan-time grep for a distinct `attempt==1` entry, wire only if it truly exists. → **D-25.**
- **D-11:** one line — `self._okx_connector.set_halt_signal(self._request_connector_halt)` alongside the exchange/provider/portfolio arms. → **D-26.**
- **Test bar:** all three ship a wiring/integration test that exercises the production wiring/premise, not the mechanism in isolation (unit-only reproduced the blind spot). → **D-27.**

## WR-03 — resume-while-blind-to-fills (→ D-28)

- **Framing:** pause state is a single bool + single `_pending_stream_resume` Event; both venue arms (exchange fills+orders, provider candles) share one down/up callback pair. `_on_venue_stream_up` sets the resume flag on ANY single stream's reconnect → `_maybe_resume_after_reconnect` resumes NEW submission while another stream is still down. Confirmed worse than the review implied: the exchange fires `on_stream_up` per-stream (okx.py:824-829), so even orders-up-while-fills-down resumes blind.
- **Options presented:** (A) engine-level aggregate `_streams_down` set + namespaced stream names; (B) two arm-booleans set by which arm fired; (C) each arm exposes a health predicate, engine gates resume on a compound read.
- **Owner steer:** do NOT add aggregation/namespacing to `LiveTradingSystem` (slated for a full post-Phase-6 refactor); the health-tracking should live in the execution / data-provider arms — where `_streams_down` **already is**.
- **Selected:** refined **(C)** → **D-28.** Each arm exposes `is_streaming_healthy()` over its own `_streams_down`; engine adds one `_all_venue_streams_healthy()` helper + one guard before `resume_submission()`. No engine-side set, no namespaced names. Callbacks stay dumb (Event = "re-check"). Pause-on-any-down unchanged (only resume gated). Rejected (A) (duplicates arm state into the messy engine) and (B) (needs per-arm callback identity that doesn't exist).

## WR-05 — recursive gap-backfill on an under-returning page (→ D-29)

- **Framing:** `_replay_and_deliver` clamps only the upper end and assumes the venue page starts at `first_missing`; an under-returning page makes the first `update(cb)` re-enter the gap branch → nested `spawn_gap_backfill` mid-replay.
- **Decision (accepted as recommended):** fail-loud, not skip-with-hole — low-end clamp (symmetric with the upper clamp) + raise a typed data-integrity error on a non-contiguous first bar (escalates via the provider supervisor to a connector halt, matching this phase's "halt on anything unexplained"), **plus** a `_replaying_backfill` re-entrancy guard as the structural stop against recursion regardless of page shape. → **D-29.**
- **Test bar:** both fixes ship a production-premise test (resume-gate integration driving `_maybe_resume_after_reconnect`; backfill-contiguity unit feeding an under-returning page), per the D-27 discipline — not isolated-predicate / happy-path units.
