# Phase 05.3 Plan 02: Exchange Order/Fill Lifecycle Resilience Summary

D-13 makes an ambiguous OKX submit timeout leave the order mirror in-flight (PENDING) instead of falsely terminalizing it to REJECTED, and D-12 recovers fills missed during a stream outage (bounded `fetch_my_trades` catch-up on resume, dedup-safe by D-08) while translating venue-side `watch_orders` CANCELLED/EXPIRED rows into reconciling `FillEvent`s.

## What Was Built

### Task 1 — Submit-timeout in-flight, not REFUSED (D-13 / V17-09)
`OkxExchange.on_order`'s submit-failure handler now branches on the error **class** before the `FillEvent(REFUSED)` emit:
- **Ambiguous transport** (`TimeoutError` — the class raised by `connector.call(...).result(timeout=_CALL_TIMEOUT)`, which also aliases `asyncio.TimeoutError` / `concurrent.futures.TimeoutError` on 3.11+ — or a ccxt `NetworkError`/`RequestTimeout`/`DDoSProtection`): the submit may have reached the venue and be resting/partially filled, so **no** terminal event is emitted. The mirror stays PENDING (in-flight/unknown); the pre-submit `clOrdId` correlation still resolves a streamed fill, and the next `VenueReconciler` sweep (or a `fetch_order(clOrdId)` probe) settles true venue state.
- **Definitive rejection** (e.g. `ccxt.InvalidOrder`): unchanged — still emits `FillEvent(REFUSED)` so the mirror reconciles PENDING→REJECTED.

This turned the A7 spine gate GREEN (`test_submit_timeout_inflight_does_not_terminalize_mirror`) while the over-fit guard (`test_definitive_venue_rejection_still_produces_refused`) stays GREEN.

### Task 2 — Missed-fill catch-up + venue-cancel reconciliation (D-12 / V17-08)
- **`catch_up_missed_fills()`** (new engine-thread call site): for each active-order symbol, runs a **bounded** `fetch_my_trades(symbol, since=disconnect_ts, limit=100)` (a single page, no ccxt auto-pagination) through the connector-call bridge and routes each returned trade through the existing `_handle_trade`. D-08 `{symbol}:{trade_id}` dedup makes a re-fetched (or live-redelivered) trade an idempotent no-op → the missed fill settles **exactly once**. Fetch/translation failures are swallow-and-logged (reconcile sweep is the backstop).
- **Disconnect-floor plumbing:** active-order symbols are tracked on submit/adopt (add-only, bounded by the small venue universe); the catch-up floor `_disconnect_ts_ms` is snapshotted from the stream-down transition (`_mark_stream_down`) as the last processed venue-ms timestamp (business time, never wall-clock), advanced on each emitted fill, and cleared when the catch-up consumes it.
- **`_handle_order_update()`** (new per-row translator): a venue-side `watch_orders` row with status `canceled`/`expired` becomes a `FillEvent(CANCELLED/EXPIRED)` on `global_queue` (donor shape from the `:250` REFUSED emit — order's own price/quantity, commission `Decimal("0")`, time from venue ms) so `OrderHandler.on_fill` drives the mirror terminal. `open`/`closed` stay log-only (FILLED money crosses on `watch_my_trades` — double-settle guard). `_consume_orders` now routes every row through it under a per-row swallow-and-log guard.
- **`VenueCorrelationIndex.order_for_venue_id`** (new read-only, lock-guarded accessor): resolves `venue_id → OrderEvent` for the order-status path, keeping correlation state single-sourced on the index (WR-05 R1) rather than duplicating a map on the exchange.
