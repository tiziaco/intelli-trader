# Phase 05.3 Plan 03: Live-path Correlation Carry-overs (D-16) Summary

**Closed the 05-13 fill-correlation carry-overs on `VenueCorrelationIndex` + `OkxExchange`: the dedup slot is now consumed only after a proven emit (WR-02), `release` drains buffered fills through `resolve` (WR-03), `release_pending` pairs `register_pending` on a definitive submit failure (WR-01), `capacity < 1` is rejected (IN-01), and the uncorrelated-fill buffer is bounded + alarmed.**

## Performance

- **Duration:** ~55 min
- **Tasks:** 2 (both TDD: RED → GREEN)
- **Files modified:** 5 (2 source, 3 test)

## Accomplishments
- **WR-02 (mark-seen-after-emit):** `resolve` no longer consumes the dedup slot on the emit verdict; it returns the symbol-scoped `dedup_key` and the caller (`okx._handle_trade`) marks it seen only after `_emit_fill` returns `True`. A malformed-then-corrected re-send of the same `{ticker}:{trade_id}` now emits instead of being silently dropped. None-keyed backtest fills carry `dedup_key=None` → clean skip (oracle stays dark).
- **WR-03 (release drains through resolve):** new `take_pending()` empties the buffer while correlation is intact; `okx.release_venue_correlation` routes each drained fill through `_handle_trade` → `resolve` (dedup + correlation) — no raw `_emit_fill` in the drain path — so a re-delivered buffered fill dedups against the ring instead of double-counting.
- **WR-01 (release_pending):** new paired inverse of `register_pending`; called from the OKX definitive-rejection branch (not the D-13 ambiguous-transport branch), so a failed submit does not leak a pending clOrdId correlation.
- **IN-01 (capacity guard):** `__init__` rejects `capacity < 1` (and `pending_buffer_max < 1`) with a typed `ValidationError`.
- **Bounded + alarmed uncorrelated buffer:** `_pending_fills_by_venue_id` is an `OrderedDict` bounded at `pending_buffer_max` (default 10 000) with FIFO oldest-bucket eviction and a `WARNING` alarm; a running count is kept in sync via `_buffer_pending_locked` / `_pop_pending_locked`. External unknown-venue (`clOrdId==""`) fills can no longer grow it without limit.

## Decisions Made
- **WR-02 option: mark-seen-after-True-emit** (uses `_emit_fill`'s existing bool return), not validate-payload-in-resolve. Chosen because it matches the plan's `key_link` ("mark-seen only after a True emit verdict"), avoids duplicating payload validation inside `resolve`, and keeps the None-keyed backtest path a clean skip. Fills arrive serially on the single connector loop thread, so deferring mark-seen to just-after-emit introduces no new cross-thread race.
- **New `take_pending()` helper for WR-03:** required because `release()` drops correlation atomically; the buffer must be drained (correlation still present) before eviction so each drained fill can re-correlate through `resolve()`.
