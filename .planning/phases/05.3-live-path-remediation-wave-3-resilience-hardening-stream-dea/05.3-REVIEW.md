## Summary

Reviewed the live-path resilience-hardening diff since `b903b611` (async stream supervision,
halt lifecycle, buying-power overlay accounting, order admission/preflight, restart-restore,
atomic fill persistence). The code is heavily decision-anchored and the secret-scrubbing on the
halt/connector egress paths is consistent (exception TYPE only, fixed reason literals — no
`str(exc)` reaches the CRITICAL alert or the durable record). The D-18 external-order admission
gate correctly refuses raw `EventType.ORDER` injection through `add_event`. Decimal edges are
clean (`to_money(str(x))` throughout).

However the review surfaced **three wiring/accounting defects that silently disable the very
resilience features this phase adds**, plus supporting robustness gaps:

1. The buying-power overlay drop (`drop_pending`) over-states available cash on the wired spot
   venue because the premise it rests on (venue-side netting reflected in `_venue_balance`) does
   not hold for the spot path — the balance stream never refreshes `_venue_balance`.
2. `OkxExchange.catch_up_missed_fills` (the whole D-12 missed-fill recovery) is never called from
   the resume path — it is dead code in production.
3. `OkxConnector.set_halt_signal` is never wired, so the connector-level dead-task fail-safe
   (`_on_task_done` → halt) is inert.

## Critical Issues

### CR-01: `drop_pending` over-states buying power on the wired spot venue (double-count premise is false for spot)

> ✅ **RESOLVED** — plan **05.3-10** (D-24/D-27). `drop_pending` is now gated behind
> `_cash_stream_nets_holds` (`market_type != "spot"`, `venue.py:146`) with a negation-first
> guard-clause (`venue.py:743-746`): on the single-channel spot leaf it early-returns, holding
> the `_pending` overlay from ack→terminal `release`, so `available_balance` no longer snaps back
> to the full settled balance while an order is open. The derivative leaf keeps D-15's drop-on-ack;
> `release` stays unconditional on both leaves. Fix option (a) was taken; option (b) (net venue
> used/free into `_venue_balance`) was NOT built (D-01 single-channel cash preserved). A spot
> over-statement regression test was added (`tests/unit/portfolio/test_venue_account_drop_pending.py`).
> Re-verified PASSED 2026-07-05.

**File:** `itrader/portfolio_handler/account/venue.py:702-720` (drop path); `itrader/portfolio_handler/account/venue.py:281-314` (`_write_balance_stream`); `itrader/order_handler/order_manager.py:266-278` (ack trigger)

**Issue:** `drop_pending` removes the local pending-reservation overlay entry on the venue
ORDER-ACK. Its stated justification is that keeping the overlay to terminal release "double-counts"
the hold — "once by this local overlay, once in the venue netting." That premise requires
`balance`/`available_balance` to reflect the venue's netting (i.e. `_venue_balance` must drop when
an order rests). On the **wired spot path** (`market_type='spot'`, BTC/USDC), it does not:
`_write_balance_stream` writes **positions only** and deliberately never touches `_venue_balance`
(cash is "single-channel", re-baselined only by `snapshot()`). So between snapshots
`_venue_balance` is static and `available_balance = balance − Σ_pending` — the local overlay is the
**only** tracker of a resting order's cash hold.

Dropping that overlay on ack therefore does not remove a double-count; it removes the sole record
of committed cash. For a resting limit/entry order, `available_balance` snaps back to the full
settled balance while the order is still open on the venue consuming cash. A second order can then
be admitted against cash already committed to the first (buying-power over-statement / unintended
over-commit), until the next `snapshot()` or fill re-baselines. This is exactly the buying-power
accounting correctness this phase targets, and it is wrong for the venue that is actually wired.

**Fix:** Do not drop the overlay unconditionally on ack for the spot path. Either (a) keep the
overlay hold until the order terminalizes (fill/cancel), accepting the pre-D-15 behaviour on spot
where no venue-`free` netting exists to double-count against; or (b) if the overlay must be
released on ack, parse and net against the venue `free`/`used` balance so a resting-order hold is
still reflected in `available_balance`. Gate the drop behind an explicit "venue-netting is
reflected in balance" condition rather than applying it to a spot leaf whose `_venue_balance` is
stream-static:

```python
def drop_pending(self, order_id: OrderId) -> None:
    # Only safe when a venue-side hold is reflected in _venue_balance (derivative
    # netting stream). On the spot leaf _venue_balance is snapshot-static, so the
    # overlay is the ONLY committed-cash tracker — keep it until terminal release.
    if self._market_type == "spot":
        return
    with self._lock:
        self._pending.pop(str(order_id), None)
```

## Warnings

### WR-01: `catch_up_missed_fills` is never invoked — D-12 missed-fill recovery is dead in production

> ✅ **RESOLVED** — plan **05.3-11** (D-25/D-27). `_maybe_resume_after_reconnect` now calls
> `self._okx_exchange.catch_up_missed_fills()` before `self._venue_account.snapshot()`
> (`live_trading_system.py:949`, inside the stay-paused try/except). The previously-orphaned method
> now has exactly ONE production call site (was 0, grep-confirmed). A resume-path integration test
> (`tests/integration/test_resume_missed_fill_catchup.py`) drives `_maybe_resume_after_reconnect`
> and recovers a fill settled during a simulated disconnect window (not the exchange method in
> isolation). Grounding note: grep confirmed no distinct `attempt==1` live-path reconnect entry
> point exists — every reconnect funnels through `_maybe_resume_after_reconnect`, so no second call
> site was invented. Re-verified PASSED 2026-07-05.

**File:** `itrader/trading_system/live_trading_system.py:905-939` (`_maybe_resume_after_reconnect`); `itrader/execution_handler/exchanges/okx.py:626-673`

**Issue:** `OkxExchange.catch_up_missed_fills` documents that "the composition-root resume wiring …
drives this alongside that reconcile, on resume and on the attempt==1 reconnect." But
`_maybe_resume_after_reconnect` — the only engine-thread resume drain — calls
`self._venue_account.snapshot()` and `self.resume_submission()` and **nothing else**. There is no
call to `self._okx_exchange.catch_up_missed_fills()` anywhere in the codebase (grep-confirmed:
only the definition and doc-comments). Consequently, fills that settle while the fill stream is
down are never re-fetched on reconnect: the order mirror stays PENDING and the position/cash never
reflect those trades until the next full startup reconcile (mid-session reconnect does only a REST
snapshot, not a two-sided reconcile — see WR-04 in the source). The `_disconnect_ts_ms` floor and
the entire catch-up page machinery are unreachable in production.

**Fix:** Call the catch-up on the engine-thread resume, before/alongside the snapshot:

```python
def _maybe_resume_after_reconnect(self) -> None:
    ...
    try:
        if self._okx_exchange is not None:
            self._okx_exchange.catch_up_missed_fills()   # D-12: recover downtime fills
        if self._venue_account is not None:
            self._venue_account.snapshot()
    except Exception as e:
        ...
```

### WR-02: `OkxConnector.set_halt_signal` is never wired — connector dead-task fail-safe is inert

> ✅ **RESOLVED** — plan **05.3-11** (D-26/D-27). The composition root now arms the connector's
> halt signal — `self._okx_connector.set_halt_signal(self._request_connector_halt)`
> (`live_trading_system.py:512`) — as the 4th arm alongside the exchange (`:502`), provider (`:513`)
> and portfolio (`:574`). `OkxConnector._on_task_done`'s `if self._halt_signal is not None:` guard is
> no longer always-False, so a task dying outside the stream supervisors escalates to a fail-safe halt
> instead of logging-and-vanishing. A wiring-level test (`tests/integration/test_live_system_okx_wiring.py`)
> asserts `start()` arms the real `_okx_connector`'s halt signal. Re-verified PASSED 2026-07-05.

**File:** `itrader/trading_system/live_trading_system.py:440-507` (OKX wiring); `itrader/connectors/okx.py:90-99, 209-231` (`set_halt_signal` / `_on_task_done`)

**Issue:** `connectors/okx.py` adds a done-callback (`_on_task_done`) that escalates a spawned
stream task dying with an unexpected exception to `self._halt_signal("connector-fatal")` — the
V17-07 fix so "a dead task no longer vanishes silently." But the composition root only calls
`set_halt_signal` on `self._okx_exchange` (line 502), `self._okx_data_provider` (line 505) and
`self.portfolio_handler` (line 566) — **never** on `self._okx_connector`. So
`self._halt_signal is None` in the connector and `_on_task_done`'s escalation is a no-op: a task
that dies outside the exchange/provider supervisors (e.g. a supervisor bug, or any coroutine
spawned via `connector.spawn` that is not itself supervisor-wrapped) still logs and vanishes with
no halt. The backstop this phase added is not armed.

**Fix:** Wire the connector's halt signal alongside the other arms in `__init__`:

```python
self._okx_connector.set_halt_signal(self._request_connector_halt)
self._okx_exchange.set_halt_signal(self._request_connector_halt)
...
```

### WR-03: Shared single resume Event across two independent streams can resume submission while a stream is still down

**File:** `itrader/trading_system/live_trading_system.py:502-507, 884-903`

**Issue:** Both the order arm (`_okx_exchange`) and the data arm (`_okx_data_provider`) are wired to
the SAME `_on_venue_stream_down`/`_on_venue_stream_up` callbacks, which flip the SAME
`_submission_paused` flag and set the SAME `_pending_stream_resume` Event. If the fill stream stays
down while the candle stream flaps down-then-up, `_on_venue_stream_up` sets the resume Event and
`_maybe_resume_after_reconnect` resumes NEW order submission even though the fill stream is still
disconnected — i.e. the engine resumes trading while blind to fills, so newly-submitted orders'
fills never stream back and the mirror stays PENDING. The pause is correctly idempotent, but resume
is not stream-scoped.

**Fix:** Track down streams per-name (the exchange already keeps `_streams_down`) and only resume
when ALL venue streams are healthy — e.g. resume only when the set of down streams is empty, or key
the resume decision on both arms reporting up. Alternatively give each arm its own
down/up→pause/resume aggregation so a single-arm reconnect cannot clear a pause another arm still
holds.

### WR-04: `_run_session_baseline_guard` reads a hardcoded `_OKX_STREAM_SYMBOL`, decoupled from the reconcile scope

**File:** `itrader/trading_system/live_trading_system.py:671-696`

**Issue:** The session-start baseline guard asserts the venue base-asset holding against the engine
position for exactly one symbol, the module constant `_OKX_STREAM_SYMBOL` (`"BTC/USDC"`). Any venue
position or engine position on a different symbol is not compared, so an unexplained residual on any
other instrument is silently trusted rather than halting. The guard's stated contract ("the engine
must NEVER trade on exposure it cannot explain") is only enforced for the single hardcoded pair.
`_venue_account.positions` is a full map — the guard iterates only one key of it.

**Fix:** Iterate the union of venue-reported symbols and engine open-position symbols, comparing
each within the per-instrument epsilon, rather than the single `_OKX_STREAM_SYMBOL`:

```python
symbols = set(self._venue_account.positions) | {
    p.ticker for pf in self.portfolio_handler.get_active_portfolios()
    for p in pf.open_positions}   # compare every symbol, not just the wired one
```

### WR-05: Loop-native gap backfill has no lower-bound clamp — an under-returning venue page can spawn a nested backfill

**File:** `itrader/price_handler/feed/live_bar_feed.py:329-360` (`_spawn_loop_native_gap_backfill._replay_and_deliver`)

**Issue:** `_replay_and_deliver` clamps the replayed interior only at the UPPER end
(`if cb["ts"] > last_ms: break`) and relies on the assumption "each advancing L by one tf
(in-sequence, no nested gap)." If the venue's `fetch_ohlcv` page does not begin exactly at
`first_missing` (a common venue behaviour — pages can start later, or omit the earliest bar), the
first `self.update(cb)` re-enters the monotonic guard with `t > last + tf` and spawns ANOTHER
loop-native gap backfill, recursively — while still inside the first backfill's replay. There is no
guard that the first replayed bar is contiguous with `last`.

**Fix:** Assert/verify the first replayed bar is exactly `first_missing` (or skip bars below
`first_missing` and hard-fail if the earliest expected bar is absent) before replaying, so an
under-returning page fails loud instead of recursively spawning overlapping backfills.

## Info

### IN-01: `_publish_and_continue` binds `error_message=str(exc)` on the live ERROR route

**File:** `itrader/trading_system/live_trading_system.py:601-611`

**Issue:** The halt/connector egress paths are carefully scrubbed (TYPE + fixed literal only), but
the general live handler-failure path emits `error_message=str(exc)` into an `ErrorEvent` consumed
by the alert sink. On the live path this is normally an engine-thread handler exception (not a raw
connector payload, since the OKX arm swallows+scrubs its own), so the leak surface is small — but it
is the one remaining unscrubbed `str(exc)` reaching operator egress. Consider scrubbing to the
exception type + a fixed message for consistency with the halt discipline, or documenting why the
engine-thread handler exceptions are considered secret-free.

### IN-02: `release_venue_correlation` drains the buffer twice (redundant, not incorrect)

**File:** `itrader/execution_handler/exchanges/okx.py:609-624`; `itrader/execution_handler/exchanges/venue_correlation.py:357-378`

**Issue:** `release_venue_correlation` calls `take_pending(venue_id)` (drains the buffer) and then
`release(venue_id)`, but `release` also drains `_pop_pending_locked(venue_id)` and returns those
trades — which the caller discards. After `take_pending` the bucket is empty, so `release`'s drain
is always `[]` here; harmless but the two drain calls invite confusion about which one owns the
buffered fills. Consider having `release_venue_correlation` consume `release`'s returned drained
trades (single drain) instead of calling `take_pending` separately.

### IN-03: `_upsert_position_on` reaches into store privates (`_store._position_to_row`, `_store.positions`)

**File:** `itrader/portfolio_handler/storage/cached_sql_storage.py:112-135`

**Issue:** The atomic fill-transaction position upsert reuses `self._store.positions` and
`self._store._position_to_row(...)` (a private) to stay byte-identical to the independent-commit
path. This couples the cached wrapper to store internals; a change to `SqlPortfolioStateStorage`'s
open-row shape or the meaning of its private mapper silently diverges the two write paths. Consider
promoting `_position_to_row` to a documented internal seam or having the store expose a
`set_position_on(connection, ...)` so the shared-connection path is not built on a private.
