<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| connector loop thread → engine thread | A stream-task death (`_on_task_done`) or a reconnect (resume drain) crosses from the asyncio loop thread to the engine thread; only flag-setting happens on the loop, blocking I/O and halts on the engine |
| venue REST (`fetch_my_trades`) → mirror/portfolio | Re-fetched trades on resume cross into `_handle_trade`; the D-08 `{symbol}:{trade_id}` dedup guard makes a re-fetched/again-reconciled trade a no-op (no double-settle) |
| composition root → connector halt escalation | Arming `set_halt_signal` enables an unexpected task death to escalate to a fail-safe engine halt (fail-safe: unknown ⇒ halt, never silent task death) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05.3-25 | Tampering / Repudiation | fills settled during a stream outage never recovered on resume (WR-01) | mitigate | wire `catch_up_missed_fills` into the resume drain before `snapshot()`; dedup (D-08) makes a later reconcile idempotent, no double-settle |
| T-05.3-26 | Denial of Service (fail-open) | a connector task dies outside the supervisors and vanishes silently (WR-02) | mitigate | arm `_okx_connector.set_halt_signal(_request_connector_halt)` so `_on_task_done` escalates to a fail-safe halt instead of logging-and-continuing on a dead task |
| T-05.3-26b | Information disclosure | halt egress on escalation | accept | escalation reason is the fixed literal `'connector-fatal'` (no exc text); the existing `_on_task_done` secret-scrub is preserved (this plan only supplies the caller) |
