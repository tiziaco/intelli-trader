<objective>
Portfolio storage seam (M2-08, #18) + order/transaction timestamp determinism (M2-09, #19).

Part A — Storage seam (D-09/D-10): build ONE unified `PortfolioStateStorage` in a peer
`portfolio_handler/storage/` package, generalizing the proven `order_handler/storage/` pattern
(ABC in `order_handler/base.py` per RESEARCH drift correction, + in-memory backend + factory).
Route ALL four managers' state through the injected seam — working state (open positions, reserved
cash) AND append-only records (transaction history, closed positions, cash operations, metrics
snapshots), exactly as order storage holds active + historical orders. `Portfolio` injects the
storage via `PortfolioStateStorageFactory.create(environment)`; the managers stop owning their
containers. The `storage/` package is a PEER of the subdomain folders (NOT inside each manager —
D-09 anti-pattern). Live backend raises/defers to D-sql. Durable record shapes use Decimal money +
native UUID ids + event-derived time — no DB code.

Part B — Timestamp determinism (D-12): thread the real event/fill time through
`Order.add_state_change` (default to event time, never `datetime.now()`); route `add_fill`'s
`fill_time` into the recorded transition timestamp; route `modify_order` through the single
validated `add_state_change` path (remove the duplicated direct append at `:436-448`). Use M2a's
injected `Clock` only where a wall-clock fallback is genuinely needed (`BacktestClock.now()` raises
if not advanced). Transaction record timestamps stay event-derived (`transaction.py:101` already
passes `filled_order.time`) — ensure no manager re-stamps with `datetime.now()` while routing.

Purpose: Make portfolio state swap-ready (live persistence = pure backend swap) and timestamps
deterministic/event-derived — both behavior-preserving against the oracle (timestamps excluded from
the oracle per Phase-1 D-12).
Output: storage seam package, routed managers, deterministic order timestamps, oracle unchanged.
</objective>
