# Phase 11.1 Plan 01: De-heavy the Connectors Barrel + Fold ConnectorProvider into GATE-01 Summary

Closed RESEARCH finding **F-2**, the hard prerequisite for **D-04**: `itrader.connectors.provider`
is now genuinely import-inert (it pulls neither `ccxt` nor `itrader.connectors.okx`), and the
GATE-01 inertness probe imports and constructs a real, empty `ConnectorProvider({})` inside its own
ccxt-absent window — so plan 11.1-07's backtest wiring lands against a gate that *proves* the new
posture instead of a comment documenting why it could not.

## What Was Built

**Task 1 — de-heavy the barrel** (commit `e899cbcd`)

- `itrader/connectors/__init__.py` no longer imports `OkxConnector` from `.okx`; `__all__` is
  `["LiveConnector"]`. `.base` stays — it is a pure `typing` Protocol module with no heavy imports.
- The barrel docstring gained a decision-tagged paragraph recording the posture: only the structural
  `LiveConnector` marker is exported; every concretion (`OkxConnector`, `PaperConnector`) is imported
  directly from its own module; a re-export of a ccxt-backed concretion makes importing *any* module
  under `itrader.connectors` heavy because the import machinery runs `__init__.py` first, and D-04
  puts `itrader.connectors.provider` on the backtest import graph. Explicitly records that no lazy
  `__getattr__` shim is provided and why.
- `itrader/connectors/provider.py`'s docstring paragraph claiming `connectors/__init__.py` "is
  deliberately NOT edited so the barrel/inertness surface stays unchanged" was false as of this
  commit. Rewritten: the module is still imported directly, AND the barrel no longer re-exports a
  concretion, so the import is genuinely inert rather than inert-by-call-site-discipline.

**Task 2 — fold `ConnectorProvider` into the gate** (commit `ffc15453`)

- Added `from itrader.connectors.provider import ConnectorProvider` alongside the existing
  `itrader.venues.*` imports in the P5 register-vs-build window, and `_connectors =
  ConnectorProvider({})` beside the plugin registrations — before the `_okx_leaked` assertion. The
  empty dict is deliberate and matches D-04: absence is a real empty collection, never `None`.
- The `_okx_leaked` failure message now names `ConnectorProvider` and the no-concretion-re-export
  barrel invariant (RESEARCH F-2), so a future regression reads as an instruction rather than a
  puzzle.
- Replaced the `:214-223` NOTE whose premise (importing anything under `itrader.connectors` pulls
  `ccxt`) no longer holds. The replacement states why the provider is now *inside* the window and
  what re-introducing a barrel concretion would do. The surrounding D-04 triple-deferral
  register-vs-build explanation is intact.

## Negative Control (required by the plan, run and discarded)

Temporarily restoring `from .okx import OkxConnector` in the barrel made the gate go **red on the new
assertion specifically** — not merely on `_FORBIDDEN`:

```
AssertionError: P5/11.1 register-vs-build inertness violation (VENUE-02/VENUE-03): importing +
registering the OKX/paper venue/data plugins, and importing + constructing the ConnectorProvider({})
the backtest wires (D-04), pulled the OKX/ccxt stack: ['ccxt.pro', 'ccxt', 'itrader.connectors.okx']
```

The re-export was then reverted by editing the file back (no `git checkout`/`git clean`), and
`git status` confirmed the barrel matched its committed state before Task 2 was staged. The gate is
non-vacuous: the import sits inside the ccxt-absent window, as intended.
