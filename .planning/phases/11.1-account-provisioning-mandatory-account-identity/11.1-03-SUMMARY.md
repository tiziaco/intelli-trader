## Deviations from Plan

### D1 тАФ [Rule 1: bug / Rule 2: missing critical functionality] The storage seam read `portfolio`; "unchanged" was not implementable

**This is the plan's one materially wrong factual claim, and it was a silent-corruption trap.**

The plan's Task 1 action says: *"the `portfolio: Any` parameter and the `self.portfolio = portfolio` assignment at `:90` are deleted. Everything below (`self.logger`, the `to_money(...).quantize(...)` balance, **the storage seam**) is unchanged."*

The plan's supporting truth is narrowly correct тАФ the only three **`self.portfolio`** occurrences were `:90`, `:831`, `:873`. But the constructor body read the **local** `portfolio` parameter five more times, in the M2-08 storage-seam block:

```python
storage = getattr(portfolio, "state_storage", None)
if storage is None:
    storage = PortfolioStateStorageFactory.create(
        getattr(portfolio, "_environment", "backtest"),
        sql_engine=getattr(portfolio, "_sql_engine", None),
        portfolio_id=getattr(portfolio, "portfolio_id", None),
    )
    try:
        portfolio.state_storage = storage      # WR-02 write-back
    except AttributeError:
        pass
```

Deleting the parameter and leaving this block "unchanged" does not fail тАФ that is the problem. `getattr(None-existent, ...)` would not have compiled, but the naive fix (drop the getattr, keep the fallback) *does* compile and **silently changes behaviour**: every account would fabricate its OWN in-memory backend, disjoint from the `state_storage` its three sibling managers share, and a `live` portfolio's account would silently drop to in-memory instead of the SQL backend. Reserved cash, locked margin and the cash-operation audit trail would diverge from the positions/transactions they must reconcile against тАФ a money-arithmetic integrity defect that no existing test asserts against.

**Fix:** the seam is now an explicit **keyword-only** parameter:

```python
def __init__(self, initial_cash=0.0, *, state_storage: "PortfolioStateStorage | None" = None) -> None
```

`Portfolio._init_managers` passes `state_storage=self.state_storage` (built immediately above, so identical to what the getattr used to find); `PaperVenuePlugin.new_account` forwards `getattr(portfolio_ref, "state_storage", None)`, preserving its behaviour exactly. Standalone construction falls back to a plain in-memory backend.

This **strengthens** D-01 rather than diluting it: the account's real dependency was never the `Portfolio`, it was the storage seam, and it now names that dependency instead of walking an object graph for it. The getattr-and-write-back dance (including the old WR-02 sibling-sharing workaround) is deleted as redundant.

Keyword-only is a deliberate fail-loud choice: with a positional `state_storage`, a stale `SimulatedCashAccount(portfolio, 100000.0)` call site would silently bind a `Portfolio` into the storage slot. Keyword-only makes it a `TypeError`.

**Verified:** `p.account._storage is p.state_storage is p.position_manager._storage` тЖТ `True`.
