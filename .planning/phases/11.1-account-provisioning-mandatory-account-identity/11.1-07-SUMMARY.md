# Phase 11.1 Plan 07: Paper Venue Plugin Builds Its Own Exchange; ExecutionHandler Resolves Summary

`PaperVenuePlugin` now builds its own `SimulatedExchange` from an `ExchangeConfig` injected at
construction and the one shared `ctx.rng` — exactly as `OkxVenuePlugin` builds its `OkxExchange` —
`ExecutionHandler` asks `VenueBundles` for that exchange instead of minting one, `compose_engine`
takes `venue_bundles` and no exchange config, and all three composition roots (both backtest arms
and the live root) are on the same venue path with the byte-exact oracle green at
`134 / 46189.87730727451`.

## What Landed

**Task 1 — the plugin builds its own exchange (commit `695665ed`, 3 files).**
`PaperVenuePlugin.__init__(exchange_config)` replaces the injected-exchange slot; `build_bundle`
lazy-imports `SimulatedExchange` inside the body and builds
`SimulatedExchange(ctx.bus, config=self._exchange_config, rng=ctx.rng)`. Both false docstrings (the
module's and the class's "REUSES the compose-built simulated exchange … injected at register time")
are rewritten. `new_account`'s margin/cash branch order is untouched — the non-margin arm is the
oracle path. The `connectors` argument is now explicitly commented as deliberately unread.

**Task 2 — the seam move (commit `53588d36`, 11 files).**
`_default_backcompat_config` promoted to the module-level public `default_exchange_config()`.
`ExecutionHandler.__init__` gains a REQUIRED keyword-only `venue_bundles` and loses
`exchange_config`; `init_exchanges` resolves `('paper', DEFAULT_ACCOUNT_ID)` off the memo and keeps
the identity-dedup connect loop verbatim. `compose_engine` gains required `venue_bundles`. BOTH
backtest arms build `ExecutionVenueRegistry` → `PaperVenuePlugin(spec.exchange)` →
`ConnectorProvider({})` → `VenueBundles(...)`. The five direct `ExecutionHandler(...)` test sites
share the new `tests/support/venue_wiring.py`.

**Task 3 — the live hoist (commit `d46615f7`, 1 file).**
The whole registration block (lazy plugin imports, `EnvCredentialResolver`, the SQL-gated
`venue_account_store`, both registries, the `ConnectorProvider`, the three `register` calls, the
test-only `data_plugins` injection) moved above `EngineContext`. The reach-in
`PaperVenuePlugin(execution_handler.exchanges[('paper', DEFAULT_ACCOUNT_ID)])` is deleted;
`default_exchange_config()` takes its place, preserving live's previous behaviour exactly (live
previously passed no exchange config, so its exchange was built from this same preset ∪ `{BTCUSD}`).
`assemble_venues` and the per-account registration loop are untouched.

## Evidence Against the Inert-Wiring / Silent-Corruption Shape

The dangerous failure here is a decorative provider: `VenueBundles` gets wired in, everything is
green, and `ExecutionHandler` quietly keeps its own mint on a surviving parallel path — two
exchanges over one venue, each with its own resting-order book.

**1. The mint is structurally gone.**
`grep -v '^#' itrader/execution_handler/execution_handler.py | grep -c "SimulatedExchange(self.global_queue"` → **0**.
The module's `SimulatedExchange` import survives only because `update_config` / `validate_config`
isinstance-narrow against it (verified before deleting).

**2. The composed exchange IS the provider's object (new test).**
`test_the_composed_exchange_is_the_one_the_bundle_provider_returned` asserts
`engine.execution_handler.exchanges[('paper', DEFAULT_ACCOUNT_ID)] is venue_bundles.get('paper', DEFAULT_ACCOUNT_ID, None).exchange`
— identity, because an equal-but-distinct `SimulatedExchange` would satisfy any
isinstance/config check while proving the mint survived. It also asserts
`list(venue_bundles._memo) == [('paper', DEFAULT_ACCOUNT_ID)]`, so the `get` above was a memo HIT,
not a second build.

**3. The oracle provably traverses the new legacy-arm wiring (fail-first probe, recorded).**
The oracle drives `scripts/run_backtest.py::main()` → the LEGACY construction arm, so a
factory-only migration would go false-green (RESEARCH F-5). Temporarily registering the legacy
arm's plugin under `'probe-not-paper'` produced **`KeyError: 'paper'`**, failing all three
`test_backtest_oracle.py` cases at `execution_handler.py`'s memo lookup. The probe was reverted
(`git diff` clean afterwards) and the oracle returned to `3 passed`, byte-exact.

**4. The live registry holds the memo's exchange, verified at runtime.**
Booting `LiveTradingSystem.for_exchange('paper', data_provider='replay', data_plugins={...})`:

```
REGISTRY KEYS: [('paper', 'default')]
REGISTRY paper exchange: 4561782160 SimulatedExchange
BTCUSD admitted: True
LIFECYCLE default connector= None exchange id= 4560547472
registry values are all the memo exchange: True
```

`BTCUSD admitted: True` is the live-behaviour-preservation proof for `default_exchange_config()`.
The distinct `LIFECYCLE` exchange id is deviation 5 below.

**5. GATE-01 tests the REAL wiring shape, per 11.1-01's instruction.**
The probe's standalone `_connectors = ConnectorProvider({})` was replaced with the production
sequence — registry + `PaperVenuePlugin(ExchangeConfig.default())` + `ConnectorProvider({})` +
`VenueBundles(_exec_registry, _connectors, _ctx)` — all inside the ccxt-absent window. Constructing
the memo is register-only; `get` is deliberately never called. Gate green at **4 passed**.

## Deviations from Plan

**1. [Rule 1 - Bug] `ExecutionHandler`'s `rng` parameter became dead and was REMOVED (not in the plan).**
- **Found during:** Task 2, after deleting the mint.
- **Issue:** `self._rng` had exactly one reader — the `SimulatedExchange(...)` construction this plan
  deletes. Keeping the parameter would leave a constructor argument that is accepted and silently
  ignored, alongside a D-07-tagged docstring claiming "the exchange AND its slippage model draw from
  THIS object" — false the moment the plugin became the builder. That is threat T-11.1-33's shape
  (a decision-tagged docstring asserting a contract that no longer exists).
- **Fix:** removed the `rng` parameter, `self._rng`, `self._rng_seed`, `_resolve_rng_seed`, the
  two-branch init log, and the now-unused `import random`; `compose.py` stops passing `rng=`. A
  `Notes` block in the constructor docstring records why. Nothing read `_rng`/`_rng_seed` anywhere
  in `itrader/` or `tests/` (grepped before removing).
- **Determinism is unaffected and strictly better proved:** `ctx.rng` still reaches the exchange,
  now through `PaperVenuePlugin.build_bundle`, and `test_the_composed_exchange_holds_the_ctx_rng_instance`
  (11.1-04's identity assertion) was RE-POINTED, not weakened — it still asserts `is`, never `==`.
  Oracle byte-exact.
- **Commit:** `53588d36`.

**5. [Known consequence — flagged for 11.1-08, NOT fixed here] `assemble_venues` now builds a SECOND, unread paper exchange.**
- **What changed:** before this plan, `PaperVenuePlugin` returned the injected exchange, so the
  bundle `assemble_venues` built for a live paper spec carried the SAME object as
  `execution_handler.exchanges[('paper', DEFAULT_ACCOUNT_ID)]`. Now the plugin builds a fresh one per
  `build_bundle` call, and `assemble_venues` calls the registry DIRECTLY rather than through the
  memo — so a live paper boot constructs two `SimulatedExchange` objects (confirmed at runtime, ids
  `4561782160` vs `4560547472` above).
- **Why it is inert today (classified, every reader):**

  | Reader | Disposition |
  |--------|-------------|
  | `live_trading_system.py:2142` per-account registration loop | guarded by `if lifecycle.bundle.connector is None: continue` — a paper bundle is never registered |
  | `:637` venue-metadata resolution | `_streaming_lifecycles()` filters on `connector is not None`; for paper it is empty, so the else-branch reads the REGISTRY's exchange |
  | `:808` stream connect | iterates `streaming` only |
  | `:2397-2398` halt/stream-state wiring | `if bundle.connector is None: continue` |
  | `safety/stream_recovery_handler.py:120` | reached only for streaming lifecycles |

  No order, bar or fill ever reaches the second object; it holds an empty resting-order book that is
  never driven.
- **Why it was not fixed here:** the plan explicitly forbids touching `assemble_venues` or the
  registration loop ("They are plan 11.1-08's subject; touching them here couples two waves").
- **What 11.1-08 must do:** route `assemble_venues` through the SAME `VenueBundles` instance
  (`build_live_system` already has it in scope as `venue_bundles`). Until then, the risk is dormant
  rather than absent: the moment anything asks the memo for `('okx', account_id)` — which plan
  11.1-09 giving `PortfolioHandler` the memo would do — a SECOND `OkxExchange` per account would be
  built, and `OkxExchange.connect()` is the sole spawn site for `_stream_fills`/`_stream_orders`.
  That is exactly the duplicate-session defect D-08 exists to close.

**6. [Plan/reality] Task 2 could not be individually green on the test suite.**
The plan's Task 2 `<verify>` runs `tests/unit/execution`, but 19 tests there drive the LIVE root,
which the plan defers to Task 3. The intermediate commit is therefore red on those by construction
(`TypeError: compose_engine() got an unexpected keyword argument 'exchange_config'` — itself proof
that the live path genuinely goes through compose). Production gates (`mypy`, oracle, inertness)
were green at every commit; the full suite is green at Task 3. Task sequencing preserved as
authored rather than reshuffled.
