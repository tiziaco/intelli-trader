# Phase 11.1 Plan 09: Portfolio receives its account; new_account is the sole factory — Summary

`Portfolio` now takes a built `Account` and selects no account kind; `VenuePlugin.new_account(config)`
is the only margin-vs-cash owner in the tree, and `PortfolioHandler.add_portfolio` builds each
portfolio's account through the same `VenueBundles` memo `ExecutionHandler` holds.

## What was built

**Task 1 — `new_account` becomes the sole factory** (`5aa212ad`)

- `VenueAccountConfig` gains `enable_margin: bool = False` (D-03 leaf selection, consumed only by the
  paper arm) and `state_storage: Any = None` (the shared persistence seam — see Deviations).
- `VenuePlugin.new_account(self, config)` on the Protocol and both plugins; `portfolio_ref` is gone.
- `OkxVenuePlugin._account_id_for(config)` collapses to the single `config.account_id` rule. The D-11
  raise is copied verbatim; only the interpolated source word changed (`"portfolio"`/`"venue spec"` →
  `"venue account config"`). The two-branch no-cross-fallback rule did not weaken into a fallback —
  the resolution moved OUT to the caller that holds a portfolio's account id.
- Both `account_factory` closures re-signed `(*, initial_cash, enable_margin, account_id,
  state_storage)`. The 11-07 removal of the `(*args, **kwargs)` catch-all is not reverted, and is now
  pinned by a signature assertion on each plugin.
- `_attach_venue_accounts` passes the `account_id` its loop already resolved.

**Task 2 — `Portfolio` receives, `PortfolioHandler` builds** (`75d7f59c`)

- `Portfolio.__init__(..., *, account: Account)` — required, keyword-only, no default. The leaf
  selection at `_init_managers` is deleted along with the `SimulatedCashAccount` import.
- `_init_managers` ADOPTS the account's `state_storage` seam (new read-only property on
  `SimulatedCashAccount`), falling back to the environment/sql_engine factory build only for a leaf
  that exposes none.
- `_validate_initial_state(cash)` refuses an account whose opening balance disagrees with the
  portfolio's `cash` (T-11.1-42); venue-truth leaves are exempt.
- `PortfolioHandler.__init__` gains `venue_bundles` (the D-08 shared memo) and `compute_venue`.
  `add_portfolio` resolves the portfolio id, builds the state storage, computes `to_money(cash)`
  once, resolves the config that drives the leaf selection the same way `Portfolio` does, and mints
  the account via `bundle.account_factory(...)`.
- `COMPUTE_VENUE` and `DEFAULT_ACCOUNT_ID` homed once in `venues/registry.py`, re-exported from
  `execution_handler` so every existing import path is unchanged; the eight `'paper'` lookup literals
  in `ExecutionHandler` and `compose.py` now read the constant.

**Task 3 — test-tree migration** (`4ddfba0e`)

- `tests/support/venue_wiring.py::compute_account(...)` reaches the leaf THROUGH
  `PaperVenuePlugin.new_account`, never by constructing a leaf directly, so the test tree follows
  D-03's single owner automatically.
- `tests/support/venue_wiring.py::backtest_portfolio_handler(...)` injects the `VenueBundles` the
  handler now needs.
- 42 test files migrated; new `tests/unit/portfolio/test_account_ownership.py` (12 tests) pins the
  VENUE-02 ownership property.

## Deviations from Plan

### Auto-fixed issues

**1. [Rule 2 — missing critical functionality] The account would have landed on a private
state-storage backend**

- **Found during:** Task 1, reading `PaperVenuePlugin.new_account` before editing.
- **Issue:** the plan's step 2 says `new_account` should read only `initial_cash` and `enable_margin`
  off the config, and its `account_factory` signature carries only
  `(initial_cash, enable_margin, account_id)`. The real body ALSO read
  `getattr(portfolio_ref, "state_storage", None)` and forwarded it — 11.1-03 added that read
  deliberately. Following the plan literally would have dropped it, so every account
  `add_portfolio` mints would fall back to `SimulatedCashAccount`'s private
  `PortfolioStateStorageFactory.create("backtest")` backend, disjoint from the portfolio's.
- **Why it is silent:** the leaf's three containers (reserved cash, locked margin, cash-operation
  audit trail) have NO other reader in the tree, so backtest behaviour — and the byte-exact oracle —
  is unaffected. In live, `PortfolioHandler.rehydrate` calls
  `portfolio.state_storage.rehydrate(portfolio.account)` and `CachedSqlPortfolioStateStorage`
  repopulates the reservation/locked-margin caches on THAT instance; an account on a different
  instance loses every reservation across a restart with nothing red.
- **Fix:** `VenueAccountConfig.state_storage` + a fourth keyword on both `account_factory` closures;
  `PortfolioHandler.add_portfolio` builds the seam (moved up from `Portfolio._init_managers`, which
  could not build it before the account existed) and passes the SAME instance to the factory;
  `Portfolio._init_managers` ADOPTS the account's seam so one instance is structural rather than a
  convention two call sites must keep agreeing on. `SimulatedCashAccount.state_storage` exposed
  read-only (no setter — rebinding is the late-init shape this phase removes).
- **Files:** `itrader/venues/bundle.py`, `itrader/venues/paper_plugin.py`,
  `itrader/venues/okx_plugin.py`, `itrader/portfolio_handler/portfolio.py`,
  `itrader/portfolio_handler/portfolio_handler.py`, `itrader/portfolio_handler/account/simulated.py`
- **Proven by:** `test_the_account_and_the_managers_share_one_state_storage` — mutation-verified: it
  fails when the adoption is removed.

**2. [Rule 2] `Portfolio` had no guard against the account and the portfolio opening at different
cash**

- **Found during:** Task 2. The plan's T-11.1-42 mitigation is "compute `to_money(cash)` once and
  pass it to both", which is a discipline, not a guard — and a 68-site test migration is exactly
  where that discipline slips.
- **Fix:** `_validate_initial_state(cash)` compares `account.balance` against
  `to_money(cash).quantize(0.01, HALF_UP)` and raises `ValidationError`. Venue-truth leaves are
  exempt (their balance comes from the venue, not from wiring). This turned several would-be silent
  migration errors into loud construction failures.

**3. [Rule 3] Direct-construction `PortfolioHandler(...)` sites had no `VenueBundles`**

- 33 test sites construct `PortfolioHandler` directly and now need the memo. Rather than let the
  handler mint one (a handler minting its own collaborator, the shape this phase deletes) or fall
  back to an inline `SimulatedCashAccount` (a second account-kind selection), `add_portfolio` REFUSES
  with `PortfolioConfigurationError` and the test sites go through
  `tests/support/venue_wiring.backtest_portfolio_handler`.
