<objective>
Create the `VenueBundles` provider that **D-08 (VENUE-07)** specifies: one small memoized provider over
`(registry, connectors, ctx)` that both `ExecutionHandler` and `PortfolioHandler` will hold, each
asking for the view it needs — the exchange, or the account factory.

It lands as a **standalone module with no consumers yet**, so it can be unit-tested against fake
plugins in isolation and so plan 11.1-07 (which wires it into both handlers and both backtest
construction arms) does not also have to design it. That separation is deliberate: D-08's memoization
is the mitigation for a real duplicate-session defect, and the assertion that proves it — the
plugin's `build_bundle` runs exactly once across two `get`s — is easier to write and far harder to
fool against fakes than against a fully composed engine.

Purpose: give plan 11.1-07 a finished, tested collaborator to inject.

Output: `itrader/venues/bundles.py` (new, import-inert, not barrel-exported) and
`tests/unit/venues/test_venue_bundles.py` (new).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| venue name (from a spec / a portfolio) -> plugin resolution | The venue string selects which external venue's plugin builds the exchange that will submit real orders. An unregistered name must never resolve to some other venue's plugin. |
| bundle construction -> authenticated venue session | Each `build_bundle` on the OKX arm opens a `ccxt.pro` client and, on `connect()`, spawns fill/order streams. Building twice per account duplicates the session. |
