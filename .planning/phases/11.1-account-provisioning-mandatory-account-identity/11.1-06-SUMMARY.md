# Phase 11.1 Plan 06: Rename the Backtest Venue to `paper` + Collapse the Exchange Registry Summary

The backtest venue is now `paper` — one name for the simulated fill engine across backtest and
live-paper — and `ExecutionHandler.init_exchanges` returns a single `('paper', DEFAULT_ACCOUNT_ID)`
entry with the deliberate two-key aliasing and the dead `('ccxt', ...): None` placeholder both
deleted, every reader converted in the same wave, and the byte-exact oracle green at
`134 / 46189.87730727451`.

## What Landed

**Task 1 — the rename and the registry collapse (commit `b9e4877f`, 10 files).**
`init_exchanges` returns one key. `update_config` / `validate_config` re-keyed with their
`ConfigurationError(config_key=)` renamed (F-11). Both construction arms migrated — the FACTORY arm's
`add_portfolio` now passes `venue_name='paper'` explicitly alongside `exchange='paper'` (D-19), and
the LEGACY arm's default plus `curate_run_settings`' lookup moved. The oracle generator
(`scripts/run_backtest.py`, F-5's real oracle path), `scripts/crossval/trailing_run.py`, the
`compose.py` commission reach-in, and both live sites (`:638` symbol-validation fallback, `:2039`
paper-plugin reach-in) all re-keyed. Every alias-describing comment rewritten.

**Task 2 — the three guard sites (commit `6489dcd2`, 5 files).**
- **F-3:** `supported_exchanges` drops `"csv"`/`"simulated"` and admits `"paper"`. It stays
  default-deny — the membership check and the typed `UNSUPPORTED_EXCHANGE` refusal are untouched, and
  a comment now says so explicitly.
- **F-4:** `wire_universe`'s soft `.get()` + isinstance skip-guard became a fail-loud subscript. The
  isinstance check survives only as a mypy narrowing and now RAISES instead of skipping.
- Docstring examples on `add_portfolio` / `exchange_for` repointed.

**Task 3 — test migration and the inverted invariant (commit `b394b663`, 72 files).**
`test_registry_is_pair_keyed_and_aliases_share_one_object` → `test_registry_is_pair_keyed_with_no_venue_aliases`,
asserting the inverse (one entry; no two keys resolving to one object). The alias-dedup test was
re-framed onto the case that still exists — one object under two ACCOUNTS.
`test_distinct_per_account_exchanges_are_both_driven` stayed green with only its venue name changed.

The dangerous failure here is a reader that still resolves a global and succeeds anyway because, in a
single-account backtest, the global and the per-account value are the same object.

**Fail-first proof (F-4, recorded as required).** Pointing the `universe_wiring.py` lookup at
`('nonexistent-venue', DEFAULT_ACCOUNT_ID)` produced
`KeyError: ('nonexistent-venue', 'default')` at `universe_wiring.py:104`, failing BOTH
`test_universe_spans.py` **and** `test_backtest_oracle.py`. That is a double proof: the site is
genuinely fail-loud, and the site is on the oracle path. The probe was reverted (`git diff --stat`
empty afterwards) and both tests returned green.

**Oracle exercises the new routing on the legacy arm.** `test_backtest_oracle.py` calls
`scripts/run_backtest.py::main()` in-process — the LEGACY construction arm — which now builds with
`exchange="paper"` and a portfolio carrying `exchange="paper", venue_name="paper"`. The fail-first
probe above confirms the oracle actually traverses the paper key rather than passing around it.

## Deviations from Plan

**1. [Rule 3 - Blocking] `scripts/run_live_paper.py` migrated (not in the plan's file list).**
- **Found during:** Task 1, grepping every reader before editing.
- **Issue:** The plan enumerated `run_backtest.py` and `crossval/trailing_run.py` but not
  `run_live_paper.py`, which created its portfolio with `exchange="simulated"`. Once `'simulated'`
  left both the registry and `supported_exchanges`, that script's orders would be refused at
  admission and route nowhere.
- **Fix:** `exchange="paper", venue_name="paper"`, plus its module docstring and the two stale
  comments. This also makes the script agree with the live spec's venue name, which was already
  `'paper'` — the mismatch existed only because the exchange registry used a different name.
- **Commit:** `b9e4877f`.

**2. [Rule 1 - Bug] The alias-dedup case in `test_per_account_exchange_routing.py` would have
become a false-green under literal substitution.**
- **Found during:** Task 3 review of the plan's four semantically-loaded files.
- **Issue:** That test builds two keys onto one object. A mechanical `('simulated'|'csv', DEFAULT)` →
  `('paper', DEFAULT)` substitution writes the SAME key twice, and the dict literal silently collapses
  to one entry. The test would pass with the dedup never running.
- **Fix:** Re-expressed as two ACCOUNTS (`('paper', 'acct-1')` / `('paper', 'acct-2')`) per the plan's
  item (c)2, plus an explicit `len(handler.exchanges) == 2` premise guard so the collapse cannot recur.
- **Commit:** `b394b663`.

**3. [Rule 2 - Missing critical] `test_universe_spans.py` had no direct universe-injection assertion.**
- **Found during:** Task 2 acceptance check ("at least one of those files asserts that the exchange
  resolved by `wire_universe` received the universe — add the assertion if absent"). It was absent
  from both named files.
- **Fix:** Added an IDENTITY assertion (`simulated._universe is system.engine.universe`) after the run.
  Identity rather than equality: an equal-but-distinct `Universe` would mean two objects were built and
  the exchange holds the wrong one.
- **Commit:** `b394b663`.

**4. [Plan/code discrepancy — code wins] The allowlist test needed three edits, not one.**
- The plan's acceptance criterion said the diff should touch "ONLY the parametrize list and its
  comment". The file also carried `"exchange": "csv"` in its order fixture and
  `_validator_reporting_venue("csv")` in the structural default-deny test. Both are venue-meaning and
  would have left a retired name in the file. Updated; the allowlist was repointed, never widened —
  the unknown-venue refusal, empty-name refusal and default-deny structural tests all pass UNCHANGED
  in substance.

**5. [Plan/code discrepancy — code wins] Line numbers and one count.**
- The plan cited `init_exchanges` at `:269-330` with the dict literal at `:295-304`; the real file had
  them at `:302-362` and `:328-337`. The plan cited `compose.py:239-240`, `backtest_trading_system.py:395`,
  `live_trading_system.py:638`/`:2028` — actual `:246`, `:405`, `:639`, `:2035`. Implemented against the
  real file in every case.
- The plan predicted "27 venue-name files / 26 registry-key files"; the tree yielded 52 files matching
  the venue patterns plus a further ~20 positional-argument sites the plan's greps did not cover
  (`Portfolio("pf", "simulated", ...)`, `Order.new_order(signal, "simulated")`,
  `add_portfolio("p", "simulated", cash)`). All were migrated — a narrower sweep would have left
  admission-refusing venues behind.

**6. [Plan/reality] Task 1 and Task 2 could not be individually green on the test suite.**
- The plan's Task 1 `<verify>` runs `pytest tests/unit/execution`, but the plan itself defers those
  test files to Task 3. The intermediate commits are therefore red on the migrated test sites by
  construction. Production gates (`mypy`) were green at every commit, and the full suite is green at
  the end of Task 3. Sequencing preserved as authored rather than reshuffling the plan's task split.

**7. Comment truthfulness beyond the plan's enumeration.** Task 1 step 9 listed four comment sites.
Two more described the compose-built exchange as `'simulated'` — `venues/paper_plugin.py` (module
docstring, class docstring, constructor comment) and `venues/bundle.py`. Both rewritten, since a stale
comment is a claim the next reader trusts.

## Follow-ups for Later Plans

- **Plan 11.1-07** deletes the `live_trading_system.py:2039` reach-in entirely when `PaperVenuePlugin`
  builds its own `SimulatedExchange`. The key it reaches for — `('paper', DEFAULT_ACCOUNT_ID)` — already
  exists and resolves, which is the ordering inversion this plan exists to enable.

  as an `exchange_type` KIND field and never as a routing venue name. The name now reads as a venue
  synonym that D-05 retired. Left alone deliberately — renaming a config enum member is out of this
  plan's scope and has its own migration surface.
