# Phase 11.1 Plan 02: Wave-0 Validation Guards Summary

Installed the two admission-side regression guards VALIDATION.md flagged as missing entirely — a
default-deny venue-allowlist guard and a non-zero-fee commission-reservation guard — both green
against the current tree and each proven to go red when the control it guards is removed.

## What Was Built

### Task 1 — Venue allowlist pinned as a default-deny control (VENUE-04 / D-05 / D-19 / F-3)

`tests/unit/order/test_order_validator_venue_allowlist.py` (6 collected tests, 4 test functions with
one parametrized over 3 venues). Calls `EnhancedOrderValidator._validate_exchange_support` directly
so the verdict is unambiguously the venue check and not price/quantity/market-hours noise.

- `test_backtest_venue_names_are_admitted` — parametrized over `csv` / `default` / `simulated`.
  Goes red on a MISSED D-05 rename, which today would silently refuse every SMA_MACD signal.
- `test_unknown_venue_is_refused_with_the_typed_error_code` — asserts exactly one message, at
  `ValidationLevel.ERROR`, `field == "exchange"`, `code == "UNSUPPORTED_EXCHANGE"`. Goes red if the
  check is deleted or downgraded to a WARNING.
- `test_empty_venue_name_is_refused` — the VENUE-04 `empty` edge; the empty string is falsy, so any
  truthiness shortcut would leak an unset venue through.
- `test_allowlist_is_default_deny_not_accept_all` — structural companion; goes red if the allowlist
  is widened to accept-all or emptied into a check that cannot fire.

The VENUE-04 `ordering` edge is satisfied structurally and documented in the last test's docstring:
`supported_exchanges` is a `set`, so no iteration order is observable to any caller.

### Task 2 — Commission reservation pinned under a non-zero fee model (VENUE-08 / D-18 / Pitfall 2)

`tests/unit/order/test_admission_commission_reservation.py` (5 tests). Every assertion uses a
NON-ZERO fee, which is the entire point: `Decimal("0")` is both the correct golden value (the oracle
pins `ZeroFeeModel`) and the value a structurally-dead seam returns, so `oracle green` cannot tell
them apart.

- `test_non_zero_commission_reaches_the_reservation` — the value-identity anchor (`Decimal("7.5")`)
  plan 11.1-10 must reproduce byte-for-byte.
- `test_absent_estimator_degrades_to_zero` — the `None` guard-clause early exit.
- `test_float_returning_estimator_is_normalised_to_decimal` — WR-04 / VENUE-08 `precision`: a float
  `10.1` yields `Decimal("10.1")`, the string-path value, never the binary-float-repr artifact.
- `test_admission_convention_is_buy_market` — pins `side="buy", order_type="market"` over a REAL
  `SimulatedExchange` (a duck-typed double would be rejected by the `isinstance` guard into a silent
  zero, making the test vacuous).
- `test_fee_model_swap_is_observed_on_the_next_call` — VENUE-08 `idempotency` (two derefs of an
  unchanged model agree) and `concurrency` (a deref after a hot-swap sees the swap), i.e. the D-18
  late-binding property.

## Deviations from Plan

### 1. [Plan factual drift — no code change] Probe B does not red test 5

**Found during:** Task 2 fail-first proof.
**Plan claim:** acceptance criteria stated that forcing `_estimate_commission` to return
`Decimal("0")` "makes tests 1, 3 and 5 FAIL".
**Actual:** it reds tests 1 and 3 only. Test 5 (`test_fee_model_swap_is_observed_on_the_next_call`)
drives `FeeModelCommissionEstimator` directly — the subject the plan's own action block specified for
tests 4 and 5 — and never routes through `AdmissionManager._estimate_commission`, so no change to
that method can affect it.
**Resolution:** the plan's INTENT (every guard must be proven able to go red) was honoured by
probing the seam each test actually exercises: probes C1 and C2 red tests 5 and 4 respectively. No
production or test code was changed to match the plan's prediction — the code wins.

### 2. [Plan factual drift — no code change] mypy source-file count is 259, not 281

**Found during:** plan-level verification.
**Plan claim:** `poetry run mypy` → `Success: no issues found in 281 source files`.
**Actual:** `Success: no issues found in 259 source files`.
**Resolution:** the load-bearing half of that criterion is "unchanged — `tests/` is outside mypy's
`files` setting, so the count must not move", and that holds absolutely: `git diff --stat` against
the base commit shows only `tests/` paths, and mypy's `files = ["itrader"]`, so no input to mypy was
touched. No action taken.

> **ORCHESTRATOR CORRECTION (post-merge, Wave 1).** The literal `281` is **not** stale — it is the
> **main-checkout** count, and post-merge mypy on the main checkout confirms **282 source files**
> (281 baseline + the one module 11.1-05 added). `259` is the **worktree** count; the 22-file gap is
> `itrader/strategy_handler/my_strategies/`, gitignored and therefore absent from every worktree
