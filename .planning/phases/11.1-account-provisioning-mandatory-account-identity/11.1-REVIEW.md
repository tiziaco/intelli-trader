# Phase 11.1: Code Review Report

**Reviewed:** 2026-07-22T20:45:00Z
**Depth:** standard

## Summary

The construction inversion the phase set out to do is real and mostly holds: `VenuePlugin.new_account` is the only account-minting site left in `itrader/` (verified tree-wide), `VenueBundles` is genuinely memoized and shared by both handlers, `EngineContext.rng` is required and injected on both backtest arms, `'simulated'`/`'csv'` survive nowhere as routing keys, `mypy --strict` is clean over 282 files, and the byte-exact oracle (`tests/integration/test_backtest_oracle.py`) is green.

What the phase did **not** verify is that the venue-identity rules it hardened are actually satisfiable by the paths it shipped. Three end-to-end paths are dead:

* the shipped live-paper worker (`scripts/run_live_paper.py`) creates account-less portfolios, so `ExecutionHandler.on_order`'s loud refusal fires on every order — **demonstrated: 0 trades, final equity == starting cash**;
* the validator allowlist edited in this phase repointed `csv`/`simulated` → `paper` but still does not contain `okx`, so every live OKX order is refused at admission;
* the account KIND is now selected once, before the portfolio exists, while `enable_margin` remains runtime-mutable through `Portfolio.update_config` — a runtime or persisted flip leaves a cash leaf on the margin settlement path.

The remaining warnings are concentrated in the new venue substrate: a venue-blind fee-model provider whose in-code justification is factually wrong, a routing key that now has five string homes behind a false import-inertness rationale, an account factory that can key a connector for account B from account A's credentials, and an adopt-the-account's-seam inversion that makes `Portfolio(environment="live", ...)` silently in-memory.

## Critical Issues

### CR-01: The shipped live-paper worker creates account-less portfolios — every order is refused, zero trades

**File:** `scripts/run_live_paper.py:79-84` (with `itrader/execution_handler/execution_handler.py:219-229`, `itrader/trading_system/live_trading_system.py:1985`)

**Issue:** `build_live_system` passes `route_orders_by_account=True`, so `ExecutionHandler._resolve_account_id` reads `PortfolioReadModel.account_for(portfolio_id)` and refuses the order outright when it is `None` (correctly — no default-account fallback). `_compose` creates its portfolio with `exchange="paper"`, `venue_name="paper"` and **no `account_id`**, so `Portfolio.account_id is None` and every order is dropped.

This is demonstrable, not theoretical. Running the documented CI-safe entrypoint:

```
$ poetry run python scripts/run_live_paper.py --mode replay
... error [itrader.ExecutionHandler] Portfolio 019f8b83-... names no venue account —
    order BTCUSD Side.BUY NOT submitted (no default-account fallback: ...)
Paper replay complete — trades: 0, final equity: 10000.0
```

The same `_compose` is used by `--mode okx`, so the manual live smoke submits nothing either. The paper-replay parity harness is unaffected only because `tests/integration/test_paper_parity.py:136-139` passes `account_id=DEFAULT_ACCOUNT_ID` by hand — i.e. the test knows the rule the shipped script does not, so the suite stays green while the shipped worker is inert. That is exactly the "green suite, dead path" shape this phase claims to close.

**Fix:**
```python
from itrader.execution_handler.execution_handler import DEFAULT_ACCOUNT_ID
...
    portfolio_id = system.portfolio_handler.add_portfolio(
        name="paper_pf",
        exchange="paper",
        venue_name="paper",
        account_id=DEFAULT_ACCOUNT_ID,   # live routes by (venue, account_id); None == refused
        cash=CASH,
    )
```
Additionally, make the failure loud rather than silent: the worker should assert `trades > 0` (or at minimum that no order was refused) before printing "Paper replay complete", so an account-less composition can never again report success with an empty trade log.

### CR-02: Account KIND is fixed at construction while `enable_margin` stays runtime-mutable — a margin flip breaks settlement for every subsequent fill

**File:** `itrader/portfolio_handler/portfolio_handler.py:348-354`, `itrader/portfolio_handler/portfolio.py:307-321`, `itrader/portfolio_handler/portfolio.py:448-451`, `itrader/portfolio_handler/portfolio.py:540-560`

**Issue:** `add_portfolio` resolves `resolved_config.trading_rules.enable_margin` ONCE and hands it to `VenuePlugin.new_account`, which mints either `SimulatedCashAccount` or `SimulatedMarginAccount`. `Portfolio.process_transaction` then branches on `self.config.trading_rules.enable_margin` **at fill time**, and `self.config` is atomically swapped by `Portfolio.update_config`.

`enable_margin` is a routable `TradingRules` field (`itrader/config/portfolio.py:72`), so both of these flip it after the account exists:

* `ConfigRouter._apply_portfolio` (`itrader/trading_system/config_router.py:340-383`) — an operator `CONFIG_UPDATE` on `portfolio:{id}` / `enable_margin`;
* `_layer_persisted_overrides` (`itrader/trading_system/live_trading_system.py:1361-1365`), which applies each portfolio's persisted config **after** `rehydrate_portfolios` has already minted the account (rehydrate at ~line 2073, layering at ~line 2289).

After the flip, `process_transaction` takes `_process_transaction_margin`, `_require_margin_account` sees a `SimulatedCashAccount` and raises `StateError`. In live the publish-and-continue policy swallows it, so the fill is never settled: position and cash silently diverge from venue truth while the order mirror and the venue both believe the trade happened. This phase deleted the selection branch from `Portfolio` (D-03) without adding the compensating invariant, so nothing anywhere reconciles the two.

**Fix:** Make the mismatch unrepresentable rather than latent. Either (a) reject the flip, or (b) re-mint at flip time. (a) is the smaller change and matches the fail-loud posture of the rest of the phase:

```python
# Portfolio.update_config, before the atomic swap
if (new_config.trading_rules.enable_margin
        != self.config.trading_rules.enable_margin):
    raise ConfigurationError(
        reason="enable_margin selects the account LEAF KIND at construction "
               "(VenuePlugin.new_account, D-03) and cannot be changed at runtime "
               "on a live portfolio; recreate the portfolio instead")
```
and add the same guard in `ConfigRouter._resolve_portfolio_section` (drop `enable_margin` from the routable set) so the reject happens at ingress with a reason code rather than as a swallowed exception.

### CR-03: `okx` is absent from the order validator's default-deny venue allowlist — every live OKX order is refused at admission

**File:** `itrader/order_handler/order_validator.py:125` (check at `:321-332`, verdict at `:165-169`)

**Issue:** The allowlist edited by this phase is `{"NYSE", "NASDAQ", "BINANCE", "OANDA", "default", "paper"}`. `_validate_market_conditions` reads `portfolio_handler.exchange_for(portfolio_id)`, which for a live OKX portfolio returns `"okx"` (`Portfolio.exchange` is derived from `venue_name`). A non-member yields a `ValidationLevel.ERROR` (`UNSUPPORTED_EXCHANGE`), and `validate_order_pipeline` short-circuits `PHASE 2` with `ValidationResult(False, ...)` — so no `OrderEvent` is ever emitted for an OKX portfolio. There is no second source for this set (`supported_exchanges` has exactly one assignment tree-wide) and it is not derived from the venue registry.

The rename pass repointed `csv`/`simulated` → `paper` and added a "must NEVER be widened … Repoint it; do not weaken it" comment, which now reads as an assertion that the set is complete. It is not: the one live venue the milestone ships cannot pass admission.

**Fix:** Derive the allowlist from the venue registry the phase just made authoritative, instead of maintaining a second hand-written list:

```python
# EnhancedOrderValidator.__init__ — inject the registered venue names
def __init__(self, ..., supported_venues: Optional[set[str]] = None):
    ...
    self.supported_exchanges = (
        supported_venues
        if supported_venues is not None
        # fall back to the shipped set; 'okx' is a REGISTERED execution venue and
        # must be admissible or every live order is refused at PHASE 2.
        else {"NYSE", "NASDAQ", "BINANCE", "OANDA", "default", "paper", "okx"})
```
The registry-derived form (`set(exec_registry.names()) | {"default", ...}`) keeps the default-deny property while making a venue rename structurally impossible to get half-done, which is the failure mode that produced this gap.

## Warnings

### WR-01: The admission fee-model provider is hard-bound to the paper venue in live, and the in-code justification for it is factually wrong

**File:** `itrader/trading_system/compose.py:258-278`, `itrader/trading_system/live_trading_system.py:1990-1995`

**Issue:** `compose_engine` resolves `admission_exchange = execution_handler.exchanges.get((COMPUTE_VENUE, DEFAULT_ACCOUNT_ID))` once and the `fee_model_provider` closure reads `fee_model` off that one object for **every** order, regardless of the order's venue or account. `ExecutionHandler.init_exchanges` always resolves the paper bundle (even on a live OKX boot), so `admission_exchange` is always the `SimulatedExchange`.

The comment at `live_trading_system.py:1990-1995` states: *"the live OKX exchange exposes no `fee_model` attribute, so the provider yields None here"*. That is not what happens — the provider never looks at the OKX exchange at all; it returns the **paper** exchange's `ZeroFeeModel`. The observed reservation is `Decimal("0")` today only because `default_exchange_config()` pins `FeeModelType.ZERO`. Concrete consequence: a future "fix" that gives `OkxExchange` a `fee_model` will change nothing, and a `venue:` scope fee update that mutates the paper exchange (`ExecutionHandler.update_config` is venue-blind and always targets `(COMPUTE_VENUE, DEFAULT_ACCOUNT_ID)`) will silently move OKX admission reservation amounts.

**Fix:** Make the provider account-aware, or at minimum correct the recorded contract. Minimal honest form:
```python
def fee_model_provider(venue: str = COMPUTE_VENUE,
                       account_id: str = DEFAULT_ACCOUNT_ID) -> "Optional[FeeModel]":
    return cast("Optional[FeeModel]",
        getattr(execution_handler.exchanges.get((venue, account_id)), "fee_model", None))
```
and replace the `build_live_system` note with the true statement: *the provider is bound to the paper venue, so a live order's fee headroom is estimated from the paper fee model, not the venue's.*

### WR-02: The `"default"` account-id routing key now has five homes, behind an import-inertness rationale that is false

**File:** `itrader/venues/bundles.py:53-57`, `itrader/venues/assemble.py:81-85`, `itrader/venues/venue_uid_guard.py:51-55`, `itrader/venues/okx_plugin.py:303`, `itrader/venues/okx_plugin.py:376`

**Issue:** `venues/registry.py:62` was made the declared home of `DEFAULT_ACCOUNT_ID`, yet `bundles.py` and `assemble.py` each re-declare a private `_DEFAULT_ACCOUNT_ID = "default"` with the comment *"Declared locally rather than imported … so this module keeps its zero-dependency import-inertness posture."* Both modules already perform heavier runtime imports than `venues.registry` would cost: `bundles.py:47` imports `itrader.logger`, and `assemble.py:71` imports `itrader.venues.lifecycle` (which itself pulls the logger and `venue_uid_guard`). `venues/registry.py` has **no runtime imports at all** — it is the cheapest import in the package. The stated justification therefore does not hold, and the phase has left the routing key duplicated in five places (plus two inline `spec.account_id or "default"` literals in `okx_plugin`) immediately after paying to eliminate exactly this class of drift for `'simulated'`/`'csv'`.

**Fix:** `from itrader.venues.registry import DEFAULT_ACCOUNT_ID` in `bundles.py`, `assemble.py` and `venue_uid_guard.py`, and use it for the two `okx_plugin` normalizations. Delete the three private copies.

### WR-03: `OkxVenuePlugin`'s `account_factory` overrides `account_id` but keeps the bundle's `spec` — a mismatched id keys a connector from another account's credentials

**File:** `itrader/venues/okx_plugin.py:320-348` (with `itrader/venues/okx_plugin.py:276`, `itrader/connectors/provider.py:78-86`)

**Issue:** The factory does `replace(account_config, account_id=account_id or account_config.account_id, ...)` but leaves `account_config.spec` — the spec the **bundle** was built for — untouched. `new_account` then calls `config.connectors.get("okx", account_id, config.spec)`. If `account_id` is not the bundle's own id and that key is not yet memoized, `ConnectorProvider.get` builds a connector via `OkxConnectorPlugin.build(spec_of_account_A)` and memoizes it under `("okx", "B")` — account B's session authenticated with account A's `secret_ref`. That is precisely the cross-account misroute D-11/D-12 exist to close, and the method docstring's assurance ("A SUPPLIED id always wins and is never widened") describes only half the state that has to move together.

Not currently triggered: `_attach_venue_accounts` always looks the lifecycle up by the same id it passes. But the factory is a public bundle field with no guard, so the invariant rests on caller discipline rather than structure — the exact posture this phase elsewhere rejects.

**Fix:**
```python
def account_factory(*, initial_cash=0.0, enable_margin=False,
                    account_id=None, state_storage=None) -> Account:
    if account_id is not None and account_id != account_config.account_id:
        from itrader.core.exceptions import ValidationError
        raise ValidationError(
            "account_id", str(account_id),
            f"This bundle was built for account '{account_config.account_id}' with "
            f"that account's credentials; minting '{account_id}' from it would key a "
            "connector under one account using another account's secret_ref (D-11/D-12)")
    ...
```

### WR-04: `Portfolio` adopts the account's seam unconditionally, so its own `environment`/`sql_engine` selector is unreachable — a live portfolio can silently run in-memory

**File:** `itrader/portfolio_handler/portfolio.py:205-213`, `itrader/portfolio_handler/account/simulated.py:161-165`

**Issue:** `_init_managers` does `account_seam = getattr(account, "state_storage", None)` and uses the factory fallback only when that is `None`. But every simulated leaf ALWAYS exposes `state_storage`: when none is passed, `SimulatedCashAccount.__init__` builds `PortfolioStateStorageFactory.create("backtest")` — hardcoded to the in-memory backend. Since the only account kind that exists at portfolio-construction time is a compute leaf (`_new_compute_account` always asks `COMPUTE_VENUE`), the documented `environment`/`sql_engine` fallback branch in `_init_managers` is **dead in production**, and the two constructor parameters are inert for any caller that does not route through `PortfolioHandler.add_portfolio`.

Failure scenario: `Portfolio(name=..., environment="live", sql_engine=engine, account=SimulatedCashAccount(10_000))` produces a portfolio that reports itself as live and persists nothing — no durable ledger, no restart restore, no exception, and (per the phase's own analysis) invisible in backtest because nothing else reads those containers.

**Fix:** Make the mismatch loud instead of silently preferring the leaf's private backend:
```python
account_seam = getattr(account, "state_storage", None)
if account_seam is None:
    ... factory fallback (venue-truth leaves) ...
elif self._environment != "backtest" and not _is_durable(account_seam):
    raise ConfigurationError(
        reason=f"portfolio environment={self._environment!r} but the supplied account "
               "carries an in-memory state_storage; the seam must be built by the "
               "caller and passed to VenuePlugin.new_account (D-01/D-03)")
```
Alternatively make `SimulatedCashAccount.state_storage` a required keyword with no default, which removes the silent-default path entirely (see WR-09).

### WR-05: The opening-cash guard re-implements the account's quantize as a literal on the far side of the boundary

**File:** `itrader/portfolio_handler/portfolio.py:233-242`, `itrader/portfolio_handler/account/simulated.py:139`, `itrader/portfolio_handler/account/simulated.py:170`

**Issue:** `_validate_initial_state` computes `to_money(cash).quantize(Decimal('0.01'), rounding=ROUND_HALF_UP)` and requires exact equality with `account.balance`. The account produced that balance with its own independent literal `Decimal('0.01')` in `__init__` — which itself ignores the account's declared `self.precision = Decimal('0.01')` set nine lines later. Three copies of one money scale, two of them magic literals, spanning a domain boundary, guarding an exact-equality check.

Failure scenario: changing the cash scale (or making the account honour `self.precision`) on one side makes `_validate_initial_state` raise `ValidationError` on **every** portfolio construction, including the oracle path — a total outage from a one-line scale change.

**Fix:** Have the guard ask the account rather than re-derive: compare against `account.balance` using the account's own rounding, e.g. expose a `quantize_cash(value)` helper on `Account` (or reuse `core.money.quantize(value, instrument=None, kind="cash")`) and call it from both `SimulatedCashAccount.__init__` and `_validate_initial_state`.

### WR-06: The bundle and connector memos are check-then-set with no lock, while `add_portfolio` is documented as a post-boot application API

**File:** `itrader/venues/bundles.py:83-98`, `itrader/connectors/provider.py:78-86`, `itrader/portfolio_handler/portfolio_handler.py:418-434`

**Issue:** `VenueBundles.get` is `if key not in self._memo: self._memo[key] = self._registry.get(venue).build_bundle(...)`. `build_bundle` is a long, I/O-adjacent call (it constructs an exchange and, for OKX, resolves credentials and memoizes a connector), so the window between the membership test and the assignment is wide. `ConnectorProvider.get` has the identical shape.

This phase newly routes `PortfolioHandler.add_portfolio` through `VenueBundles.get`, and `PortfolioHandler.__init__` documents that "live portfolios are added by the application after `build_live_system` returns". If any such caller runs off the engine thread (a FastAPI request handler is the stated destination — see the project's own FastAPI plan), two concurrent `add_portfolio` calls can each build a bundle for `('paper', 'default')` — two `SimulatedExchange` objects, two resting-order books — and on a venue arm, two `OkxExchange`/connector pairs for one authenticated account, which is the exact double-`_stream_fills` defect D-08 exists to prevent.

**Fix:** Guard both memos with a `threading.Lock` (or `RLock`, since `build_bundle` re-enters `ConnectorProvider.get`) around the build, keeping the fast path a plain dict read:
```python
def get(self, venue: str, account_id: str, spec: Any) -> VenueBundle:
    key = (venue, account_id)
    bundle = self._memo.get(key)
    if bundle is not None:
        return bundle
    with self._lock:                       # re-check under the lock
        if key not in self._memo:
            self._memo[key] = self._registry.get(venue).build_bundle(
                self._ctx, spec, self._connectors)
        return self._memo[key]
```
If instead the single-writer contract is meant to cover `add_portfolio`, say so at the method and reject off-engine-thread calls — but do not leave the invariant undocumented and unenforced.

### WR-07: Hardcoded `('paper', DEFAULT_ACCOUNT_ID)` routing keys survive the rename, one of them as a silent `.get`

**File:** `itrader/trading_system/universe_wiring.py:104-105`, `itrader/trading_system/live_trading_system.py:637-639`, `itrader/trading_system/backtest_trading_system.py:432-433`

**Issue:** `venues/registry.py:29-42` pins `COMPUTE_VENUE` as the single home of the `'paper'` name precisely because "a second literal … is how the two arms end up asking the bundle memo for different venues". Three lookups still spell it literally. Worse, they are inconsistent with each other: `universe_wiring` was hardened this phase into a fail-loud subscript, while `live_trading_system.py:637-639` still uses `.get(('paper', DEFAULT_ACCOUNT_ID))` — so a future rename yields `venue_exchange = None`, which is then passed into `SessionInitializer` and silently degrades venue metadata (`validate_symbol` / `resolve_precision`) with no exception, exactly the silent-corruption shape `universe_wiring` was fixed to avoid.

**Fix:** Import `COMPUTE_VENUE` at all three sites and use it as the key half. In `live_trading_system.py`, promote the `.get` to a subscript (or an explicit `is None` raise) so the fallback path fails as loudly as the backtest one.

### WR-08: `PaperVenuePlugin.new_account` reads every input through `getattr(..., default)`, including the state-storage seam

**File:** `itrader/venues/paper_plugin.py:142-148`, `itrader/venues/bundle.py:102-110`

**Issue:** The sole account factory resolves its inputs as `getattr(config, "initial_cash", 0.0)`, `getattr(config, "state_storage", None)`, `getattr(config, "enable_margin", False)` — and `VenueAccountConfig` itself defaults `state_storage: Any = None`. There is no fail-loud anywhere on the compute arm for an omitted seam, so a caller that forgets `state_storage=` gets a leaf on a private in-memory backend, which `Portfolio._init_managers` then **adopts** (WR-04). The phase's own docstrings identify this as the headline defect class ("a leaf on a private backend would silently lose every reservation across a restart while a backtest … stayed byte-exact"), yet the only thing preventing it is that one call site currently remembers to pass the argument. Contrast the venue arm, which does raise (`_account_id_for`).

Note also that `getattr(config, "initial_cash", 0.0)` silently substitutes a float `0.0` for missing cash — a money value entering via a default rather than `to_money`.

**Fix:** Require the seam on the compute arm the way the venue arm requires the id:
```python
state_storage = getattr(config, "state_storage", None)
if state_storage is None:
    from itrader.core.exceptions import ValidationError
    raise ValidationError(
        "state_storage", "None",
        "Cannot mint a compute account without the owning portfolio's shared "
        "PortfolioStateStorage seam: a private backend loses every reservation "
        "across a live restart while backtest stays byte-exact (D-01/D-03)")
```
and read `config.initial_cash` / `config.enable_margin` directly (the argument is a typed frozen dataclass — `getattr` with a default buys nothing and hides a wiring omission).

### WR-09: Dead code and stale references left behind by the phase's own deletions

**File:** `itrader/venues/bundles.py:47,81`; `itrader/venues/assemble.py:184`; `itrader/portfolio_handler/account/conformance.py:6`; `itrader/execution_handler/execution_handler.py:6`; `itrader/order_handler/order_handler.py:2,14`; `itrader/order_handler/order_manager.py:16`; `itrader/portfolio_handler/portfolio.py:2`

**Issue:**
* `VenueBundles.logger` is constructed (`bundles.py:81`) and never read — the module imports `itrader.logger` solely for a dead attribute, which is also the import that falsifies its own "zero-dependency import-inertness posture" comment (WR-02).
* `venues/assemble.py:184` and `portfolio_handler/account/conformance.py:6` both still document the factory as `lifecycle.bundle.account_factory(portfolio)` — the positional-portfolio signature this phase deleted. `conformance.py` is the module whose stated purpose is to keep that wiring honest under `mypy --strict`, so its description of the contract being wrong is worse than an ordinary stale comment.
* Unused imports: `FillEvent` (`execution_handler.py:6`), `Callable` and `PortfolioUpdateEvent` (`order_handler.py:2,14`), `Callable` (`order_manager.py:16`), `List` (`portfolio.py:2`). These predate the phase but sit in files it rewrote; `live_trading_system.py` is under a `[[tool.mypy.overrides]]` `ignore_errors` block, so nothing but review catches leftovers there.

**Fix:** Delete `VenueBundles.logger` and the now-unneeded `get_itrader_logger` import (or use it — the fail-loud `KeyError` path could reasonably log the registered venue names). Update both `account_factory(portfolio)` references to the keyword-only signature. Remove the five unused imports.

### Decision: CR-02 — reject the flip

**Resolution:** drop `enable_margin` from the reachable routable key set in `ConfigRouter`, and
raise in `Portfolio.update_config` when the incoming config would change it. Operators recreate
the portfolio instead of flipping the flag.

**Rationale.** `enable_margin` is read at **construction** by five separate collaborators:

1. `AdmissionManager._enable_margin` — the leverage gate and the reservation math (`compose.py:318`);
2. `EnhancedOrderValidator.enable_margin` (`order_manager.py:126`), used by the cash-vs-cost check at `:528`;
3. `ManagedStrategies.enable_margin` — the SHORT-01 / D-07 short-selling gate;
4. the account leaf KIND (new in 11.1 — this is the mismatch the review reported);
5. `Portfolio.process_transaction` at fill time (`portfolio.py:448`).

`ConfigRouter._apply_portfolio` calls only `portfolio.update_config`. A runtime flip therefore
already desynchronizes **four of the five** — and that is true *today*, before 11.1 touched
anything. The account-leaf mismatch CR-02 describes is the newest and loudest symptom of a
pre-existing defect, not the whole of it.

That reframing is what settles the choice. Making the flag genuinely mutable means propagating
the change to all five collaborators (three of which have no re-configuration path at all).
Rejecting the flip is a far smaller change and matches the fail-loud posture the rest of the
phase adopted. Queued for **Phase 11.2**.

Tracked at `.planning/todos/pending/enable-margin-runtime-flip-vs-fixed-account-kind.md`.

### Decision: WR-04 + WR-08 — required keyword (one decision, two sites)

These are not two findings; they are one decision applied at two sites.

**Resolution:**

- make `state_storage` a **required keyword argument** on `SimulatedCashAccount` and
  `SimulatedMarginAccount`, and delete the
  `if state_storage is None: PortfolioStateStorageFactory.create("backtest")` fallback;
- fail loud in `PaperVenuePlugin.new_account`, reading `config.initial_cash` /
  `config.enable_margin` **directly** rather than through `getattr` defaults — as WR-08 notes,
  `getattr(config, "initial_cash", 0.0)` silently substitutes a **float** for missing cash and
  bypasses `to_money`.

**Rationale.** The alternative the review sketched — a durability guard inside
`Portfolio._init_managers` — was rejected for two reasons.

First, that guard needs an `_is_durable` predicate that **does not exist**. There is no
durability marker on `PortfolioStateStorage`, so writing one requires either importing a
storage concretion inside `Portfolio` (cutting directly against the layering) or adding a new
`is_durable` property to the base and all three implementations.

Second, even with the predicate, the guard only catches the mistake downstream: the in-memory
default stays reachable for any caller that does not route through `Portfolio`. The required
keyword removes the trap rather than catching what falls into it.

**Blast radius (verified).** Six leaf constructions omit the kwarg, all in three test files —
`tests/unit/portfolio/test_cash_manager.py:39,50`;
`tests/unit/portfolio/test_portfolio_handler.py:746,770,782`;
`test_wr04_lock_fits_buying_power.py:70`. The sole production construction site,
`paper_plugin.py:145-148`, already passes it.

**Ordering constraint — this must land BEFORE 11.2's ACCT-05.** ACCT-05 ("attach the venue
account on every portfolio creation path") *adds* creation paths. Landing the required kwarg
first means the new paths structurally cannot forget the seam. Landing it after means adding
paths through an open trap and then closing it behind them.

The required-kwarg fix is also a down-payment on the narrow-port shape described in
`.planning/todos/pending/account-reservation-ledger-narrow-port.md`, not throwaway work — the
port would be a required constructor argument regardless.

### Decision: WR-01 — venue-aware provider + correct the comment

**Resolution:** make the `fee_model_provider` closure take `(venue, account_id)`, and replace
the false comment at `live_trading_system.py:1990-1995`.

**Rationale.** The numbers are unchanged today — OKX yields `None`, paper yields
`ZeroFeeModel`, and both are zero — so the change is oracle-safe. What it buys is breaking the
coupling whereby a venue-scoped fee update aimed at the paper exchange silently moves **OKX**
admission reservations (`ExecutionHandler.update_config` is venue-blind and always targets
`(COMPUTE_VENUE, DEFAULT_ACCOUNT_ID)`). It also means a future `OkxExchange.fee_model` takes
effect the moment it exists, whereas today nothing would read it.

The comment claiming *"the live OKX exchange exposes no `fee_model`, so the provider yields
None here"* must be replaced with the truth: **the provider never looks at the OKX exchange at
all — it returns the paper exchange's fee model.** Verified: `OkxExchange` has no `fee_model`
attribute, and `ExchangeConfig.default()` pins `FeeModelType.ZERO` (`config/exchange.py:190`),
which is the *only* reason the reservation is `Decimal("0")` today.

Giving `OkxExchange` a real fee model was considered and **rejected for this scope**: it changes
live reservation amounts, and belongs with the work that makes live OKX tradeable rather than
with a review closure. Queued for **Phase 11.2**; tracked at
`.planning/todos/pending/fee-model-provider-venue-blind.md`.

### Decision: WR-03 — guard, not support

Implemented in `95b2929c`; recorded here because it deliberately **overrode two existing tests**.

Both tests asserted the behaviour the guard now forbids, and both carried D-11 rationale
docstrings — one of them asserting outright that a connector keyed under `acct-b` is built from
`acct-a`'s spec.

**Rationale.** D-11's stated concern is the closure preferring its *bundle's* id over a
*supplied* one, which would attach a portfolio to a different real venue balance while every
lookup above it still looked correct. **Refusing a mismatch is not widening**, so D-11's
rationale survives intact; what changed is only the assertion those two tests chose for a case
that is unreachable today.

Supporting cross-account minting was not implementable in any case: the bundle holds only its
own account's spec, so there is no other spec to substitute. The guard was verified unreachable
on every current call path before being written, which makes it a latent-hazard guard rather
than a live behaviour change.

It was deliberately **not** mirrored onto `paper_plugin.py`, where minting under an arbitrary
portfolio `account_id` is legitimate and the same guard would refuse every non-default portfolio.
