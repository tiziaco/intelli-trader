# Phase 11.1 Plan 04: EngineContext carries the determinism RNG — Summary

`rng: random.Random` is now a REQUIRED field on `EngineContext`, built once per run at the
wiring seam from `int(config.rng_seed)` and injected through `compose_engine` into
`ExecutionHandler` — which receives rather than derives it — so the venue plugin arriving in
plan 11.1-07 can reach the single shared instance without touching a handler-private attribute.
The oracle is byte-exact and the change is proven to be on the arm the oracle actually runs.

## What Was Built

**Task 1 — the field and all five construction sites** (`48b25396`)

`EngineContext` gains `rng: random.Random`, inserted after `feed` and before the defaulted
`store`/`sql_engine` per the dataclass's own "required before defaulted" ordering invariant.
The design-invariant block and the NumPy `Parameters` docstring both document it with a `D-07`
tag, including why it is deliberately not defaulted. `random` is stdlib, so no `TYPE_CHECKING`
guard was needed and GATE-01 is untouched.

All five construction sites supply it:

| Site | File:line | Arm |
|------|-----------|-----|
| Legacy direct-construction | `backtest_trading_system.py:178` | the arm the oracle drives (RESEARCH F-5) |
| Factory | `backtest_trading_system.py:506` | `build_backtest_system` |
| Live | `live_trading_system.py` (`build_live_system`) | uses the in-scope `_system_config` binding |
| Inertness probe | `tests/integration/test_okx_inertness.py` | `rng=random.Random(42)` |
| Compose fixture | `tests/integration/test_compose_wiring.py` | held by the fixture so tests can assert identity |

Both backtest arms were migrated, not just one — editing only the factory would have left the
oracle running the un-migrated legacy arm and passing while proving nothing (RESEARCH Pitfall 3).

**Task 2 — ExecutionHandler receives the RNG** (`f61e269c`)

New keyword parameter `rng: Optional[random.Random] = None`. When supplied, `self._rng` IS that
object and `self._rng_seed` is `None`; otherwise the pre-D-07 derivation runs. `compose.py`
passes `rng=ctx.rng`. `_resolve_rng_seed` and the fallback are kept and documented as
backward-compat (mirroring the existing `_default_backcompat_config` note) — a number of
unit/integration tests construct `ExecutionHandler(global_queue)` with no `ctx` at all.

The initialisation log no longer prints a misleading `rng_seed=None` after injection
(T-11.1-14); it reports `rng=injected from EngineContext` vs `rng=derived, rng_seed=42`.

**Task 3 — identity, not equality** (`413b903c`)

New `tests/unit/trading_system/test_engine_context_rng.py` (2 tests): `rng` is required
(omitting it raises `TypeError`) and `dataclasses.fields(EngineContext)` lists it before `store`
and `sql_engine` — the mechanical guard for an invariant that was previously only a docstring
rule. Plus one compose-level test asserting the composed simulated exchange holds the `ctx.rng`
OBJECT by `is`.

## Fail-First Proof (recorded per the plan's acceptance criterion)

Ran once and reverted: `compose.py` changed to pass a fresh `random.Random(42)` instead of
`rng=ctx.rng`.

| Gate | Result under the break |
|------|------------------------|
| `test_the_composed_exchange_holds_the_ctx_rng_instance` | **FAILED** — `assert <random.Random at 0x878b1c020> is <random.Random at 0x878b0a820>` |
| `tests/integration/test_backtest_oracle.py` | **3 passed — still GREEN** |

That divergence is the entire reason this test exists. Two `random.Random(42)` instances are
seed-equal and behaviourally identical until either is drawn from; the oracle only diverges once
the call ORDER differs, which today's golden path (zero-slippage preset, `simulate_failures`
off) does not exercise. A seed-equality assertion would have been satisfied by the broken
wiring. `compose.py` was restored to the committed state and verified clean via `git status`.

## Deviations from Plan

### Plan-prose discrepancies (code wins — no code was changed to match the plan)

**1. mypy source-file count — 260, not 281.** The plan's `281` is the MAIN-CHECKOUT figure. In
a worktree `itrader/strategy_handler/my_strategies/` (gitignored, ~22 files) is absent, so the
measured count is 260. This plan adds **no production module**, so the correct assertion is the
DELTA: 260 before → 260 after, unchanged. Not a stale plan figure; an environment offset.

**2. The `EngineContext(` grep-agreement criterion is imprecise as written.**
`grep -rn "EngineContext(" itrader/ tests/ | wc -l` returns **8**, not 5 — three of those hits
are prose, not construction: a module docstring (`backtest_trading_system.py:10`), a comment
(`test_okx_inertness.py:174`) and an assertion message (`test_okx_inertness.py:197`). The
substantive check holds: there are exactly **5 real construction sites and 5 `rng=` arguments**
(`grep -rn -A9 "EngineContext(" itrader tests | grep -c "rng="` → 5). Reported rather than
worked around.

**3. `live_trading_system.py:1869` config binding.** The plan said to "use whatever config
binding is already in scope". The real binding is `_system_config`, lazy-imported at
`:1841` (`from itrader import config as _system_config`), so the live arm reads
`int(_system_config.rng_seed)` — same resolution, different local name.

### Scope notes

- `rng` was deliberately NOT added to the `Engine` holder dataclass in `compose.py`, per the
  plan's explicit instruction — it is a ctx-level wiring input, and re-surfacing it would create
  a second place to read it from.
- `_rng_seed` changed type from `int` to `Optional[int]`. Grepped for every reader before
  changing it: the only consumers are the initialisation log (updated) and `_resolve_rng_seed`
  itself. `backtest_trading_system.py:405` reads `config.rng_seed` for run-settings
  serialization — an independent read of the config value, not of this attribute, and untouched.
