# Phase 11.1 Plan 05: VenueBundles memoized provider Summary

Created the `VenueBundles` provider — a two-method, import-inert memo over `(venue, account_id)` that
builds each venue bundle exactly once and hands the same object to every caller — plus a six-test unit
contract proving the memo, its keying, its fail-loud lookup, and its hands-off treatment of the
connectors object.

## What Was Built

**`itrader/venues/bundles.py`** (new, 98 lines, 4-space, `mypy --strict` clean)

`VenueBundles.__init__(registry, connectors, ctx)` stores the three collaborators, an empty
`_memo: dict[tuple[str, str], VenueBundle]`, and a component-bound logger. `get(venue, account_id, spec)`
resolves the plugin through `ExecutionVenueRegistry.get` and calls `build_bundle(ctx, spec, connectors)`
only when the key is absent, then returns the memoized value.

Four load-bearing properties, each documented in the module docstring because each is a defect if
inverted:

1. **Build-once is the mitigation, not an optimization.** Two independent `build_bundle` calls produce
   two `OkxExchange` instances per account, and `OkxExchange.connect()` is the sole spawn site for
   `_stream_fills` / `_stream_orders` — a second bundle double-spawns the fill/order streams for one
   authenticated account (T-11.1-16).
2. **The key is the pair, shared with the connector memo.** A bundle memo on any other key lets the two
   layers drift (T-11.1-18).
3. **Fail-loud.** `get` delegates to the registry's bare `self._plugins[name]` subscript, which raises
   `KeyError`. No soft `.get(...) or default` wrapper (T-11.1-17).
4. **Not barrel-exported.** `itrader/venues/__init__.py` is untouched; the module is imported directly,
   mirroring `connectors/provider.py` (T-11.1-20, GATE-01).

Nothing beyond `__init__` and `get` was added — no teardown, no eviction, no `__contains__`, no eager
pre-build. `grep -c "def "` returns exactly `2`.

**`tests/unit/venues/test_venue_bundles.py`** (new, 189 lines, 4-space, 6 tests, package-less dir)

Local doubles `_FakeVenuePlugin` (a `build_calls` counter plus a fresh `exchange` sentinel per call, so a
second build is detectable by identity as well as by the counter) and `_ExplodingConnectorProvider`
(`__getattribute__` raises on every attribute access).

## Fail-First Proof (required by the plan, run once and reverted)

The memo was temporarily removed — `get` rewritten to build unconditionally on every call — and the
suite re-run. **Observed: `2 failed, 4 passed`**, exactly the two tests the plan predicted:

- `test_repeated_get_returns_the_same_bundle_and_builds_once` —
  `AssertionError: assert VenueBundle(exchange=namespace(built_by='paper', build_seq=1), ...) is VenueBundle(exchange=namespace(built_by='paper', build_seq=2), ...)`
  (the `build_seq` 1-vs-2 in the failure message is the second build being observed directly).
- `test_memo_iteration_order_is_stable_across_repeated_gets` — `assert False` on
  `all(a is b for a, b in zip(built, rebuilt, strict=True))` — every entry was displaced by a rebuild.

The edit was then reverted; `git diff --stat` against the Task 1 commit is empty, confirming the shipped
file is byte-identical to the verified one.

### Additions beyond the literal plan text

**Task 2, test 6 strengthened.** The plan specified asserting key order is unchanged. Since a
build-unconditionally regression does *not* reorder a `dict` (reassignment preserves insertion order),
an order-only assertion would have passed against the broken implementation. Added a per-key identity
assertion (`all(a is b ...)` plus `[memo[k] for k in keys] == built`) so "no entry was displaced" is
actually tested. This is what makes test 6 part of the recorded fail-first proof rather than a
false-green.
