Land **D-04 + D-17 + D-06 + D-08's wiring** (VENUE-03, VENUE-05, VENUE-07): the paper venue plugin
builds its own `SimulatedExchange` from a config it receives at construction, the backtest joins the
same venue path live already uses, and `ExecutionHandler` obtains its exchange by ASKING
`VenueBundles` rather than by minting one.

Ordering within this wave is strict and each step is a prerequisite of the next: `rng` must already
be on `EngineContext` (plan 11.1-04) before the plugin can build a stochastic component; the plugin
must hold its `ExchangeConfig` (D-17) before it can build one; `VenueBundles` must exist (plan
11.1-05) before `ExecutionHandler` can ask. And plan 11.1-06 must already have re-keyed the registry
to the paper venue, so the key `ExecutionHandler` publishes is the key every reader already uses —
which is what prevents the green-and-wrong intermediate state the owner ruled out.

Purpose: dissolve the compose-versus-venue-assembly cycle at its source. Once the plugin builds its
own exchange, `PaperVenuePlugin(execution_handler.exchanges[...])` at `live_trading_system.py:2028`
has nothing left to reach for.

Output: a self-sufficient `PaperVenuePlugin`; a `compose_engine` that takes `venue_bundles` and no
longer takes `exchange_config`; an `ExecutionHandler` that resolves rather than mints; both backtest
arms and the live root on the venue path; a shared test helper replacing five ad-hoc constructions.
