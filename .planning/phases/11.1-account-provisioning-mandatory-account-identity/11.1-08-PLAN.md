<objective>
Land **D-14 and the live half of D-08 (VENUE-07)**: exactly one data provider is built per boot, and
`VenueBundles` replaces `assemble_venues`' eager per-account bundle map.

`11-REVIEW.md`'s WR-07 observed that per-account data providers were constructed and left unwired.
The review proposed wiring them all; D-14 rejects that and removes the constructions instead. The
evidence is in the code: `live_trading_system.py:2346-2347` documents that "the DATA provider stays
deliberately single (one feed, the primary's provider), so it is wired outside the loop", and
`VenueLifecycle` takes `provider` as a REQUIRED positional it never actually uses. Building N
credential-bearing providers so that N-1 can be discarded is the defect; the fix is to build one.

Purpose: close WR-07 structurally — an object that is never constructed cannot be left unwired — and
complete D-08's swap on the live path.

Output: an `assemble_venue` that no longer eagerly builds a provider; a `VenueLifecycle` whose
`provider` is genuinely optional; a live root that builds one provider explicitly for the feed and
reads bundles through the shared memo; a call-count assertion that would have caught WR-07.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| account credentials -> data provider | Each `build_provider` resolves that account's own `secret_ref` through the `CredentialResolver` and opens a client. A constructed-but-unwired provider is a live credential-bearing object with no owner and no halt path. |
| market data -> the single feed | One `LiveBarFeed` consumes one provider. A second provider pushing the same venue's bars into the same feed corrupts the bar stream. |
| connector-fatal halt -> trading arms | 11-09 collapsed the primary/secondary split so a second account cannot keep trading after a halt latches. A partially-halted engine is worse than a single-account one, because the surviving arm looks healthy. |
