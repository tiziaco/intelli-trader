# Phase 11.1 Plan 08: One Data Provider, One Bundle Per Account — Summary

`assemble_venue` stopped constructing objects: it now asks the shared `VenueBundles` memo
for the bundle and builds no data provider at all, so a live boot creates exactly one
exchange per `(venue, account_id)` and exactly one feed-bound provider — both proven by
build COUNT and object IDENTITY, not by existence.

## What Was Built

**Task 1 — `assemble.py` / `lifecycle.py` (commit `bcf2ea55`)**

- `assemble_venue(spec, connectors, bundles, *, provider=None, account_store=None, alert_sink=None)`
  — the eager `data_plugin.build_provider(...)` call is gone, and the bundle comes from
  `bundles.get(venue, account_id, spec)` instead of `exec_registry.get(venue).build_bundle(...)`.
- `assemble_venues(specs, connectors, bundles, *, primary_provider=None, ...)` hands
  `primary_provider` to the FIRST spec's lifecycle and to no other — the primary-is-first
  ordering contract is now enforced where it is documented, rather than re-derived by callers.
- `VenueLifecycle.__init__`'s `provider` is keyword-only, `Optional[LiveDataProvider]`,
  default `None`; the property widens to `LiveDataProvider | None`. Confirmed by full-file
  read that `_provider` has no reader inside the class — `start()`/`stop()` drive the
  connector only, so a provider-less lifecycle is fully functional, not degraded.
- `VenueBundles.plugin_for(venue)` — a narrow read for the D-04 venue-UID guard's plugin,
  off the memo's OWN registry.

**Task 2 — the live composition root (commit `c4f500bd`)**

- Step `(3a)`: the ONE `LiveDataProvider` is built explicitly, before assembly, for
  `account_specs[0]` via `data_registry.get(primary_spec.data_provider).build_provider(...)`,
  and injected into the primary lifecycle. `provider = next(iter(venue_lifecycles.values())).provider`
  is gone.
- The per-account registration loop resolves `bundle = venue_bundles.get(exchange, account_id, account_spec)`
  (a memo HIT) and registers `bundle.exchange`. The streaming-venue guard and the
  blackhole-warning comment are preserved verbatim.
- Feed wiring is now `if provider is not None:`-guarded, so a zero-account boot (D-09)
  builds zero providers and does not raise. It previously raised `StopIteration`.
- The `:2346-2347` D-14 documentary comment survives, updated: the provider is now BUILT
  outside the loop as well as wired outside it, and the review's wire-them-all alternative
  is recorded as REJECTED with its reason.

**Commit `a0324e66`** reworded two prose mentions so the
`grep -c build_provider itrader/venues/assemble.py → 0` tripwire is not defeated by

## Deviations from Plan

**1. [Plan/code discrepancy — code wins] `assemble_venue` also lost `ctx` and `exec_registry`, not just `data_registry`.**
- The plan says only "remove `data_registry`". Once `build_bundle`/`build_provider` moved out,
  `ctx` had ZERO readers in the function — leaving a required-but-unread parameter is the exact
  defect this plan removes from `VenueLifecycle.provider`, so it went too.
- `exec_registry` was replaced by `bundles.plugin_for(...)`. Passing BOTH a registry and a memo
  built over a (possibly different) registry is a live divergence hazard: the guard would
  interrogate one plugin while another built the bundle. One source of truth makes it
  unrepresentable. `plugin_for`'s docstring forbids calling `build_bundle` on the result.
- Fail-loud is unchanged: `bundles.plugin_for` → `registry.get` → `KeyError`, resolved BEFORE
  any build. A new test asserts a failed lookup memoizes nothing.

**2. [Plan/code discrepancy — code wins] The plan's premise that nothing reads `lifecycle.provider` is false OUTSIDE `lifecycle.py`.**
- The must-have is accurate about the 149-line file itself, but the facade reads
  `streaming[0].provider` at TWO sites: `_initialize_live_session` (`SessionInitializer(data_provider=...)`,
  ~`:661`) and `start()` (`primary_provider`, the feed-warmup + `subscribe()` loop, ~`:774`).
  Constructing every lifecycle with `provider=None` — the plan's literal step — would have left
  both `None`, silently killing live market-data subscription behind a fully green suite (no
  offline test drives the OKX subscribe loop).
- Implemented the plan's stated INTENT instead — its own must-have says "Only the ONE lifecycle
  whose provider is wired to the feed carries one." The root builds the provider BEFORE assembly
  and passes `primary_provider=`, so the primary lifecycle carries it at construction (no
  late assignment — the standing owner preference) and both facade readers are byte-identical
  to before at N=1 and at N=2.
- Consequence: the plan's step-3 instruction to build the provider "immediately before the feed
  wiring" could not be followed literally — construction-time injection requires it earlier.
  It is built at `(3a)`, immediately above the `assemble_venues` call, and the feed wiring stays
  at `(4)`.

**3. [Plan/code discrepancy — code wins] Two test files outside the plan's `files_modified`.**
- `tests/unit/venues/test_venue_uid_guard.py` constructs `assemble_venue(...)` twice with
  `exec_registry=`/`data_registry=` keywords (`:379`, `:397`). Migrated to the memo form; its
  local `_DataPlugin` fake is now unused and removed. This is the test that proves the D-04
  guard is not dead code, so it had to move with the signature.
- `tests/integration/test_multi_account_composition.py` is where the plan's own
  composition-level count gate belongs, and it was not in `files_modified` either.
- (Third consecutive wave in which `files_modified` was a floor, not a ceiling.)

**4. [Deliberate removal] `test_assemble_unregistered_data_provider_fails_loud` deleted.**
- It asserted `assemble_venue` raises on a bad `data_provider`; assembly no longer resolves data
  plugins, so the assertion is unrepresentable. The guard itself did NOT move to nowhere: it now
  fires at the root's single `DataProviderRegistry.get`, and `DataProviderRegistry.get`'s
  fail-loud contract is covered at `tests/unit/venues/test_registry.py:101-102`. Net venue-unit
  tests: 81 → 89.

**5. [Plan/reality] `mypy` source-file count is 260 here, not 282.**
- Worktree measurement — `itrader/strategy_handler/my_strategies/` (22 files) is gitignored and
  absent. DELTA is zero: this plan adds and removes no production module. The plan's 282 is the
  main-checkout figure and is correct there.

**6. [Behaviour change, intended] A live PAPER boot now reuses the compose-built `SimulatedExchange`.**
- Because assembly reads the memo, `('paper', 'default')` is a memo hit on the entry
  `ExecutionHandler.init_exchanges` already created. Before this plan the paper lifecycle held a
  second, unread exchange (11.1-07 deviation 5). No behaviour depends on the difference — every
  reader of the second object was connector-gated — but the paper-replay parity gate
  (`test_paper_parity.py`) and the whole `tests/` suite were run to confirm. It also removes a
  latent bug in the replay harness: with two paper accounts, `TestDataPlugin.provider` (the
  LAST-built) and the feed's provider (the FIRST-built) would have disagreed.
