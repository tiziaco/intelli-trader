## ACCT-02 Provisioning Bootstrap

### Q1 — A live boot with zero `venue_accounts` rows

| Option | Description | Selected |
|--------|-------------|----------|
| Router scope + admin writer → boot fails loud | Two entry points; boot refuses on zero accounts, matching D-15 | |
| Router scope only → boot starts empty | One writer; boot succeeds with no portfolios | |
| Admin writer only → boot fails loud, no router scope | Provisioning never a runtime config update | |

**User's choice:** Free-text — *"the boot shouldn't fail with 0 account, that's going to be normal for every first startup in the first place. and a portfolio can't exist with[out] an account, that's also normal"*
**Notes:** Settled the coupling surfaced during discussion — `VenueAccountStore` is reachable only from inside the composition root, and `ConfigRouter` only drains a RUNNING engine's queue, so "fail loud on empty" + "router is the only writer" is a deadlock. Boot-empty dissolves it.

### Q2 — The provisioning shape

| Option | Description | Selected |
|--------|-------------|----------|
| Whole-account payload, create-or-update | One payload upserted, mirroring the `portfolio:{id}` scope | |
| Per-key like `venue:{name}`, row created on first write | Consistent with `_apply_venue`'s existing create-on-first-write | |
| Two verbs — explicit create, then per-key edits | Most legible; a router special case | |
| **Facade method, not a config update** | `provision_venue_account(...)`, mirroring `add_portfolio` | ✓ |

**User's choice:** Facade method (recommended)
**Notes:** User asked twice for the problem to be explained and for per-option examples before choosing. A correction was made mid-discussion: the `portfolio:{id}` scope is NOT a multi-field payload — it is still one `key`/`value`, and its "whole-model merge-validate" refers only to its validation strategy. That removed the first option's premise.

### Q3 — Re-provisioning an existing pair

| Option | Description | Selected |
|--------|-------------|----------|
| **Refuse with a typed error** | Create-only; duplicates surfaced, never silently overwritten | ✓ |
| Update in place, preserving `venue_uid` | One idempotent verb | |
| Refuse + a separate update verb in this phase | Both, from day one | |

**User's choice:** Refuse — asked "is it the best and cleanest way?"
**Notes:** Confirmed yes. The "you can never change credentials" objection is weak because `secret_ref` is a POINTER, so routine key rotation changes the env var's value and never moves the pointer. The genuine corner (wrong `secret_ref` with a portfolio already attached) was recorded as a deferred update verb.

### Q4 — `add_portfolio` against a missing account row

| Option | Description | Selected |
|--------|-------------|----------|
| **FK enforces; classify on the error path** | Catch `IntegrityError`, read only then, raise a typed domain error | ✓ |
| Pre-check in `add_portfolio` | Read the store first | |
| Let the raw `IntegrityError` propagate | Zero new code | |

**User's choice:** FK + error-path classification — asked "help me decide"
**Notes:** Research killed the "propagate raw" option: `engine.py:61` enables SQLite FK enforcement, tests run SQLite while prod is Postgres, and `portfolios` carries TWO `IntegrityError`-raising constraints that SQLite reports without names. The error-path-classification shape was not in the original option set — it emerged from that finding.

### Q5 — `venue_accounts.config_json`

| Option | Description | Selected |
|--------|-------------|----------|
| Drop the column | Removes the undefined-precedence overlap | |
| Keep + wire `sandbox`/`region`/`market_type` | The three account-owned traits | |
| **Keep + wire + define precedence** | Also fixes the documented `venues` vs `venue_accounts` overlap | ✓ |
| Record only, wire nothing | Tightest boundary | |

**User's choice:** Keep, wire, and define precedence
**Notes:** User pushed back on the initial "drop it" framing — *"I'd keep it, especially if it's the only place telling me the type of account it is. what do that config column contain?"* — and later observed *"the region concept is specific to some exchanges or broker. not a common field"*, which is the argument FOR the JSON blob rather than a column. Research confirmed the column is an unfinished wire, not dead weight: `okx_plugin.py:112-130` already names it as the destination. It also surfaced that `market_type` is hardcoded at `okx_plugin.py:308`. A claim that D-03's `enable_margin` sourcing was a defect was raised and then RETRACTED — `enable_margin` is consumed only by the paper arm.

### Q6 — Ownership of `VenueAccountStore`

| Option | Description | Selected |
|--------|-------------|----------|
| **`VenueAccountManager` in `venues/`** | One venue-domain owner for four jobs | ✓ |
| Same component named `VenueAccounts` | Provider naming, beside `VenueBundles` | |
| No owner — composition injects the raw store | Smallest diff | |

**User's choice:** `VenueAccountManager`
**Notes:** User interrupted an earlier framing of this question with *"do I smell that maybe we need a separate module for venues?"* and *"so this `VenueAccounts` you proposed would basically be a venue handler right?"*. Both were productive: `venues/` already exists (what was missing was an owner inside it), and it is a Manager not a Handler because `core/enums/event.py` declares no venue event types. A tension was surfaced and recorded — `PortfolioHandler.definition_store` is itself a handler minting its own collaborator, contradicting the phase's governing principle.

## WR-06 / WR-07

### Q1 — WR-07, where the data provider gets built

| Option | Description | Selected |
|--------|-------------|----------|
| **Move 1 only — one provider, built explicitly** | `VenueBundles` memoizes bundles; one `build_provider` call | ✓ |
| Move 1 + 2 — full separation now | Also a credential-free venue-level data session | |
| Move 1, and spec the data-path phase now | Same scope plus a written phase proposal | |

**User's choice:** Move 1 only (recommended)
**Notes:** User's instinct drove this — *"it does share the connector with the account, but nothing else. Maybe I need to separate the provider from the account."* Confirmed by the code. Also established that the review's SECOND proposed fix contradicts a documented decision at `live_trading_system.py:2347` (one feed, one provider) and is rejected. Move 2 recorded as its own future phase.

### Q2 — WR-06, the UID guard's silent-inertness paths

| Option | Description | Selected |
|--------|-------------|----------|
| **Flag only, covering all six paths** | In-memory armed/inert per pair, on `get_status()` | ✓ |
| Flag + edge-triggered alert on transition to inert | Both channels, fatigue designed out | |
| Defer WR-06 entirely | Tightest boundary | |

**User's choice:** Flag only (recommended) — asked "what's the problem here? what's the most architecturally sound thing to do?"
**Notes:** The recorded lean turned out to be right for a better reason than size: for a detective control, state beats events, and the degraded paths fire on every reconnect so a level-triggered alert is alert-fatigue territory. Two conditions attached — the flag must reflect the real outcome across all six paths, and must be in-memory rather than a column.

## Follow-up Structural Gray Areas

User selected all four when offered.

### GA-1 — How `PaperVenuePlugin` gets its `ExchangeConfig`

| Option | Description | Selected |
|--------|-------------|----------|
| **Constructor at registration** | `PaperVenuePlugin(exchange_config)` | ✓ |
| `EngineContext` gains `exchange_config` | Symmetric with D-07's `rng` | |
| Ride it on the `VenueSpec` | No new seam | |

**User's choice:** Constructor injection
**Notes:** User proposed a simpler route first — *"can't we simply import the config module at the top of the file? as simple as that, no passing around"* — which is the established pattern for `ITraderConfig` but fails here on two counts: `ExchangeConfig` is not a field on the singleton, and it is RUN-DERIVED (`_seed_supported_symbols` folds in this run's ticker set). A module-level default would carry the wrong symbol set and the validator would refuse the golden ticker.

### GA-2 — The commission estimator under D-06

| Option | Description | Selected |
|--------|-------------|----------|
| Substitute the source only; record decomposition for Phase 12 | Same adapter, new source | |
| **Do the decomposition now** | Convention into `AdmissionManager`, seam narrowed, adapter deleted | ✓ |
| Estimator holds `VenueBundles` + pair key | Lazy end to end | |

**User's choice:** Decompose now — **overrode the recommendation to defer**
**Notes:** Prompted by *"what is this commission estimator, and why do we even need it? we already have a fee modelling logic"* and then *"can't it live in the admission manager directly?"*. Established it is an adapter, not a duplicate implementation, and that late binding is load-bearing (`simulated.py:775` REPLACES the fee model on config update). The sharpened form — the adapter does two jobs, and the admission convention belongs to admission — came out of that exchange. Reopens a prior-phase D-15; flagged deliberately.

### GA-3 — Backtest `venue_name`

| Option | Description | Selected |
|--------|-------------|----------|
| Change the `exchange` string only | Minimal oracle exposure | |
| **Pass `venue_name='paper'` explicitly too** | Backtest and live structurally identical at creation | ✓ |

**User's choice:** Explicit `venue_name` — **overrode the minimal-diff recommendation**

**Resolved by research, no decision required.** `maintenance_margin` / `margin_ratio` are not abstract methods and exist only on `SimulatedMarginAccount`; `conformance.py` references neither. D-01 touches two methods on one class plus two call sites.

## Claude's Discretion

None. Every decision was taken explicitly by the owner. Five overrode the recommendation given:
D-01 (full break), D-02 (no factory), D-04 (backtest on the venue path), D-18 (decompose the
commission estimator now), D-19 (explicit `venue_name`).

Two "You decide" selections were withdrawn by the user before being acted on — both times to ask
a clarifying question that changed the option set (the venue-owner question, which produced
`VenueAccountManager`; and the `config_json` question, which produced the keep-and-wire outcome).

## Deferred Ideas

- A market-data phase separating the data path from accounts entirely (WR-07 "Move 2")
- A targeted `secret_ref` update verb (repointing an account's credential source)
- `quote_currency` / `symbol` being process-wide rather than per-account
- Live commission under-reservation (`compose.py:78`'s isinstance guard zeroes live OKX estimates)
- Edge-triggered alert-sink rerouting for the D-04 UID guard
- A first-class `VenueHandler` owning registry + lifecycles + bundles + `VenueAccountManager`
- Consolidating the venue domain's six homes into one package
- `PortfolioHandler.definition_store` self-minting, contradicting the governing principle
- Registering `'simulated'` as a venue plugin — rejected outright, not deferred
- `AccountProvider` Protocol in `core/` — made unnecessary by D-04
