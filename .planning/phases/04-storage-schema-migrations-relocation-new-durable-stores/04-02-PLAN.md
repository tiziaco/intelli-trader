<objective>
Movement (B) part 1 of Phase 4: land the three new durable SQL stores as tested standalone units,
each a disciplined clone of the `HaltRecordStore` template (STORE-01/02/03/04/05). `SystemStore`
(cardinality-1 KV), `VenueStore` (per-venue config + enabled, secret-scrub guarded), and
`StrategyRegistryStore` (two-table registry + normalized subscriptions). Each composes `sql_engine`,
owns its `build_*_table` registrar, and is exercised over an in-memory SQLite `SqlEngine` plus a
file-backed restart round-trip.

Purpose: Prove the store shapes, typed-column queryability, secret-scrub, and rehydrate at the STORE
level (D-02) — construction into the live composition root and applying rehydrated state to live
handlers are deferred to P6/P9/P10.
Output: 3 new `itrader/storage/*_store.py` modules + 3 new `tests/unit/storage/*` unit tests. Backtest
oracle byte-exact.

Forward note (D-09): the method surface is intentionally MINIMAL and is to be finalized in P9
(RuntimeConfig) / P10 (StrategyRegistry) against the real consumers — do NOT add consumer-domain
methods here.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller (future composition root / consumer) → store write | Untrusted `config_json`/`value_json` blobs cross into persistence — could carry a secret or a SQL-injection payload |
| store → SQLite/Postgres engine | All SQL crosses here; must be parameterized Core, never f-string |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-01 | Information Disclosure | `VenueStore.config_json` | high | mitigate | Structural: credentials are connector/`OkxSettings` (`SecretStr`)-owned and never passed to the store. Defensive (D-05): a write-time recursive denylist guard rejects secret-like keys at any depth with `ValidationError` before the write. |
| T-04-02 | Tampering | all three stores' SQL | high | mitigate | Parameterized SQLAlchemy Core against the constant `Table` object (insert/select/delete/update) — never f-string SQL (SEC-01 / T-05.2-19). |
| T-04-04 | Tampering | durable identity key | medium | mitigate | Natural NAME PKs (D-06); the ephemeral runtime `strategy_id` UUID is never persisted as the key, so a per-run identity can't corrupt rehydrate. |
| T-04-SC | Tampering | npm/pip/poetry installs | high | accept | No install step — SQLAlchemy is already pinned; milestone gate forbids any poetry change. |
</threat_model>
