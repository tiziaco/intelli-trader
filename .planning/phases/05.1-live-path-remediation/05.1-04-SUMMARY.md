# Phase 05.1 Plan 04: Account ABC Settlement Surface + VenueAccount Ledgered Impl (D-01) Summary

**Widened the `Account` ABC to the 7-member settlement surface and implemented it on `VenueAccount` as a locally-ledgered account (delta-ledger reconciled to venue truth on snapshot), so a fill on a VenueAccount-linked portfolio actually settles — turning the A1 venue-conformance gate fully GREEN while the SMA_MACD oracle stays byte-exact.**

## Accomplishments
- **7-member Account ABC (`base.py`):** renamed abstract `available` → `available_balance`; added `reserved_balance`, `assert_funds_invariant(required)`, and `apply_fill_cash_flow(amount, fee, description, reference_id, timestamp)` abstract members. Final surface: `balance`, `available_balance`, `reserved_balance`, `reserve`, `release`, `assert_funds_invariant`, `apply_fill_cash_flow`.
- **Simulated promotion (`simulated.py`):** deleted the `available` alias; the existing `available_balance` / `reserved_balance` / `apply_fill_cash_flow` / `assert_funds_invariant` are now the ABC implementations — no math change (`apply_fill_cash_flow` still skips the 2dp quantize, keeping the oracle byte-exact).
- **VenueAccount settlement (`venue.py`):** renamed `available` → `available_balance` (settled `balance − pending`, not the venue `free` field); added `reserved_balance`, `assert_funds_invariant`, `apply_fill_cash_flow` as a **locally-ledgered account** — `balance` = cached venue balance + `_ledger_delta`; `apply_fill_cash_flow` moves the delta; `snapshot()` reconciles it to zero. `assert_funds_invariant` raises BEFORE any mutation (T-05.1-01: no partial mutation).
- **A1 fully GREEN:** `tests/unit/portfolio/test_account_conformance.py` — 15 passed (all three leaves incl. VenueAccount): admission read, reserve/release, BUY settle, SELL atomicity (no half-applied fill), `to_dict` serialization all money-correct exact-Decimal.

## Decisions Made
- **Delta-ledger, not absolute snapshot ledger:** `test_venue_account_cache.test_push_stream_mutates_cache` reads `balance` after `start_streaming()` with NO `snapshot()`, so `balance` must reflect the streamed venue cache. Settled `balance` = cached venue balance + `_ledger_delta`; `snapshot()` resets the delta (reconcile).
- **`available_balance` nets settled balance, not venue `free`:** the SPOT fixture has `free(57999.79) ≠ total(78999.79)`; the admission-read invariant (`available_balance == balance` with no reservations) requires netting against the settled total, not the venue `free` field.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] A1 BUY facet hardcoded `149990` is venue-incompatible; made leaf-relative**
- **Found during:** Task 2 (VenueAccount settlement)
- **Issue:** `test_buy_settlement_through_transact_shares` asserted `account.available_balance == Decimal("149990")` uniformly across all three parametrized leaves. That value assumes a 150000 starting balance — true for the two Simulated leaves, but the venue leaf is snapshotted from the SPOT fixture's `total[USDC] == 78999.79`, so its correct post-BUY figure is `78989.79`. The assertion was authored in 05.1-01 but never reached for the venue leaf (it failed earlier on `AttributeError`), so the latent mismatch only surfaced once the settlement surface was implemented here.
- **Fix:** Captured `before = account.available_balance` and asserted `account.available_balance == before − Decimal("10")` — this asserts the BUY committed exactly 10 of buying power on EVERY leaf (cash/margin still 150000→149990; venue 78999.79→78989.79). Production code is unchanged and correct; this tightens the assertion (ties it to the exact settle magnitude) rather than weakening it, and keeps all three leaves in the parametrized gate.
- **Files modified:** tests/unit/portfolio/test_account_conformance.py
- **Verification:** `test_account_conformance.py` 15 passed (all leaves); venue settle confirmed debiting exactly 10 (78999.79 → 78989.79).
- **Committed in:** `345936a4` (Task 2 commit)

**2. [Rule 3 - Blocking] Conformance facet (b) referenced the renamed ABC member `.available`**
- **Found during:** Task 1 (ABC widen)
- **Issue:** `test_reserve_release_conforms` read `account.available`. The plan's Task-1 acceptance requires `available` gone from `base.py` entirely (`grep -c 'def available\b' == 0`), so `.available` no longer resolves on ANY leaf — the facet would `AttributeError` on all three.
- **Fix:** Updated the three references to `.available_balance` (same reserve/release assertion, renamed surface — not weakening).
- **Files modified:** tests/unit/portfolio/test_account_conformance.py
