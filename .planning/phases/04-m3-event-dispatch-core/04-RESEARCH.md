### Locked Decisions

#### Event schema (M3-01, #11)
- **D-01: Frozen dataclass base.** A `frozen=True`/`slots=True`/`kw_only=True` `Event` base dataclass
  carries `event_id`, `created_at`, `time`, and `type` as a **real field**; all concrete events
  subclass it. Python 3.13 dataclass inheritance with slots works via kw_only. isinstance-able for
  dispatch, mypy --strict friendly.
- **D-02: event_id = uuid7 default_factory; created_at = business time.** `event_id` auto-generates
  via `uuid_utils` uuid7 (same scheme as entity IDs; oracle excludes ID values per M1 D-12).
  `created_at` **defaults to the event's business `time`** ‚Äî fully deterministic in backtest, zero
  Clock plumbing into the ~15 construction sites. No wall clock on the engine path (M2-05 holds).
- **D-03: Drop `SignalEvent.verified` ‚Äî typed outcome instead.** The validator/risk-manager verdict
  flows through the order pipeline as typed state (see D-13), never as event mutation. The signal
  stays a pure immutable strategy fact.
- **D-04: Event money fields stay float until M4.** Preserve the existing Decimal‚Üífloat‚ÜíDecimal
  boundary coercions (e.g. `OrderEvent.new_order_event`) exactly ‚Äî zero numeric-drift risk for the
  byte-exact gate. M4-07 retypes.
- **D-05: New dedicated `Side`/`Action` enum (BUY/SELL) in `core/enums`** types `action` on
  Signal/Order/Fill events; `SignalEvent.order_type` becomes `OrderType` (already exists). Portfolio
  maps Action‚ÜíTransactionType at its boundary, exactly like FillStatus‚ÜíOrderStatus today (Phase 3
  D-04 distinct-vocabularies precedent). Follow the Phase 3 `_missing_`/`from_string` enum pattern.
- **D-06: ErrorEvent hierarchy (user-proposed, FastAPI-style).** A **concrete, instantiable**
  `ErrorEvent` base (source/domain, error_type, message, severity, correlation_id, details) + a
  per-domain child `PortfolioErrorEvent`; ALL carry `type=EventType.ERROR` (one registry entry;
  consumers isinstance for domain specifics). Mirrors the `core/exceptions` hierarchy shape. Future
  domain error events (Execution/Order) added only when a domain actually emits them. This replaces
  the current PortfolioErrorEvent's `type = EventType.UPDATE` hack.
- **D-07: Dict payloads ‚Äî type what's cheap.** Tighten `PortfolioUpdateEvent.portfolios` if an
  existing shape fits; `strategy_setting` may stay a documented `dict[str, Any]`. NO new DTO classes
  invented purely for typing (M4/M5 re-own those shapes).
- **D-08: PING family ‚Üí TimeEvent family.** Rename `PingEvent` ‚Üí `TimeEvent`, `EventType.PING` ‚Üí
  `EventType.TIME`, `PingGenerator` ‚Üí `TimeGenerator`, `simulation/ping_generator.py` ‚Üí
  `simulation/time_generator.py` (git mv). Nautilus-precedent naming ("the clock advanced to T"),
  pairs with `core.clock.Clock`; kills the factually-wrong PingEvent docstring. `TICK` is RESERVED
  for future live market-data ticks (deferred). Oracle-safe: event names appear nowhere in the oracle.
- **D-09: Events package split.** `events_handler/events/` package: `base.py` (Event), `market.py`
  (Time/Bar/PortfolioUpdate), `signal.py`, `order.py`, `fill.py`, `error.py` (ErrorEvent hierarchy) ‚Äî
  with `__init__` re-exports keeping consumer import paths short (Phase 3 D-11 reorg precedent).
  `EventType` relocates to `core/enums` (closing Phase 3's D-05 deferral); `event_type_map` deleted.

#### Linkage IDs (M3-01)
- **D-10: `SignalEvent.quantity: float | None = None`.** `None` means "order/risk layer sizes me"
  (kills the `0` sentinel); explicit caller-supplied quantity preserved (and its regression test
  `test_zero_quantity_signal`). Full removal deliberately NOT done ‚Äî M5-06 owns the sizing-policy
  schema and may replace the field then.
- **D-11: Brackets via create-all-then-emit.** Restructure `process_signal`: build parent + SL/TP
  Order entities FIRST (all UUIDv7 ids exist), populate `parent.child_order_ids` (today declared at
  `order.py:74` but never populated), THEN emit complete OrderEvents parent-first. Queue arrival
  sequence is unchanged ‚Üí behavior-preserving. Events carry two-directional linkage:
  `child_order_ids: tuple[OrderId, ...] = ()` on non-brackets, `parent_order_id` on children.
- **D-12: Required IDs via minimal conformance.** In-scope sites construct events from entities (ids
  guaranteed). The D-live `trading_interface` gets the minimal fix to keep importing/type-checking
  (generate the UUIDv7 at the call site or via small entity construction) ‚Äî no deeper live rework.
  `FillEvent` gains `fill_id` (generated by the exchange at fill construction) + `strategy_id`
  (copied from the originating OrderEvent); `order_id` required on both OrderEvent and FillEvent.
- **D-13: Order entity AS the pipeline state (no OrderSpec type).** After sizing resolves the
  Decimal quantity, create the `Order` entity (PENDING) immediately; the validator/risk-manager
  check the **entity**; acceptance ‚Üí store + emit events (D-11); rejection ‚Üí transition to REJECTED
  via the existing audited `add_state_change` path (deterministic event-derived timestamps, M2-09).
  FIX/Nautilus lifecycle shape: rejections become auditable state changes instead of vanishing.
  The resolved quantity lives Decimal-native on the entity (the WR-05 signal float coercion dies
  with the signal mutation). Sizing failures (invalid price) still short-circuit BEFORE entity
  creation, preserving the narrow sizing-before-validation gate. NOTE: rejected signals now leave a
  REJECTED order in storage (not in the oracle, but storage-count test assertions may need touching).

#### Dispatch registry (M3-02)
- **D-14: EventHandler-owned explicit route dict.** The full `dict[EventType, list[Callable]]` is
  built in `EventHandler.__init__` ‚Äî the entire dispatch order is ONE reviewable literal. List order
  IS the documented execution order (BAR: portfolios‚Üíexecution‚Üístrategies; FILL: portfolio‚Üíorder
  mirror). Handlers stay passive ‚Äî no registration API (#2: no pub-sub decoupling).
- **D-15: Drain via `get_nowait()` + `queue.Empty` ‚Üí break.** No `empty()` precheck (kills the
  TOCTOU and the `event is None` deref).
- **D-16: Layered error flow + `_on_handler_error` policy seam.** Business errors stay DATA below
  the dispatcher (FillEvent(REFUSED), ErrorEvents, typed results ‚Äî the run continues); unexpected
  handler exceptions route through a `_on_handler_error(event, handler)` seam whose backtest
  implementation **re-raises** (fail-fast, today's behavior, oracle-safe). Live publish-and-continue
  becomes a later override (D-live). Mirrors the Phase 3 D-06 replaceable-seam precedent. The ERROR
  route gets a real consumer that logs ErrorEvents with bound structlog context.
- **D-17: Explicit empty routes for consumerless types; unknown ‚Üí raise.** `SCREENER` (D-screener)
  and `UPDATE` (live API path) register with an explicit empty list + comment naming the deferral.
  Unregistered/unknown types raise `NotImplementedError` (fixes KB1's `NotImplemented`). This also
  fixes a latent crash found during discussion: `PortfolioErrorEvent` (type=UPDATE) IS queued on
  failure (`portfolio_handler.py:116`) but the current chain has no UPDATE branch ‚Üí dispatcher
  would raise. Under the new design it carries EventType.ERROR and routes to the log consumer.

## Summary

This is a brownfield, behavior-preserving refactor phase with **zero new dependencies** ‚Äî everything needed (Python 3.13.1 dataclasses, `uuid-utils` 0.16.0, `structlog` 24.4.0, stdlib `queue`) is already installed and in use. Research therefore focused on (a) empirically verifying the exact dataclass mechanics D-01/D-02 depend on by executing them in the project venv, and (b) building a complete codebase inventory of mutation sites, construction sites, exception-raise sites, and logging sites so the planner can write surgical tasks.

All locked dataclass mechanics **work as designed on Python 3.13.1** (verified by execution, not training data): a `frozen=True, slots=True, kw_only=True` base with `type: EventType = field(init=False)` and per-subclass `field(default=..., init=False)` overrides; `event_id` via `uuid_utils.compat.uuid7` default_factory; `created_at` defaulting to business `time` via `__post_init__` + `object.__setattr__`; deep hierarchy (Event ‚Üí ErrorEvent ‚Üí PortfolioErrorEvent); tuple linkage defaults; `__str__` overrides; mutation raising `FrozenInstanceError`; no `__dict__`; subclass slots correctly excluding inherited fields.

The two most important discoveries beyond CONTEXT.md: **(1)** the event-mutation inventory is larger than the signal sites listed ‚Äî `SimulatedExchange._emit_fill` rewrites `fill_event.price/quantity` after construction (`simulated.py:233-236`) and `MatchingEngine.modify` rewrites resting `OrderEvent.price/quantity` (`matching_engine.py:54-63`); both must be redesigned (construct-complete / replace-in-book) before `OrderEvent`/`FillEvent` can freeze, and the existing test `tests/unit/events/test_event_immutability.py` **asserts these mutabilities as the contract** and must be rewritten, not merely extended. **(2)** `kw_only=True` breaks all positional construction ‚Äî ~79 direct constructor call sites across `itrader/`, `tests/` (18 test files), and `scripts/` need keyword-form updates; this is the phase's largest mechanical churn and should be planned as its own bisectable pass.

**Primary recommendation:** Sequence as (1) events package + schema (with construct-complete fill/order paths), (2) dispatch registry, (3) exceptions, (4) logging ‚Äî each workstream a separate bisectable commit train with the full suite + both oracle layers green at every commit; use the verified code patterns in this document verbatim as the starting point.

### Pattern 1: Frozen Event base + per-class `type` default ‚Äî VERIFIED on Python 3.13.1

**What:** The exact D-01/D-02 mechanics. Every line below was executed successfully in the project `.venv`.
**When to use:** `events/base.py` and every concrete event.

```python
# Source: verified by execution in project .venv (Python 3.13.1, uuid-utils 0.16.0)
from dataclasses import dataclass, field
from datetime import datetime
import uuid
import uuid_utils.compat as uuid_compat

from itrader.core.enums import EventType  # relocated class enum

@dataclass(frozen=True, slots=True, kw_only=True)
class Event:
    """Immutable event fact. All concrete events subclass this."""
    type: EventType = field(init=False)          # real field, no base default ‚Äî subclass sets it
    time: datetime                                # business time
    event_id: uuid.UUID = field(default_factory=uuid_compat.uuid7)
    created_at: datetime | None = None            # defaults to business time (D-02)

    def __post_init__(self) -> None:
        if self.created_at is None:
            # stdlib-documented idiom for frozen dataclass __post_init__;
            # works with slots=True (verified).
            object.__setattr__(self, "created_at", self.time)

@dataclass(frozen=True, slots=True, kw_only=True)
class TimeEvent(Event):
    type: EventType = field(default=EventType.TIME, init=False)

@dataclass(frozen=True, slots=True, kw_only=True)
class SignalEvent(Event):
    type: EventType = field(default=EventType.SIGNAL, init=False)
    ticker: str            # required fields AFTER defaulted base fields ‚Äî fine, kw_only
    action: Side           # enum-typed (D-05)
    order_type: OrderType
    quantity: float | None = None   # D-10
    ...
```

Verified properties (all executed):
- `TimeEvent(time=t).type is EventType.TIME`; `created_at == time`; `event_id` is stdlib `uuid.UUID` version 7
- `isinstance(sig, Event)` ‚Üí True (registry/isinstance dispatch works)
- field reassignment raises `dataclasses.FrozenInstanceError`
- no `__dict__`; `Event.__slots__ == ('type','time','event_id','created_at')`, subclass slots contain only its own new fields (no duplicate-slot problem on Python 3.13 even though the subclass *redefines* `type` ‚Äî the dataclass machinery excludes inherited slots)
- deep hierarchy works: `Event ‚Üí ErrorEvent (concrete) ‚Üí PortfolioErrorEvent`, child narrowing `source: str = "portfolio"` ‚Äî fine
- `child_order_ids: tuple[uuid.UUID, ...] = ()` default ‚Äî fine
- `__str__` override on frozen+slots subclass ‚Äî fine
- `dataclasses.replace(evt, action=...)` works ‚Äî **but keeps the original `event_id`** (replace passes existing init-field values; see Pitfall 2)
- uuid7 values generated in sequence sort in generation order (time-ordered, RFC 9562)

**Frozen inheritance rule** [CITED: docs.python.org/3/library/dataclasses.html#frozen-instances]: dataclass inheritance cannot mix ‚Äî a frozen dataclass cannot inherit from a non-frozen dataclass and vice versa. Every class in the Event tree must be `frozen=True`.

### Pattern 3: EventHandler-owned route registry (D-14, D-17)

```python
# Source: design derived from full_event_handler.py:67-85 current chain ‚Äî order preserved exactly
def __init__(self, ...):
    ...
    # THE dispatch order. List order IS execution order. One reviewable literal.
    self._routes: dict[EventType, list[Callable[[Any], None]]] = {
        EventType.TIME: [
            self.screeners_handler.screen_markets,
            self.universe.generate_bar_event,
        ],
        EventType.BAR: [
            self.portfolio_handler.update_portfolios_market_value,  # 1) mark-to-market
            self.execution_handler.on_market_data,                  # 2) resting-order matching
            self.strategies_handler.calculate_signals,              # 3) new signals
        ],
        EventType.SIGNAL: [self.order_handler.on_signal],
        EventType.ORDER: [self.execution_handler.on_order],
        EventType.FILL: [
            self.portfolio_handler.on_fill,   # 1) positions/cash
            self.order_handler.on_fill,       # 2) order-mirror reconciliation
        ],
        EventType.SCREENER: [],   # explicit empty ‚Äî consuming screeners is D-screener
        EventType.UPDATE: [],     # explicit empty ‚Äî live API path consumes these (D-live)
        EventType.ERROR: [self._log_error_event],   # D-16: real consumer
    }

def _dispatch(self, event: Event) -> None:
    try:
        handlers = self._routes[event.type]
    except KeyError:
        raise NotImplementedError(f"EventHandler: unsupported event type {event.type!r}")
    for handler in handlers:
        try:
            handler(event)
        except Exception:
            self._on_handler_error(event, handler)   # D-16 seam

def _on_handler_error(self, event: Event, handler: Callable[[Any], None]) -> None:
    """Backtest policy: fail fast (today's behavior, oracle-safe). D-live overrides."""
    raise
```

Note on the seam: a bare `raise` inside the `except` block re-raises the active exception ‚Äî but `_on_handler_error` is *called from* the except block, so the exception is still active and bare `raise` works. If the planner prefers explicitness, pass the exception: `_on_handler_error(event, handler, exc)` and `raise exc` ‚Äî planner discretion; the bare-`raise`-in-callee form is valid Python (the active exception context propagates into the call).

### Anti-Patterns to Avoid
- **Mutating events anywhere** ‚Äî the entire point of M3-01. The verdict/quantity/price flows move to entities (D-03/D-13) or construction-time parameters (Pattern 5).

- **`raise NotImplemented`** ‚Äî KB1: `NotImplemented` is a comparison sentinel, raising it is a `TypeError` in Python 3; use `NotImplementedError`.

- **Pulling M4/M5 forward** ‚Äî event money stays float; `BarEvent.bars` payload + `get_last_*` ladders untouched; `engine_logger.py` untouched.

## Common Pitfalls

### Pitfall 1: The mutation-site inventory is BIGGER than CONTEXT.md lists ‚Äî and a test asserts the mutability
**What goes wrong:** Freezing `OrderEvent`/`FillEvent` with only the signal-mutation sites fixed crashes the run with `FrozenInstanceError` inside the exchange.
**Why it happens:** Two execution-layer mutation sites exist beyond the documented signal sites:
1. `itrader/execution_handler/exchanges/simulated.py:233-236` ‚Äî `_emit_fill` does `fill_event.price = executed_price; fill_event.quantity = fill_quantity` after `FillEvent.new_fill` (slippage-applied price + partial-fill quantity).
2. `itrader/execution_handler/matching_engine.py:54-63` ‚Äî `modify()` does `order.price = new_price; order.quantity = new_quantity` on the resting `OrderEvent` in the book.

Additionally `tests/unit/events/test_event_immutability.py` **asserts as its contract** that SignalEvent/FillEvent/OrderEvent mutate (e.g. `test_fill_event_stays_mutable`); the suite goes red the moment events freeze unless this file is rewritten in the same commit train.
**How to avoid:** (1) Pattern 5 ‚Äî construct-complete `new_fill` taking explicit price/quantity. (2) Replace-in-book ‚Äî `MatchingEngine.modify` builds the updated resting event via `dataclasses.replace(order, price=new_price, quantity=new_quantity)` (None-guarded per arg) and stores it back; the resting book is keyed by `order_id`, which `replace` preserves. (3) Rewrite `test_event_immutability.py` to assert FrozenInstanceError for ALL events (D-23 group 2 supersedes it).
**Warning signs:** `FrozenInstanceError` from `_emit_fill` or `modify` during the oracle run; `test_event_immutability.py` failures.

### Pitfall 2: `dataclasses.replace` silently preserves `event_id` (and any other defaulted init field)
**What goes wrong:** Assuming `replace()` produces a "new event" with a fresh `event_id`.
**Why it happens:** `replace()` passes the original values of all init=True fields (including `event_id`, whose default_factory only fires when the argument is absent at construction). Verified by execution: `replace(s, action=...)` ‚Üí same `event_id`. `init=False` fields (`type`) are correctly re-derived from the class default.
**How to avoid:** Decide semantics explicitly: for resting-order MODIFY, preserving `event_id` is arguably correct (same instruction, amended) ‚Äî but document it. If a fresh id is wanted, pass `event_id=uuid_compat.uuid7()` to `replace`.
**Warning signs:** Duplicate `event_id` values in any future event audit/log.

### Pitfall 3: `kw_only=True` breaks every positional construction ‚Äî ~79 call sites
**What goes wrong:** `SignalEvent(_TIME, "MARKET", "BTCUSDT", ...)` raises `TypeError: takes 1 positional argument`.
**Why it happens:** kw_only is required for the inheritance ergonomics (defaulted base fields before required subclass fields) ‚Äî it makes ALL fields keyword-only.
**How to avoid:** Inventory (verified by grep): ~79 direct constructor sites across `itrader/` (~15, enumerated in CONTEXT code_context), `tests/` (18 files ‚Äî heavy in `tests/unit/order/`, `tests/unit/events/`, `tests/unit/execution/`, `tests/unit/portfolio/`, `tests/integration/`), and `scripts/run_backtest.py`. Plan this as its own mechanical pass; factories (`new_fill`, `new_order_event`, `Order.new_*`) absorb most production-site churn.
**Warning signs:** TypeError storms on first suite run after freezing ‚Äî expected; fix systematically, not ad hoc.

### Pitfall 4: All classes in the Event tree must be frozen ‚Äî and PortfolioErrorEvent today is a plain mutable dataclass with `type = EventType.UPDATE`
**What goes wrong:** Mixing frozen/non-frozen in one inheritance chain raises `TypeError` at class definition.
**Why it happens:** stdlib rule [CITED: docs.python.org/3/library/dataclasses.html]. The current `PortfolioErrorEvent` (`event.py:419-458`) is mutable, carries `type = EventType.UPDATE` (the documented hack), and has a `to_dict`. D-06 rebuilds it as a frozen `ErrorEvent` child with `type=EventType.ERROR`.
**How to avoid:** Build `error.py` fresh from the Event base (verified pattern: concrete ErrorEvent base + child narrowing a default works). `tests/unit/portfolio/test_portfolio_handler.py` imports `PortfolioErrorEvent` ‚Äî update its assertions. Also note `_publish_error_event` (`portfolio_handler.py:102-116`) constructs it with `time=datetime.now(UTC)` ‚Äî a wall-clock event-construction site; with D-02 this becomes `created_at` = that time too. Error events fire only on failure paths (never during a green oracle run), so the wall clock here cannot affect determinism of the gate; planner may keep `datetime.now(UTC)` for error time or thread the operation's event time ‚Äî discretion.
**Warning signs:** `TypeError: cannot inherit frozen dataclass from a non-frozen one` (or vice versa) at import.

### Pitfall 5: The latent UPDATE-dispatch crash is REAL ‚Äî and the fix changes which route fires
**What goes wrong:** Today, if `publish_error_events` is enabled and a portfolio operation fails, `PortfolioErrorEvent` (type=UPDATE) is queued and the dispatch chain (`full_event_handler.py:67-85`) has NO UPDATE branch ‚Üí falls into `else: raise NotImplementedError`. **Warning signs:** none at runtime today (path never fires in a green run) ‚Äî that's why it survived; the new test makes it permanent.

### Pitfall 6: EventType relocation + `event.py` deletion touches 31 itrader files and 20+ test files
**What goes wrong:** Import-path churn breaks collection (`346 collected` is the Phase 3 baseline invariant) or leaves a half-shimmed module where `EventType.PING` and `EventType.TIME` coexist inconsistently.
**Why it happens:** `from itrader.events_handler.event import ...` appears in 31 itrader modules + 20+ test files (verified grep). Functional-Enum identity also means any stale duplicate definition of EventType breaks `event.type == EventType.BAR` comparisons across modules.
**How to avoid:** Phase 3 D-11 precedent: history-preserving git mv / package split with `__init__` re-exports, repoint imports in the same commit train, ONE definition of EventType (in `core/enums`), `events_handler/events/__init__.py` re-exports it for convenience. Decide shim-vs-big-bang explicitly (Phase 3 deleted the flat config shadow only after consumers moved).
**Warning signs:** pytest collection count changing; `AttributeError: PING` after the rename (sites still referencing `EventType.PING` ‚Äî grep for `EventType.PING` must return zero at the end).

### Pitfall 7: D-13 leaves REJECTED orders in storage ‚Äî storage-count tests shift
**What goes wrong:** Tests asserting order counts (e.g. `tests/unit/order/test_on_signal.py`, `test_order_storage.py`, `test_order.py`) break when rejected signals now persist a REJECTED order instead of vanishing.
**Why it happens:** Intended behavior change in *storage observability* (not in the oracle ‚Äî rejected orders never fill, so trades/equity are unchanged).
**How to avoid:** CONTEXT already authorizes touching these assertions (Claude's Discretion). Adjust counts/filters (`is_active` vs all) deliberately; keep `test_zero_quantity_signal` semantics (explicit quantity preserved ‚Äî now via `quantity=None` sentinel, D-10).
**Warning signs:** Storage-count assertion failures ‚Äî expected, fix per D-13 note.

### Pitfall 8: `Settings.log_level` exists but `init_logger` never reads it
**What goes wrong:** D-21 says "Settings.log_level restores the firehose" ‚Äî but wiring it naively into `itrader/__init__.py` import-time `init_logger(config)` can instantiate `Settings()` which **requires `ITRADER_DATABASE_URL`** (SecretStr, no default) ‚Üí `ValidationError` on every import.
**Why it happens:** `init_logger` reads `getattr(config, "LOG_LEVEL", "INFO")` (`logger.py:180`) ‚Äî `SystemConfig` has no such attribute, so it's always "INFO". The obvious fix (read `Settings()`) trips the fail-loud secret.
**How to avoid:** Read the log level from the environment without constructing full `Settings` (e.g. `os.environ.get("ITRADER_LOG_LEVEL", "INFO")`), or split a non-secret settings subset. Same for `json_logs` (hardcoded `False` at `logger.py:183`). The root-handler clear at `logger.py:100-101` needs the D-20 guard (don't clobber handlers installed by embedding applications/pytest).
**Warning signs:** `pydantic.ValidationError: database_url Field required` on `import itrader`.

### Pitfall 9: Demoting per-tick INFO logs to DEBUG can surprise log-asserting tests
**What goes wrong:** Tests using `caplog` or asserting log output at INFO break when flow logs demote to DEBUG.

**How to avoid:** Grep tests for `caplog`/log assertions before the demotion commit (none surfaced in the inventory reads, but verify at plan time); `filterwarnings=["error"]` does not affect log records, only warnings.
**Warning signs:** assertion failures referencing log text.

### Pitfall 10: `core/exceptions/execution.py` deletion ‚Äî verify zero importers INCLUDING `__init__.py` re-exports
**What goes wrong:** `core/exceptions/__init__.py` (84 lines) re-exports the execution classes; deleting the module without pruning the `__init__` breaks every `from itrader.core.exceptions import ...` import.
**Why it happens:** Package `__init__` re-export pattern.
**How to avoid:** D-13 mechanical-delete precedent ‚Äî grep for each of the 12 class names across itrader+tests, prune `__init__.py` exports in the same commit. `ExecutionErrorCode` (the ENUM in `core/enums/execution.py`) STAYS ‚Äî it's the data-path error vocabulary; only the exception classes go.
**Warning signs:** ImportError at collection.

## Open Questions (RESOLVED)

1. **Shim vs big-bang for `events_handler/event.py`**
   - What we know: 31 itrader files + 20+ test files import from the old path; Phase 3 did both styles (shim for config during transition, then delete).
   - What's unclear: whether the planner wants a one-plan-wide transitional shim (old path re-exporting from `events/`) or repoints everything in one commit train.
   - Recommendation: repoint in the same workstream (imports are mechanical; a shim that survives commits invites drift), but keep the git-mv/structure commit separate from logic commits per Phase 3 discipline.

2. **`MatchingEngine.modify` semantics under frozen events**
   - What we know: replace-in-book via `dataclasses.replace` preserves `order_id` (linkage) and `event_id` (Pitfall 2); current signature types `order_id: int` (stale ‚Äî IDs are UUID since M2).
   - What's unclear: whether the modified resting event should carry a fresh `event_id`.
   - Recommendation: preserve `event_id` (same instruction, amended) and fix the stale `int` annotation while touching the file; document the choice in the docstring.

3. **`Side` vs `Action` enum name and `ScreenerEvent` typing**
   - What we know: D-05 says "Side/Action enum (BUY/SELL)"; ScreenerEvent docstring mentions direction/action but the dataclass has neither field (only id/name/strategies/tickers).
   - Recommendation: name it `Side` (pairs with the existing `PositionSide`, FIX vocabulary); ScreenerEvent freezes structurally with no new enum fields (D-screener owns its semantics).

4. **Error-time source for ErrorEvents (D-02 tension)**
   - What we know: `_publish_error_event` uses `datetime.now(UTC)`; error paths never fire during a green oracle run.
   - Recommendation: thread the failing operation's event time where one exists, else accept wall clock for the error path and document the carve-out (mirrors the D-09 perf-telemetry carve-out).
