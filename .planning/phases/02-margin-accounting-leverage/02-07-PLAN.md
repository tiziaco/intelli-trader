<objective>
Close LEV-03: plumb strategy-declared leverage end-to-end through the run path so the
Phase-2 margin core is internally consistent under leverage > 1. This closes the two honest
integration findings surfaced by the 02-06 parked leveraged-long e2e:

- **Finding A** — `StrategiesHandler.calculate_signals` drops `SignalIntent.leverage` when it
  builds the `SignalEvent` (the field is simply not passed), so a strategy cannot express
  leverage > 1 through the normal fan-out.
- **Finding B** — `OrderEvent`, `FillEvent`, and `Transaction` carry no leverage, so
  `Position.leverage` defaults to `Decimal("1")` in the run path. The admission *reservation*
  correctly uses `notional / effective_leverage`, but the *position-life* locked margin uses
  `position.leverage = 1` (= full notional) — an internally inconsistent under-lock.

The fix carries the admission-clamped **EFFECTIVE** leverage
`min(signal.leverage, Instrument.max_leverage, portfolio.max_leverage)` (NOT the raw request)
from the Order entity through OrderEvent -> FillEvent -> Transaction. `Position` and
`PositionManager` ALREADY consume `transaction.leverage` via `getattr` (position.py:241,
position_manager.py:170), so once `Transaction` carries a real `leverage` attribute the value
flows to the position automatically. The result: position-life locked margin
(`aggregate_notional / leverage`) EQUALS the admission reservation
(`notional / effective_leverage`).

Everything stays owner-gated and oracle-dark: with `enable_margin` off the effective leverage
degrades to `Decimal("1")` and the SMA_MACD spot oracle (134 / 46189.87730727451) holds
byte-exact. Synthetic LEVUSD ticker only in the e2e — BTCUSD untouched.

Wave 5 — depends on the full margin core (02-03 admission/cap, 02-04 lock-and-settle,
02-06 parked e2e). Reworks the 02-06 e2e file to drop the workarounds.
