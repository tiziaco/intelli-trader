<objective>
Move venue assembly above portfolio rehydrate, make the account set store-derived, delete the six
boot-path helpers that derived and minted account identity, and move the feed binding out of the live
root.

Purpose: this is the phase's thesis executed on the boot path. `portfolios.venue_name` and
`.account_id` are already `NOT NULL` under a composite FK onto `venue_accounts`, so a persisted
portfolio can never lack an account — yet the boot still derives its account set from the spec,
coerces nulls, and mints `venue_accounts` rows. Plan 05 built the replacement spine; this plan hands
the boot over to it and deletes the defensive layer.

**ACCT-01 and ACCT-02 must land together** and they do, in Task 1. The asymmetry matters: a
store-derived set with minting still present is a harmless no-op, but minting deleted **unreplaced**
makes the composite FK reject the first `add_portfolio` on a fresh DB — no portfolio could ever be
created. A wave boundary between them produces a system that boots and then cannot do anything.

Output: a boot block that reads `for row in manager.read_enabled_for(venue): manager.bring_online(row)`,
a `live_trading_system.py` several hundred lines shorter, and a green suite across the 10-file test
blast radius.

**Standing executor brief — CODE WINS OVER PLAN TEXT.** Every deletion boundary below was computed
with `ast` at tree `494f5de3` (RESEARCH §A1–A5). ⚠ Several ROADMAP anchors are stale: the ROADMAP
puts `_mint_account_rows` at `:1534` (real: `:1526`), and `_attach_venue_accounts` is **120** lines,
not the 116 CONTEXT.md states. Locate symbols, not lines. If the code disagrees, **stop and report**.

**This is a heavy plan (3 tasks).** Commit after each task.
</objective>
