## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator-authored `config_json` → venue routing | `sandbox` / `region` decide which host a live order reaches. |
| `market_type` → `VenueAccount` cash accounting | `_cash_stream_nets_holds = market_type != "spot"` changes how cash is netted. |
| venue UID (trust-on-first-use) → account identity | The D-04 guard is what detects a swapped sub-account. |

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-11.2-06-01 | Tampering | `config_json` key injection / typo | high | mitigate | Task 2's explicit three-key allowlist **raises** on any other key; a silently-dropped `sandox` would leave a live order routed to a production host. The raise is the control, tested directly. |
| T-11.2-06-02 | Tampering | inert `market_type` key | high | mitigate | Task 1 removes the hardcode so the account-level key actually reaches `VenueAccount`; without it the allowlist would accept a key that changes nothing, which is worse than rejecting it. |
