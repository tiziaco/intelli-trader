<objective>
**TRACER SLICE.** Build the phase's spine end-to-end on one account: a durable `venue_accounts` row
becomes a live, routable venue account in the same process, and resolves into a constructed account
leaf — proven by one integration test on the SQL spine.

This is the thinnest path through every layer the account domain touches — store → manager →
`assemble_venue` → exchange registry → `VenueBundles` → account leaf — wired for keeps, with real
error handling on the single path. It is production code, not a prototype: nothing here is a stub
that a later plan must replace, and every later plan in this phase builds outward from it.

Purpose: the whole phase rests on `bring_online` having **exactly one implementation** with boot as
its bulk case. Proving that end-to-end on the executor's best early-context tokens is what catches an
architectural dead end after one commit instead of after ten. Every subsequent wave — the boot-order
move (plan 07), the coercion sweep (plan 08), the portfolio factory (plans 09/10) — assumes this
shape.

<assumption_delta_decision>
The assumption-delta detector fired (`pluralization`, `chosen`) on this phase's scope.

- **The noun that is now primary:** the **venue account** — the `(venue_name, account_id)` pair. It
  was previously a *derived* detail of a venue spec (computed by `_account_ids_for_spec`, coerced
  from `None` to a single implicit `'default'`); it becomes a **chosen, durably provisioned entity**
  read from `venue_accounts`.
- **Decision: `promote`.** The pair is promoted to the primary identity across the exchange registry,
  the connector memo, `VenueBundles` and the distinct-account invariant, and `DEFAULT_ACCOUNT_ID` is
  **demoted to a detail of exactly one variant** — the backtest/simulated single-account arm, where
  it stays load-bearing for the golden oracle. This is exactly what ACCT-04 encodes: delete the
  live-path coercions, keep the backtest ones.
- **Not `add-alongside`.** Keeping a coerced implicit `'default'` account alongside real provisioned
  ones is precisely the WR-03 asymmetry — the registry writes `(venue,'default')` while both readers
  construct `(venue, None)`, making the registered key unreachable. Adding the general case beside a
  still-required specific one is the failure mode here, not the safe option.
- **Invariant test (companion, suggested):** the ACCT-04 static gate in plan 08 encodes the
  generalized intent — a grep-based assertion that no live-path site re-derives an account identity
  by coercion. A future phase reintroducing the singular assumption goes red on it.
</assumption_delta_decision>
