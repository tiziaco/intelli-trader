### How does a portfolio get its account at construction?

| Option | Description | Selected |
|--------|-------------|----------|
| Same memo, real key | `_new_compute_account` resolves the portfolio's real pair through the `VenueBundles` it already holds | |
| New account-resolver port | A narrow resolver injected into `PortfolioHandler` | |
| Caller passes the built `Account` | Handler stops constructing accounts entirely | ✓ (then superseded) |

**User's choice:** Initially "caller passes the Account" — *"Honestly, option three seems the most correct one."*
**Notes:** Building the concrete example exposed that `add_portfolio` constructs the account **last**
(`portfolio_handler.py:325-355`), after four things only the handler can compute (resolved `portfolio_id`,
id-scoped `state_storage`, `to_money(cash)`, `enable_margin` from the resolved preset). A pair-bound factory was
offered as a fourth option. The owner rejected all four: *"I do not like any of the solutions proposed… I smell
the problem is elsewhere in the workflow."* That reframing produced D-24/D-25 — the real defect is that
`add_portfolio` conflates a durable **write** with a runtime **build**, with `_persist_definition`'s
"GATED ON ABSENCE" (`:454`) as the visible scar, because `rehydrate_portfolios` calls `add_portfolio`.

### Where does the resolution home live?

| Option | Description | Selected |

**Notes:** The first framing (`system.provision_venue_account(...)` as a facade method, per D-10's wording) was
rejected outright: *"I do not want any `system.provision_venue_account`… it's already a mess in that live system

### What `delete_portfolio` persists

| Option | Description | Selected |
|--------|-------------|----------|
| Soft — row survives | History and child tables intact | ✓ |
| Hard — delete the row | Orphans seven child tables | |
| Hard + archive table | A migration anyway, half-moved history | |

**User's choice:** Soft → D-31, refined to D-32.
**Notes:** *Found during discussion:* `delete_portfolio` never touches the store at all today, so a deleted
portfolio returns on the next boot. Then the owner's real workflow reframed it: *"if I disable the portfolio,
it's because I want to link the account related to that portfolio to a new portfolio… for sure I do not want to
disable the account."* That workflow is **blocked** by the plain `UniqueConstraint('venue_name','account_id')`.
The owner proposed `is_deleted` (better than the reviewer's `enabled`-keyed proposal — it closes a
pause/resume hole) and questioned whether the composite key was needed at all. Resolution: two flags plus a
partial unique index (D-32); the composite-key question captured as a deferred idea, with the note that it would
not have solved the reuse problem either.
