**Attach seam and boot order**

- **D-20:** Venue assembly MOVES ABOVE portfolio rehydrate. New boot order: distinct-account check →
  venue assembly (from `venue_accounts`) → rehydrate. ACCT-01 makes the account set store-derived, which
  removes the only stated reason a portfolio's `VenueAccount` could not exist at construction time, so
  `_attach_venue_accounts` (116 lines) is **deleted** and **COMP-07 lands in this phase, not Phase 12**.
  Venue *registration* was already hoisted above `compose_engine` by 11.1-07 (`live_trading_system.py:1886-1936`),
  so only assembly moves. — **Reversibility:** costly — Phase 12's criterion 7 declares the current boot
  order "a hard invariant, not an implementation detail" and requires `test_distinct_account_invariant.py`
  to pass **unmodified**; that criterion and that test must both be amended, and COMP-07 must be struck
  from Phase 12's scope.

- **D-21:** Account resolution is **layered**. `VenueBundles.account_for(venue, account_id, **portfolio_args)`
  is the mode-agnostic construction primitive (both backtest arms already build a `VenueBundles` —
  `backtest_trading_system.py:210` and `:546`). `VenueAccountManager.account_for(...)` is the LIVE authority:
  it asserts the row exists and is enabled, raising the typed ACCT-03 refusal, then delegates. Only the
  manager can distinguish *"no such account"* from *"provisioned after boot, not assembled this boot"* —
  a distinction D-27 makes reachable. Rejected manager-only: it needs `VenueAccountManager(store=None)` on
  the backtest arm, the `Optional`-collaborator shape COMP-06 exists to delete.

- **D-22 — SUPERSEDED by D-24/D-25.** Recorded for the audit trail: this session first locked "`add_portfolio`
  drops `venue_name`/`account_id` and derives identity from a caller-supplied `Account`". Withdrawn once the
  code showed the account is built LAST in `add_portfolio` (`portfolio_handler.py:325-355`), after four things
  only the handler can compute — the resolved `portfolio_id`, a `state_storage` **scoped to that id**,
  `to_money(cash)`, and `enable_margin` off the resolved config preset. `add_portfolio` **keeps its plain-data
  signature**; the 133 call sites are untouched.

- **D-23:** `Portfolio.venue_name` / `.account_id` become read-only **properties over `self.account`** —
  one storage site for the pair, drift structurally impossible rather than guarded. Implies the `Account`
  leaf carries `venue_name` (it already carries `account_id`). Existing readers keep working because the
  attribute names survive. **Planner note:** with D-24 in place, `Portfolio` holding the definition is an
  equally valid single-source; D-23 is the default, not a refusal of that alternative.

- **D-24:** A composition-owned **`PortfolioFactory`** owns the definition→runtime-graph mapping,
  constructed with its real deps (`VenueAccountManager`, the state-storage factory, environment/sql_engine)
  and injected into `PortfolioHandler`. The handler shrinks to what a Handler means in this codebase —
  queue seam, registry, lifecycle — and **never holds `VenueBundles`** (`self._venue_bundles` is used at
  exactly three lines, `:108`/`:418`/`:426`, all inside `_new_compute_account`, which is deleted). *The
  owner rejected three prior framings as "moving the mess around": correcting the key `_new_compute_account`
  passes, having the caller pass a finished `Account`, and having the caller pass a pair-bound factory.*
  — **Reversibility:** costly — undoing returns account construction to the handler and re-conflates the
  two operations D-25 separates.

- **D-25:** The source defect is that `add_portfolio` conflates a durable **write** with a runtime **build**,
  and `rehydrate_portfolios` calls `add_portfolio` (`portfolio_rehydrate.py:130`) — so restoring and creating
  already share one method. The visible scar is `_persist_definition`'s **"GATED ON ABSENCE"**
  (`portfolio_handler.py:454`), which exists ONLY so rehydrate re-entering creation does not wipe the row it
  just read. Split them: `create` (write the row; the composite FK *is* the account check) and
  `materialize(definition) -> Portfolio` (resolve the pair via D-21, build id-scoped `state_storage`,
  construct the `Portfolio`). `add_portfolio` = create → materialize; **rehydrate materializes only**.
  The absence-gate is deleted, which closes **ACCT-07 structurally** rather than by patching the early
  return, and makes **ACCT-05 trivially true** — there is exactly one creation path.

**Provisioning**

- **D-26 — refines D-10's placement, not its content.** Provisioning lives on
  **`VenueAccountManager.provision(...)`**, reached as `system.venue_accounts.provision(...)`. There is
  **no `LiveTradingSystem.provision_venue_account`**. D-10's content — a direct call, not a queued
  `ConfigUpdateEvent` — stands; its "facade operation" wording assumed the wrong mirror, because
  `add_portfolio` is not on the facade either (scripts call `system.portfolio_handler.add_portfolio(...)`).
  `LiveTradingSystem` gains **zero lines**.

- **D-27:** An account provisioned at runtime is **usable in the same process**. The venue-account domain
  mirrors the portfolio domain one layer up: `provision` (write the row) + `bring_online` (build the runtime),
  with **one path** shared by provisioning and boot — boot becomes
  `for row in read_enabled_for(venue): bring_online(row)`, the bulk case of the same operation rather than a
  separate assembly path. D-20 already made assembly a standalone per-account step, so this is one assemble
  call on a single spec, not new machinery.

- **D-28:** Runtime `bring_online` is a **direct inline call**, the same threading posture as `add_portfolio`,
  which already writes SQL and mutates handler state off the engine thread — D-10 explicitly accepted that
  ("inherits an existing problem rather than inventing one"). One answer to fix later under LR-12, not two
  divergent ones. Rejected: a queue-native CONTROL event (provision would return before the account is usable,
  reintroducing two-step timing through the side door).

- **D-29:** **No new script.** `scripts/run_live_paper.py` already depends on boot-time minting — a fresh boot
  yields `account_ids = [None]` and mints `('paper','default')` before the script's `add_portfolio` at `:88` —
  so existing scripts become the callers, each gaining one `provision(...)` line. The **gate** is one
  integration test on the SQL spine covering the whole fresh-deployment loop: zero-account boot comes up
  RUNNING and empty (D-09) → `add_portfolio` before provisioning refuses with the typed account-domain error
  (ACCT-03 via D-21) → provision → bring_online → the next `add_portfolio` succeeds on a venue-truth account →
  a second provision of the same pair refuses (D-11).

**Portfolio lifecycle persistence**

- **D-30:** `PortfolioHandler` gains `deactivate_portfolio` / `activate_portfolio`, each flipping
  `Portfolio.set_state` AND writing `PortfolioDefinitionStore.set_enabled` as one operation.
  `Portfolio.set_state` stays a pure in-memory domain transition holding no storage. The store is already
  handler-owned, so nothing new is injected. Rejected: an injected state-change callback on `Portfolio`
  (puts persistence inside the domain object and makes `set_state` do I/O), and an event-driven persist
  (asynchronous relative to the flip — a crash between the two loses the deactivation, the exact failure
  ACCT-10 closes).

- **D-31 — SUPERSEDED by D-32.** Recorded for the audit trail: deletion was first locked as plain "soft delete,
  `enabled=False`". The owner's reuse workflow showed that collapses pause and retire into one flag, so a paused
  portfolio would release its account and re-enabling could then collide. D-32 splits them.

- **D-32 — two flags, and a partial unique index.** *Found during discussion:* `delete_portfolio`
  (`portfolio_handler.py:495-520`) flips to `ARCHIVED` and does `del self._portfolios[...]` but **never touches
  the definition store**, so a deleted portfolio's row stays `enabled=True` and it returns on the next boot with
  its persisted id and its child tables reattached. Deletion is **soft** — row, history and child tables survive.
  Refined by the owner's actual workflow (retire a portfolio, run a NEW one on the same real venue account):
  * `enabled` = **paused**. KEEPS its account; re-enable always works. This is ACCT-10's subject.
  * `is_deleted` = **retired**. RELEASES the pair.
  The `portfolios` unique constraint becomes **partial**: `UNIQUE(venue_name, account_id) WHERE NOT is_deleted`,
  with **both** `postgresql_where` and `sqlite_where` (tests run SQLite, production is Postgres). This revises
  `portfolio_definition_store.py:23`'s documented *"PLAIN (never partial, conditional or deferrable)"* — that
  rationale is about **simultaneous** sharing conflating buying power, and it was written when `enabled` had no
  writer at all (which is WR-04/ACCT-10 itself). Sequential reuse now works; simultaneous sharing is still
  refused by the DB, so the "binds out-of-band writers" half of the original rationale survives intact — and the
  partial index is self-enforcing in both directions (flipping a retired row back while a successor is live
  violates it). — **Reversibility:** one-way — this is the phase's only schema migration (add `is_deleted`,
  swap the constraint for a partial index); reverting needs a second migration and would re-bind every account
  to the first portfolio that ever used it.
  **Planner notes:** rehydrate and `read_all` must filter `is_deleted`; `assert_distinct_accounts` must consider
  live rows only; a fresh portfolio taking over an account its predecessor left non-flat will trip the D-04
  unexplained-residual baseline guard — that is the guard working correctly, but it needs a message that says so.

- **D-33:** Account-`enabled` and portfolio-`enabled` are **independent** — disabling a portfolio never touches
  its account (owner-stated). `set_enabled(account, False)` **refuses** while any live portfolio references it
  (retire or pause them first; D-30/D-32 give the way). If the state arrives out-of-band anyway, the account is
  not brought online and its portfolios hit ACCT-03's refusal, so the boot fails loud rather than trading blind.

**Store reads, failure policy, and the facade constraint**

- **D-34:** `VenueAccountManager.read_enabled_for(venue)` returns `[]` **with a warning** when the
  `venue_accounts` table is missing (the D-09 fresh-deployment state); **any other fault propagates loud**.
  Under DB-as-sole-truth a swallowed read means zero accounts, so every persisted portfolio then fails ACCT-03 —
  N derived errors hiding one real cause. The whole policy lives INSIDE the manager and is unit-testable without
  booting a system. (Note: `read_enabled_for` does not exist yet — `VenueAccountStore` has only `upsert` /
  `record_venue_uid` / `get` / `read_all`.)

- **D-35 — phase-wide constraint, owner-stated twice.** **Nothing is added to
  `itrader/trading_system/live_trading_system.py`.** Every new behaviour lands on `VenueAccountManager`,
  `PortfolioFactory` or `PortfolioHandler`. That file may only SHRINK in this phase: out go
  `_account_ids_for_spec`, `_mint_account_rows`, `_build_account_specs`, `_attach_venue_accounts` and the
  `feed.set_provider` block; in come two loop lines. Any plan that adds a function or a branch there is wrong
  regardless of how the rest of the dependency graph falls out.

- **D-36 — "primary account" is DELETED as a concept.** *Found during discussion:* `_primary_lifecycle` has
  **zero production consumers** (read at 28 sites across 7 test files, nowhere in `itrader/`), and the data path
  provably does not care which account — `okx_plugin.build_provider` resolves an account-keyed connector but
  configures everything from `ctx.config.stream`, using only `connector.sandbox`/`ws_hostname` for host routing,
  `connector.client` for the **public** `fetch_ohlcv`, and the asyncio loop. **The feed is bound to a
  *connection*, not to an account.** It borrows the first available connection and **re-borrows if that account
  is retired or disabled**, logging the switch and warning when none is left. Bars are public data, so a switch
  changes transport not content, and `LiveBarFeed`'s monotonic guard covers the reconnect. This also closes
  D-27's open gap: a zero-account boot has no feed, and the first provisioned account gives it one with no
  restart. `feed.set_provider` moves OUT of `live_trading_system.py` (a deletion there, per D-35). No flag, no
  column, no ordering contract; `_primary_lifecycle` becomes a renamed test-only helper.
  **Verified blocker for full decoupling:** `OkxConnector.__init__` defaults to `OkxSettings()`, but the
  credential fields are *required* on that model — `okx_plugin.py:136-150` computes
  `required = {…if field.is_required()}` and raises `CredentialResolutionError` rather than letting the ambient
  environment complete a partial set. A credential-free data connection means loosening that model, which is the
  foundation of a deliberate fail-closed gate. Deferred (see below), not rejected.

- **D-38:** D-12's precedence merge happens in **`VenueAccountManager.bring_online`**, when it builds the

## Summary

This is a brownfield structural refactor with **zero external-dependency surface**. No package is
added, upgraded, or evaluated. The entire risk profile is *internal factual accuracy*: the phase's
23 locked decisions cite roughly 90 exact `file:line` anchors, CONTEXT.md was written 2026-07-23
against a tree that three quick tasks (`260722-g6w`, `260723-dc4`, `260723-dq7`) had just moved,
and the ROADMAP/REQUIREMENTS line numbers predate all three.

**The good news, stated plainly:** `11.2-CONTEXT.md`'s own `<canonical_refs>` block is
**substantially accurate** — its four deletion anchors (`:1439`, `:1526`, `:1576`, `:1614`), its
`_venue_bundles` three-site claim, its `_persist_definition:454` "GATED ON ABSENCE" citation, its
`VenueAccountStore` method-set claim, and its per-file indentation table all verify exactly. The
**ROADMAP and REQUIREMENTS line numbers are the stale layer**, and D-41 already caught one class of
that staleness. A planner that treats CONTEXT.md as primary and ROADMAP line numbers as advisory
will be right far more often than not.

**The bad news, which is what this document is for:** fourteen anchors are wrong in ways that would
produce a wrong plan, and three of them are wrong in the specific way that produced the Phase 10
failure mode — an edit that lands on the wrong line, passes every gate, and leaves the real defect
in place. The three highest-value corrections are **F-1** (ACCT-04's coercion count is 8, not 6, and
two of the extra sites are in `okx_plugin.py`, a file no decision mentions), **F-2** (both ACCT-06
docstring anchors are off, one "real" anchor is off, and an uncited third phantom site exists),
and **F-4** (the test blast radius is 10 files / 53 references, not the single file D-37 names).

**Primary recommendation:** plan from `11.2-CONTEXT.md`'s `<canonical_refs>` block plus the
corrected anchor table in this document; treat every `file:line` in ROADMAP.md and REQUIREMENTS.md
as unverified. Give every executor the §Anchor Audit table and the standing instruction that
**code wins over plan text** — the same brief that caught the Phase 10 defects.

## Architectural Responsibility Map

This phase is single-tier (a Python backend library). The meaningful axis is not client/server but
**which internal owner** a capability lands on — which is exactly what D-13/D-24/D-26/D-35 legislate.

| Capability | Primary Owner | Secondary Owner | Rationale |

### Pattern 1 — Manager, not Handler (D-13)

**What:** the venue-account domain gets a `VenueAccountManager` that owns storage and business logic
and has **no queue access**.
**When:** whenever a domain has state and rules but no event types. `EventType` has no venue member.
**Precedent in-tree:** `OrderManager` (business logic, no queue) behind `OrderHandler` (queue seam).

```python
# Source: the codebase's own convention (CLAUDE.md §Function & Module Design;
# itrader/order_handler/order_manager.py is the reference implementation)
class VenueAccountManager:
    """Business logic + storage for the venue-account domain. No queue, no handler back-ref."""
    def __init__(self, store, bundles, connectors, *, venue_name, alert_sink) -> None:
        self._store = store            # collaborators are RECEIVED, never minted (11.1 principle)
        self._bundles = bundles
        ...
```

### Pattern 2 — degrade-on-missing-table, propagate everything else (D-34)

**What:** a missing table is the fresh-deployment state; any other fault is loud.
**Existing precedent D-34 explicitly mirrors** — `live_trading_system.py:2059-2070`:

```python
# Source: itrader/trading_system/live_trading_system.py:2059-2070 (VERIFIED in-tree)
# "deliberately narrow — it probes for the TABLE and nothing else, so a genuine store
#  fault still propagates LOUD out of rehydrate rather than being converted into the
#  silent zero-portfolio boot D-19 forbids."
has_table = definition_store is not None and _sa_inspect(
    system_db_backend.engine).has_table("portfolios")
if definition_store is not None and not has_table:
    logger.warning("The `portfolios` table is not provisioned — ...")
```

Relocate this shape into the manager; **do not** widen it to a bare `except Exception`.

### Pattern 3 — rejection as an event, not an exception (D-42)

**What:** `ExecutionHandler` never raises across the queue boundary; it emits `FillEvent(REFUSED)`,
which also reconciles the order mirror out of `PENDING`.
**Precedent:** `SimulatedExchange.execute_order()` returns `ExecutionResult(success=False, ...)` and
emits `FillEvent(REFUSED)`; `OrderHandler.on_fill` maps `REFUSED → REJECTED`.

### Pattern 4 — lazy concretion imports inside plugin/live bodies (GATE-01)

**What:** every OKX / SQL / async import stays **inside a function body**, never at module top, and
new live-only modules stay out of package barrels.

## Common Pitfalls

### Pitfall 1 — GATE-01 inertness is a POSITIVE assertion now, not just a name list

**What goes wrong:** a new live-only module (`venue_account_manager.py`) gets exported from
`itrader/venues/__init__.py`, or imports `sqlalchemy` at module top, and the backtest import path
pulls SQL.
**Why it happens:** the `venues/` barrel is the natural place to export a new venue-domain class.
**How to avoid:** CONTEXT.md is explicit — *"`VenueAccountManager` stays out of it."* Keep SQL imports
lazy in-body.
**Warning signs:** `tests/integration/test_okx_inertness.py` fails. Note the gate has **two** halves
`[VERIFIED: read]`: a `_FORBIDDEN` name list (`:38-...`, checked at `:115-119`) **and** a
strictly-stronger positive assertion added in Phase 10.1 — *"the backtest import graph pulls NO SQL,
no matter which module carries it"* (`:121-127`). A module not on the name list is still caught.

### Pitfall 2 — the oracle test drives the LEGACY construction arm

**What goes wrong:** D-24's `PortfolioFactory` is wired into `compose.py` and both backtest roots, the
oracle stays byte-exact, and everyone concludes the factory path is covered. It is not.
**Why it happens:** `tests/integration/test_backtest_oracle.py` drives
`scripts/run_backtest.py:68` — the legacy construction arm — so **a factory-only edit goes
false-green** (recorded in CONTEXT.md §Gates and in the project's memory).
**How to avoid:** any plan touching `PortfolioFactory` needs its own direct assertion; the oracle is a
**regression** gate, not a **coverage** gate.
**Warning signs:** a plan whose only verification for a factory change is "oracle byte-exact".

### Pitfall 3 — tab/space mixing (the change set spans both)

**What goes wrong:** an edit to `execution_handler.py` (**TABS**) or `compose.py` (**TABS**) is written
with 4 spaces; the file breaks or the diff is unreviewable.
**Why it happens:** most of the *new* files in this phase are 4-space, so an executor generalizes.
**How to avoid:** the measured table in §Environment Availability. **Never generalize per package** —
`itrader/trading_system/` contains both (`live_trading_system.py` 4-space, `compose.py` and
`backtest_trading_system.py` TABS).
**Warning signs:** a verification step phrased as "no space-indented lines in the file" — that gate
**false-fails** on untouched tab files with wrapped docstring prose (recorded failure mode). Scan
**added diff lines only**.

### Pitfall 4 — `mypy` cannot see regressions in `live_trading_system.py`

See **F-10**. Dead code and unused imports left by the five deletions pass both gates. Add an explicit
sweep and a `grep` assertion for the deleted symbol names.

### Pitfall 5 — the credential-gated e2e file is skipped, and it references deleted helpers

See **F-3**. `tests/e2e/test_okx_sandbox_recon.py` holds **10** references to symbols this phase
deletes/renames and is **SKIPPED** in CI (no OKX credentials). A green suite proves nothing about it.
Use a static `grep` gate.

### Pitfall 6 — ACCT-01 and ACCT-02 must land in the same wave

D-39 states it; the failure mode is asymmetric and worth restating: store-derived-set **with** minting
still present is a harmless no-op, but minting deleted **without** a replacement makes the composite FK
reject the first `add_portfolio` on a fresh DB — **no portfolio can ever be created**. A wave boundary
between them produces a system that boots and then cannot do anything.

### Pitfall 7 — the WR-11 fixture fails *after* ACCT-03, not with it

`test_multi_portfolio_lifecycle.py` is a flagship multi-portfolio test whose fixture becomes illegal
the moment ACCT-03 lands. D-39 puts both in W3 and D-37 marks the fixture mandatory. A plan that
schedules ACCT-03 without the fixture rewrite **in the same plan** produces a red wave.

### Pitfall 8 — SQLite tests never run Alembic

See **F-5**. The D-32 schema change must be made in **both** `build_portfolio_definition_tables` (which
`provision_schema` uses for SQLite tests) and the new migration (which Postgres production uses). A
migration-only change is invisible to the entire test suite; a registrar-only change never reaches
production.

## Project Constraints (from CLAUDE.md)

Extracted directives the planner must verify each plan against:

1. **Queue-only cross-domain writes.** Components take `global_queue` in the constructor and emit
   events; they never call other handlers directly. Read-only cross-domain access goes through an
   injected read-model (`PortfolioReadModel`, `BacktestBarFeed`). — *Relevant: `VenueAccountManager`
   has no queue (D-13); `PortfolioFactory` is a composition collaborator, not a handler.*
2. **Money is `Decimal` end-to-end.** Enter via `to_money(x)`; **never** `Decimal(float)`. `float()`
   only at the serialization/logging edge. — *Relevant: `paper_plugin.py:142`'s `0.0` default is a
   live violation this phase closes.*
3. **Single UUIDv7 scheme** via the `idgen` singleton. Do not introduce a second ID scheme.
4. **Determinism:** one shared seeded `random.Random` injected at wiring; never seed per-call.
5. **Indentation: match the file, never normalize.** Handler modules use tabs; `config/`, `core/`,
   `price_handler/feed/`, `events_handler/events/` use 4 spaces. **Measure per file** — see the table
   below.
6. **Typed exceptions, not bare `Exception` or boolean returns.** Domain exceptions live in
   `itrader/core/exceptions/`. — *Relevant: ACCT-03's refusal and D-11's duplicate refusal both need
   typed account-domain errors.*
7. **Test strictness:** `filterwarnings=["error"]`, `--strict-markers`, `--strict-config`. Any
   unexpected warning fails the suite; every marker must be declared. — *Relevant: D-34's warning-on-
   missing-table must be a `logger.warning`, not a `warnings.warn`, or it fails the suite.*
8. **Docstrings are decision-anchored and load-bearing.** Modules cite decision tags (`D-01`,
   `M5-04`, `T-07-14`). Preserve the style; these are references to planning artifacts. — *Relevant:
   ACCT-06 is entirely about keeping these accurate.*
9. **`<Domain>Handler` = thin queue seam; `<Domain>Manager` = business logic, no queue, no handler
   back-reference.** Layering is one-directional: facade → manager → storage.
10. **Import side effects:** `itrader/__init__.py` initializes `config`, `logger`, `idgen` on import.
11. **Backtest error policy is fail-fast; live is publish-and-continue.** Documented as intentional in
    `.planning/codebase/CONVENTIONS.md` — do not "fix" the asymmetry.
12. **Bar-timing contract** (`price_handler/feed/bar_feed.py`) is the single home of look-ahead safety.
    — *Relevant to D-36's feed re-binding: `LiveBarFeed`'s monotonic guard is what covers a connection
    switch.*
