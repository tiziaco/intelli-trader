<objective>
Reject the `enable_margin` runtime flip: drop it from the reachable routable key set in
`ConfigRouter`, and raise in `Portfolio.update_config`.

Purpose: `enable_margin` is a field of `TradingRules`, and `_PORTFOLIO_MUTABLE_SECTIONS` includes
`trading_rules`, and `_resolve_portfolio_section` introspects `model_fields` — so the key is routable
**today**. `_apply_portfolio` then calls only `portfolio.update_config(update)`, which does
`recursive_merge → model_validate → atomic swap` and **notifies nobody**. Four of the five readers
bind the value at construction (`AdmissionManager`, `EnhancedOrderValidator`, `ManagedStrategies`
/SHORT-01, and the account leaf's very *kind*); only `Portfolio.process_transaction` follows the
flip. So a runtime flip already desynchronizes four of five readers today. Making it genuinely
mutable means propagating to all five; **rejecting is the smaller change and matches the phase's
fail-loud posture** — this is the locked owner decision.

Output: a two-layer, key-scoped, typed refusal.

## Trust Boundaries

|----------|-------------|
| operator / external caller → `ConfigRouter` | The default-deny allowlist decides which portfolio configuration an out-of-band caller may mutate at runtime. |
| `ConfigRouter` → `Portfolio` domain object | A caller reaching `Portfolio.update_config` directly bypasses the router entirely. |

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
