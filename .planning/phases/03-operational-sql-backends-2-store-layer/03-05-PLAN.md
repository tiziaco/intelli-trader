<objective>
Wire the framework's Phase-1 Alembic skeleton to the operational MetaData and autogenerate the first operational-baseline migration (D-09, MIG-01 continuation). Today `env.py` ships a bare `target_metadata = MetaData()` (empty Phase-1 chain). This plan points it at a `NAMING_CONVENTION`-stamped MetaData with all operational tables registered via the three `build_*_tables` functions, then generates and hand-reviews one baseline revision into the single framework chain. Tests keep using `create_all` (no Alembic in tests, D-09); the ephemeral results store keeps no `alembic_version`.

Purpose: gives the live deploy path a deterministic, framework-owned migration for the operational tables — the FastAPI app will `alembic upgrade head` on this chain and keep its app-specific tables in a separate chain.
Output: an edited `env.py`, one autogenerated baseline revision under `versions/`, and (optional) a drift-guard extension to `test_migrations.py`.

NOTE: generating the migration requires a reachable Postgres (autogenerate reflects the empty DB to diff) — use testcontainers or a local Postgres on port 5544. There is NO Dockerless fallback for GENERATING the migration (research §Environment Availability). The executor MUST have Docker/Postgres available for this plan.
</objective>
