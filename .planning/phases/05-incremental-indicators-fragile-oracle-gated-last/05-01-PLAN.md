<objective>
Build the **shared recent-bars feed data layer** (Plan A): promote `BarFeed` to own a shared
recent-bars API — a **newest-bar provision** plus a **consumer-registration / capacity-derivation
INTERFACE** that mirrors `universe/instruments.py::derive_instruments` (P5-D16). Unify the
newest-bar pass so one per-symbol-per-tick walk feeds both the `BarEvent` payload and the cache
newest-row write (P5-D16a / G5). Define the update-trigger seam interface-only, asserting
`base_timeframe <= min(timeframe)` (P5-D16b / G1).

Purpose: Plan B's stateful indicators (Model B, P5-D07) self-buffer and do NOT read this cache, and
Plan A deliberately **DEFERS the deep multi-bar cache** (P5-D16/P5-D22 — no current consumer needs
deep history). This plan therefore ships the newest-bar provision + the registration interface only,
so the screener / raw-history-strategy consumer can extend it later with zero structural change.

Output: extended `BarFeed` ABC, the new pure `cache_registration.py` derive function, the unified G5
walk + interface-only G1 trigger seam in `bar_feed.py`, and integration coverage. This plan is
**byte-exact** — it adds plumbing only; the SMA_MACD oracle stays `134 / 46189.87730727451`.
</objective>
