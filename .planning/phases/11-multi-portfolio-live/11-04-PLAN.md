<objective>
Land the W2 credentials boundary (D-28): the seam that lets each `(venue, account_id)` connect with
its OWN credentials, and the trust-on-first-use guard that notices when it does not.

The owner's stated target was credentials stored in the database, driven by a real workflow — an
integrations page in the web app where venues are connected by entering keys, with no `.env`
involvement. **Two blocking findings redirected it, and this plan must not re-litigate them:** the
venue store already ships a recursive any-depth secret denylist raising on exactly this, added
deliberately earlier in this milestone; and the milestone-wide zero-new-dependency gate blocks every
secret-manager SDK across P1-P12 and also blocks promoting the transitively-present crypto library to
a direct dependency. Envelope encryption would only relocate the problem (the master key still needs
a safe home — the secret manager you were going to run anyway), while a decryptable blob puts live
exchange keys into every database dump, read replica, and backup snapshot.

So the answer is the pattern this milestone has now established three times — **`itrader` ships the
seam, the app owns the data**: a `CredentialResolver` Protocol plus an env-backed implementation, over
a `secret_ref` POINTER on the durable row.

D-04 belongs in this phase specifically because per-account credentials are what make the misroute
REACHABLE: a mistyped `secret_ref` or a swapped vault entry means an account id connects with a
different account's keys, orders route to the wrong REAL account, and reconciliation then succeeds
cleanly against it. That is a silent, money-losing failure that only exists once multi-account exists.

Output: the Protocol, the env resolver, `credential_model` on the plugin Protocol, and a
trust-on-first-use UID guard wired into the post-connect seam.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `secret_ref` pointer -> credential material | The pointer is operator-authored and durable; the material it names is not. A wrong pointer silently produces a wrong-but-valid authenticated session |
| resolver -> connector -> venue | Resolved credentials cross into an authenticated session; anything logged or persisted on this path is a live-key leak |
| venue -> engine (account UID) | The venue's self-reported UID is the only external evidence that the session belongs to the account the engine thinks it does |

## STRIDE Threat Register (ASVS L1, block on `high`)
