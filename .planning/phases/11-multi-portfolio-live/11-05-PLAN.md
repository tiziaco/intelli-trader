<objective>
Land the identity plumbing that every later wave depends on: make `portfolio_id` supplyable (F-1),
give every portfolio an `account_id` (D-06), derive `exchange` from `venue_name` instead of storing it
twice (D-07), and add the `account_for` read-model seam that plan 11-06 needs to route an order to the
right account's exchange (D-27).

**F-1 is a real, confirmed, pre-existing defect and it is larger than it looks.** `Portfolio.__init__`
mints a fresh UUIDv7 on every construction with no way to pass an existing one, while two separate
in-tree comments assert restart-stability that does not exist — one on `Portfolio` claiming state
survives a process restart, one in the live composition root claiming strategy rehydrate binds to ids
that already exist and are restart-stable. Both hold only within a single process. On restart the
prior run's portfolio-state rows are orphaned and the persisted strategy-subscription rows point at
portfolios that no longer exist. Phase 11 is where this stops being survivable, because per-portfolio
reconciliation must match a portfolio to its venue account **across** a restart.

**F-5 is why these are one plan and not three:** the three changes land on the same lines of the same
two methods. Planning them separately invites a merge conflict on the exact lines that matter.

**Sequencing note (planner-resolved refinement of D-28):** the seven-wave decomposition groups
"pinnable `portfolio_id`" under W4 Bootstrap and account minting under W3 Accounts. But `account_for`
reads `portfolio.account_id`, so the signature change is a hard prerequisite of the W3 exchange
keying. This plan therefore runs the signature change early. The phase is not split — the work simply
sequences in dependency order.

Output: two rewritten signatures, one derived attribute, one new read-model method on both the
Protocol and its implementation, `PortfolioSpec.account_id`, and a unit gate proving restart-stability
and the untouched default path.
</objective>
