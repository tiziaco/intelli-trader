<objective>
Land the W4 bootstrap boundary (D-28): portfolios rehydrate from their definition rows at boot, in the
one position in the live composition builder that satisfies every constraint at once, and a
distinct-account invariant refuses to start on a collision before any account is minted.

**This plan contains a genuine ordering change that CONTEXT did not flag and RESEARCH found.** The
strategy-rehydrate block carries an unusually explicit four-constraint justification comment. Read
literally, its first constraint says portfolios are already layered above, so subscription binds to ids
that already exist and are restart-stable. **That comment is doubly wrong today.** The layering call
does not create portfolios — it iterates existing ones, and the reason the ordering "holds" is that
the live system has ZERO portfolio-creation call sites, so it boots with zero portfolios and the
constraint is vacuously satisfied. And "restart-stable" was the false claim plan 11-05 fixed. Making
portfolios real makes the first half true for the first time — and simultaneously turns the layering
call into a silent no-op unless it MOVES BELOW the rehydrate. That is a near-miss of exactly the
silent-corruption class this repository's Phase 10 retrospective flagged. **The comment must be
rewritten, not merely inserted above.**

Purpose: without a definition row and a rehydrate, per-portfolio reconciliation cannot match a
portfolio to its venue account across a restart — which is the whole point of the phase.

Output: a rehydrate collaborator, a two-source invariant, the corrected boot ordering, a rewritten
comment, and gates at both invariant layers.
</objective>
