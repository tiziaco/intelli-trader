<objective>
Complete the W1 schema boundary (D-28) with the D-29 migration chain: two hand-authored Alembic
revisions off the current head `p10_strategy_portfolio_subs`, plus the D-09 config move that relocates
per-portfolio config from a STATE row onto the new DEFINITION row.

**This plan carries the single highest-regression-risk edit in the phase, and the risk is not that it
breaks loudly — it is that it fails silently.** Verified: `load_config()` returning `None` is guarded
by a truthiness check and wrapped in a warning-only degrade-clean, so a migration that repoints reads
without moving data produces no exception, no warning, a successful boot, a fully green test suite,
and every live portfolio silently running on default configuration. RESEARCH found no shipped analog
for the data-movement step — all eleven existing revisions are pure DDL; nothing in the chain has ever
moved data between tables. Plan it with proportionate care.

Purpose: config belongs on a definition row, not a state row. It only ever lived on the state row
because no definition row existed. Splitting "create new tables" (revision 1) from "modify existing
data" (revision 2) puts the risk and the guards where they belong, under separate downgrade paths.

Output: two revisions, a rehomed `save_config`/`load_config`, and a migration test that asserts the
migrated VALUE.
</objective>
