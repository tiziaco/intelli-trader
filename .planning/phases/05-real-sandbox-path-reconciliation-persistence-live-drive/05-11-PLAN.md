<objective>
Close WR-02 (correlation-map adoption for rehydrated orders) and WR-03 (reconcile_manager mutation
ordering), and record the WR-05 documentation-only note.

- **WR-02 (RECON-05):** rehydrated (pre-restart) orders never repopulate `OkxExchange`'s in-memory
  correlation maps (`_orders_by_venue_id`, `_venue_id_by_order_id`, `_orders_by_clOrdId`), which are
  populated ONLY by `_submit_order` — which never runs for a pre-restart order. Consequence: a fill
  for a rehydrated resting order after restart is buffered under `_pending_fills_by_venue_id` and never
  drained (silently lost); a cancel of a rehydrated order is a silent no-op. This becomes reachable now
  that CR-01 (05-10) spawns the live fill stream. Fix: add an `adopt_venue_correlation(order)` seam on
  `OkxExchange` and call it during restart rehydration (from `VenueReconciler`).
- **WR-03 (RECON-02):** `reconcile_manager.py:177-200` mutates `filled_quantity` BEFORE
  `add_state_change` validates the transition (dormant only because `allow_same_status=True` never
  fails today). Reorder so the transition is validated before the mutation.
- **WR-05 (INFO, DOCUMENT ONLY — no code):** `_compare_symbol_drift`'s adopt-and-continue branch
  (`portfolio_handler.py:733-743`) logs without correcting engine state; dormant because
  `_drift_reconciler` defaults None. Record it as an intentional extension point so a future reviewer
  does not re-flag it — write NO code for it.

Purpose: make post-restart fills/cancels for rehydrated orders actually reach the mirror (RECON-05),
and remove the latent WR-03 mutation-before-validation trap. Output: the adopt seam + its wiring, the
reorder, two regression tests, and the WR-05 note.
</objective>
