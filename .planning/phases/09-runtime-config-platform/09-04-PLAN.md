<objective>
Build the UI read-model half (RTCFG-06): a new `system_stats` append-only table + store (its own table, cloning the `equity_snapshots` shape — D-18) holding ONLY engine-operational counters no domain store owns, written event-driven by a thin engine-thread stats writer; the `state.*` writers (`state.status`/`state.halt_reason`/`state.last_started_at`) at their own event sources into `SystemStore` (D-19, alongside the existing `state.last_error`); and the phase's migration finalization. The UI reads entity data directly from its own domain store (equity from `portfolio_account_state`/`equity_snapshots`, orders from the order store, halts from `halt_records`) — NO entity duplication (D-17). All lock-free DB reads (RTCFG-06).

This plan is the phase's **migration-owner**: it chains ALL of P9's new tables single-head as `strategy_registry` → `module_config` (creates the D-25 `order_config`/`portfolio_config` config tables from Plan 03's registrars) → `system_stats`, registers the three new registrars in `migrations/env.py`, and extends the `test_migrations.py` parity + single-head gate to cover them. The **read-model logic itself is unaffected by the D-21/D-25 config-store move** (it is `state.*` + `system_stats` + domain-store entity reads — it never reads config back from `SystemStore`); only the migration chain gains the two config tables.

Live-only, backtest-dark — the backtest oracle stays byte-exact and inertness stays green.

Purpose: give the (future FastAPI) UI a lock-free operational + state read surface without an aggregation layer.
Output: `system_stats` store/table + migration + env.py registration, the thin stats writer + `state.*` writers, and the store unit test.

Self-gated planner contribution (recorded): schema-gate = no-op — iTrader uses SQLAlchemy + Alembic migrations, not a push-based ORM, so the migration IS the schema change (no "schema push" task).
</objective>
