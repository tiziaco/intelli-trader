<objective>
Add the durable config surface + wire the durable/restart half of the platform (D-21/D-22/D-23/D-25). First (D-25 storage-mechanism refinement) give each module's OWN store a `save_config`/`load_config` surface — the order store (`OrderStorage`) persists a GLOBAL singleton config to a NEW dedicated `order_config` table, and the per-`{id}` portfolio store (`PortfolioStateStorage`) persists its bound-`portfolio_id` config on a NEW nullable `config_json` COLUMN added to the EXISTING `portfolio_account_state` carrier (no new stores, no new `portfolio_config` table — extend `build_portfolio_tables`), so config is persisted by its owning module, never centralized in `SystemStore`. Then extend `add_event`'s D-10 fail-closed allowlist to admit `CONFIG_UPDATE` as the third external type with ingress 400-style validation (D-23), and in `build_live_system` construct `SystemStore`/`VenueStore` over the shared `SqlEngine` and inject the OWNING stores per D-21 (`system_store`, `venue_store`, the ORDER store via `order_handler.storage`, and the `portfolio_handler` reaching each Portfolio's bound `state_storage`) + the `config` + push handlers + venue-kind resolver into the `ConfigRouter` (Plan 02), plus the boot layering `defaults ← YAML ← env ← persisted` (D-10/D-22) reading each scope back from its OWN store so persisted overrides survive restart. With no FastAPI driver (LR-01), P9's OWN tests drive the external `add_event(ConfigUpdateEvent(...))` path directly (D-23 — mandatory).

This satisfies RTCFG-01/03 through the owner-override model: there is NO `RuntimeConfig` overlay injected as `EngineContext.config` (D-05/D-06) — the imported `config` aggregator IS the runtime config, and `EngineContext.config` stays vestigial.

Purpose: make config changes durable and restart-surviving, and make the mutation path reachable from the (future FastAPI, now test) external surface.
Output: the extended `add_event`, the store construction + restart layering in `build_live_system`, and the two mandatory integration tests.
</objective>
