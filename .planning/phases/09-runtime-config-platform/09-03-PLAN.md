<objective>
Open the external `CONFIG_UPDATE` ingress and wire the durable/restart half of the platform (D-22/D-23). Extend `add_event`'s D-10 fail-closed allowlist to admit `CONFIG_UPDATE` as the third external type with ingress 400-style validation (D-23), then in `build_live_system` construct `SystemStore`/`VenueStore` over the shared `SqlEngine`, inject them + the `config` + push handlers + venue-kind resolver into the `ConfigRouter` (Plan 02), and add the boot layering `defaults ŌåÉ YAML ŌåÉ env ŌåÉ persisted` (D-10/D-22) so persisted overrides survive restart. With no FastAPI driver (LR-01), P9's OWN tests drive the external `add_event(ConfigUpdateEvent(...))` path directly (D-23 ŌĆö mandatory).

This satisfies RTCFG-01/03 through the owner-override model: there is NO `RuntimeConfig` overlay injected as `EngineContext.config` (D-05/D-06) ŌĆö the imported `config` aggregator IS the runtime config, and `EngineContext.config` stays vestigial.

Purpose: make config changes durable and restart-surviving, and make the mutation path reachable from the (future FastAPI, now test) external surface.
Output: the extended `add_event`, the store construction + restart layering in `build_live_system`, and the two mandatory integration tests.
</objective>
