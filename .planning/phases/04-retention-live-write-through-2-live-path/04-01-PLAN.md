<objective>
Build `CachedSqlOrderStorage` — the live-only decorator wrapper for the order seam (D-04) — composing the
untouched Phase-3 `SqlOrderStorage` (system of record) with an in-memory `InMemoryOrderStorage` working set.
It implements the 14-method `OrderStorage` ABC by forwarding store-first (persist-then-acknowledge, Pitfall 8),
purges terminal orders on terminalize behind a mandatory terminal-state gate with the bracket-parent-resident
invariant (D-02), serves the open set from cache and terminal/cold records via read-through to Postgres, and
rehydrates open-only on restart (D-03). Wire the factory `'live'` arm to return the wrapper — the order seam is
the one exercised end-to-end (`LiveTradingSystem` already calls `OrderStorageFactory.create('live', …)`).

Resolves research assumption **A1** (cross-method atomicity): per-write store-first, FK-ordered (parent before
children); within-method atomicity is the Pitfall-8 verification target; cross-method bracket atomicity is
documented as N+4 reconciliation's job — NOT a Phase-4 failure (honors D-04 unchanged ABC + D-01 un-rewired root).
Resolves **A4**: read-through is daemon-only as-wired but built API-thread-safe with one `threading.RLock`.
Resolves **A5**: the new module enters `mypy --strict` (siblings are not overridden — do NOT add an override).

Purpose: turn the Phase-3 order store into a long-running, restart-safe live system of record without touching
the gate-passed `SqlOrderStorage` (D-04 — a cache bug cannot compromise the store's proven correctness).
Output: `cached_sql_storage.py`, the wired factory `'live'` arm, and the testcontainers Postgres integration suite.
</objective>
