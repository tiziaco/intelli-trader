<objective>
Build `CachedSqlPortfolioStateStorage` — the live-only decorator wrapper for the portfolio-state seam (D-04) —
composing the untouched Phase-3 `SqlPortfolioStateStorage` (one instance per Portfolio, bound `portfolio_id`)
with an in-memory `InMemoryPortfolioStateStorage` working set. It implements the 21-method
`PortfolioStateStorage` ABC store-first (Pitfall 8), serves open/current state from cache and history via
read-through (D-02), and rehydrates open-only on restart (D-03). Per **D-01/A3**, the wrapper is **built +
component-tested but NOT wired into the live composition root** (`portfolio.py:93` stays `"backtest"`); the
factory `'live'` arm is wired so the wrapper is constructable, and restoration of the accumulators INTO the
managers is N+4.

Resolves research assumption **A2** (account-state carrier): adopt the **dedicated single-row
`portfolio_account_state` upsert table** (cleaner, O(1) latest, no equity-curve pollution, matches RETAIN-02's
"current account snapshot stays resident") over reusing `equity_snapshots`. This costs one small Alembic
migration on the **framework** operational chain — in-bounds because Phase-3 D-09 established the framework owns
that chain. The table holds `{portfolio_id, cash_balance, realized_pnl, total_equity, peak_equity,
open_positions_count, updated_time}`; the wrapper writes it via `save_account_state` and reads it via
`load_account_state`. Per **A3**, Phase 4 builds + component-tests the wrapper's *ability* to persist / return /
rehydrate these scalars; wiring the synchronous call on every fill and restoring `_balance` /
`_realised_pnl_accumulator` into `CashManager` / `PositionManager` is **N+4**.

Resolves **A5**: the new module enters `mypy --strict` (no override). The Phase-3 `SqlPortfolioStateStorage`
class stays **UNTOUCHED** (D-04) — adding the new table to the model builders is additive and does not modify it.

Purpose: a restart-safe live portfolio-state store with a bounded working set, without touching the gate-passed
Phase-3 store. Output: the `portfolio_account_state` table + migration, the wrapper, the wired factory arm, and
the testcontainers Postgres integration suite.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| daemon write → Postgres portfolio-state store | account-state + position writes; a crash mid-write must not leave the store ahead of acknowledged engine state |
| persisted rows → rehydration | the boot path reads back open positions + the account-state row; trust bounded to typed columns (Numeric/Uuid/UtcIsoText), no eval/pickle |
| portfolio A ↔ portfolio B | one wrapper instance per Portfolio; reads must never cross the bound portfolio_id |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03 | Tampering / Info disclosure | rehydration row-reads (reservations / locked-margin / account-state) | mitigate | every wrapper read is parameterized Core scoped to the bound `self._portfolio_id` (`.where(col == self._portfolio_id)`); no cross-portfolio bleed — verified by `test_cross_portfolio_isolation` |
| T-04-01 | Tampering | account-state upsert SQL | mitigate | parameterized Core insert/delete on the Table object — never f-string SQL; reuse the Phase-3 delete-then-insert idiom |
| T-04-02 | Information disclosure | logger bind | mitigate | never log the resolved DB URL; the wrapper sources the injected SqlBackend, never re-resolves creds (SEC-01) |
| T-04-08 | Tampering | crash mid-write (durability) | mitigate | store-first persist-then-acknowledge (Pitfall 8); account-state upsert rides the synchronous txn so the latest persisted state is never behind the working set (D-03) — verified by `test_crash_restart_accumulators` |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | Phase 4 installs NO packages — nothing to verify |
