## Summary

Phase 05.1 lifts the short-increase admission rejection behind the `allow_increase`
flag, making the SELL-add gate byte-symmetric with the existing long INCREASE gate.
The diff is a single hunk in `admission_manager.py` (the SELL branch of
`_enforce_position_admission`, lines 537-565) plus four new/changed tests and two
e2e fixtures.

The core logic change is **correct**. I verified:

- **Gate conditional correctness.** The new SELL branch correctly checks
  `snap.side is PositionSide.SHORT`, rejects only when `not allow_increase`, and
  falls through to entry sizing when `allow_increase=True`. The short-add is NOT
  misclassified as a reduction (`is_reduction` at :782-785 only matches SELL-vs-LONG
  or BUY-vs-SHORT), so it correctly routes to `resolve_entry`.
- **Long-arm byte-exactness.** The long INCREASE block (:586-600) and the
  `max_positions` block are untouched (`git diff` shows a single `@@ -535,24 +535,33`
  hunk). The `is not Side.BUY` SELL branch is self-contained and returns before any
  BUY-path code.
- **SMA_MACD oracle intact.** SMA_MACD is LONG_ONLY; the changed SHORT branch is
  unreachable for it. `tests/integration/test_backtest_oracle.py` — 3 passed.
- **Money discipline.** No float `==` on money in the new e2e tests; repr-exact
  `str(x) == str(expected)` assertions throughout. The two `==` Decimal comparisons
  in the unit test assert identity (unchanged available_cash), not computed arithmetic.
- **Indentation convention.** `admission_manager.py` changed lines are TABS; the e2e
  tests are 4-SPACE. Both correct per CLAUDE.md.
- **All 68 reviewed-file tests pass.**

The findings below are documentation/robustness defects, not logic bugs. The most
important is a stale method docstring that now contradicts the shipped behavior — in
a codebase where decision-anchored docstrings are explicitly load-bearing (CLAUDE.md:
"these tags are load-bearing references to planning artifacts").

## Warnings

### WR-01: `_enforce_position_admission` docstring contradicts the new shipped behavior

**File:** `itrader/order_handler/admission/admission_manager.py:511-516, 523`
**Issue:** The method-level docstring still describes the OLD behavior. Lines 511-516
state the short add "is out of v1 scope with the margin model (D-09, WR-01): it is an
AUDITED rejection ... mirroring the long INCREASE gate — it must NOT fall through to
first-entry sizing." Line 523 states "SELLs pass UNLESS they add to an open short (the
WR-01 gate above)." Both are now **false**: with `allow_increase=True` the short add is
ADMITTED and DOES fall through to entry sizing. The inline comment block (:537-549) was
correctly updated to SCALE-01 wording, but the docstring that a maintainer reads first
was not. Per CLAUDE.md these decision-anchored docstrings are load-bearing.
**Fix:** Update the bullet and the preserved-paths note to match the inline comment:
```python
		* OPEN short for the ticker — a BUY is a cover/exit (passes; sized
		  downstream). An unsized SELL is a same-side ADD (short increase),
		  now gated behind ``allow_increase`` (SCALE-01) byte-symmetrically with
		  the long INCREASE gate: ``allow_increase=False`` is an AUDITED rejection
		  (triggered_by=OrderTriggerSource.ADMISSION_INCREASE); ``allow_increase=True``
		  falls through to the SAME direction-agnostic resolve_entry sizing.
```
And amend line 523 to "SELLs pass UNLESS they add to an open short with
allow_increase=False".

### WR-02: Inline comment overstates the admission-side reservation for a SELL-add

**File:** `itrader/order_handler/admission/admission_manager.py:560-563`
**Issue:** The new comment claims the admitted SELL-add "reaches the SAME resolve_entry
+ check-and-reserve gate the long add uses (the reservation books notional/leverage on
the reserve side)." But the check-and-reserve gate reserves ONLY when
`primary.action is Side.BUY` (:264). A SELL-add books **NO** admission-side reservation
— and the test `test_allow_increase_true_sizes_short_increase_on_remaining_cash_and_reserves`
explicitly asserts `reserved_available == remaining` (unchanged) to prove exactly this.
The comment therefore contradicts both the code at :264 and the test that validates the
path, implying a reservation that never happens.
**Fix:** Correct the parenthetical to state the SELL-add books no admission reservation
(SELLs are exempt, :264 / D-03) and that its margin lock is a settlement-side concern
recomputed at `portfolio.py:423-441`:
```python
				# allow_increase=True: fall through to entry sizing. The SELL-add
				# is sized by resolve_entry but books NO admission-side reservation
				# (the reserve gate at :264 is BUY-only; a SELL credits cash) — the
				# margin LOCK rides settlement (portfolio.py:423-441 re-locks to
				# aggregate_notional / leverage).
```

### WR-03: Admitted short-increase has no admission-side solvency gate (asymmetric with the long arm)

**File:** `itrader/order_handler/admission/admission_manager.py:550-565`
**Issue:** When a long increase cannot be funded, the BUY-only check-and-reserve gate
(:264-301) produces an AUDITED `cash_reservation` REJECTED order at admission. An
admitted short-increase (SELL) is exempt from that gate, so an unfundable short-add is
caught only at settlement by `assert_lock_fits_buying_power` (`portfolio.py:439`), which
raises `InvalidTransactionError`. In backtest mode the error policy is fail-fast
(`EventHandler._on_handler_error` re-raises), so an unfundable short scale-in **aborts
the run** rather than producing an audited rejection — a different and harsher failure
mode than the long arm it claims to mirror "byte-symmetrically." This is not exercised
by the new e2e tests (both are funded). Note the change does not introduce this gap (it
is inherent to the SELL exemption), but the change is what first makes it reachable for
a same-side short add.
**Fix:** Either (a) document this asymmetry explicitly at the admit site so it is a known
deferred limitation rather than a silent gap, or (b) add an admission-side margin
solvency check for the admitted short-add mirroring the long reserve gate. At minimum,
add a test asserting the unfunded short-increase failure mode so the behavior is locked
rather than incidental.

## Info

### IN-01: `process_signal` step-0 docstring omits the short-increase case

**File:** `itrader/order_handler/admission/admission_manager.py:125-133`
**Issue:** The numbered step-0 summary in `process_signal` enumerates the LONG_ONLY
direction case, the `BUY-while-long allow_increase=False` increase case, and the
`max_positions` case, but does not mention the now-symmetric short SELL-add increase
case. Lower severity than WR-01 because this docstring is descriptive prose rather than
a per-branch contract, but it is incomplete after this change.
**Fix:** Add a clause: "an unsized SELL-add against an open short with
allow_increase=False is audited REJECTED (triggered_by=OrderTriggerSource.ADMISSION_INCREASE),
symmetric with the long arm."
