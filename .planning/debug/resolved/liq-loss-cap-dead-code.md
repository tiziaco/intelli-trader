## Symptoms

- **Expected behavior:** On a maintenance-margin breach, the forced-close settlement should guarantee the realized loss is floored so portfolio equity cannot drop below -WB (the DEF-01-C / D-07 guarantee the phase advertises), via the documented `min(realized_loss + penalty, WB)` clamp.
- **Actual behavior:** `_capped_realized_loss` (itrader/portfolio_handler/portfolio_handler.py:445) is invoked from **no production code path** — grep confirms it is called only from tests/unit/portfolio/test_liquidation.py:153,160. Production settlement runs through `_liquidate_position` (line 517) which sets `fill_price = liq_price` (line 550) and books the close via `portfolio.transact_shares` (line 714), realizing the **uncapped** loss. The loss is bounded only *by construction* because the fill is booked at the maintenance liq price (~80.81), not at the breaching bar close.
- **Error messages:** None — no test fails. The liquidation e2e leaves pass because the fill-at-liq-price construction bounds the loss near WB, so the dead clamp is never exercised and never needed for the crafted scenarios.
- **Timeline:** Introduced in Phase 04 plan 04-03 (commits bdb86e7 / 6d4c285 / 9966c85 / 130533a). The 04-03 SUMMARY and the phase verifier both credited the clamp with closing DEF-01-C.
- **Reproduction:** `grep -rn _capped_realized_loss itrader/ tests/ scripts/` shows test-only callers. A true gap-through bar (close gaps well below the isolated liq price) settled with an honest fill would book a loss far exceeding WB with no floor. Cross-val reference: scripts/crossval/liquidation_run.py gaps to 10 against an 80.808 floor (WB=4000) — an honest gap fill would book ~18,080 loss vs WB=4000.

## The Core Question (drives the fix)

Determine the INTENDED semantics from the planning artifacts (decision tags D-01-CORR, D-03-CORR, D-05, D-07, DEF-01-C; .planning/phases/04-liquidation-cross-validation-re-baseline/04-RESEARCH.md; 04-03-PLAN.md; 04-PATTERNS.md):

- **(a) fill-at-liq-price IS the deliberate loss-bounding mechanism** → then `_capped_realized_loss` is redundant dead code: delete it, correct the SUMMARY/docstrings/verifier claim that attribute the bound to the clamp, and document fill-at-liq-price as the actual mechanism. Add a gap-through test asserting fill-at-liq-price holds even when the bar gaps below the liq price (i.e. confirm the engine does NOT fill at the gapped close).
- **(b) the clamp is meant to be load-bearing** → then wire `_capped_realized_loss` into the production `_liquidate_position` settlement path, model a pessimistic gap-through fill (fill at the gapped bar close, not the liq price), apply the WB clamp there, and add a gap-through regression test that books the clamped loss.

Either resolution must keep SMA_MACD byte-exact (134 / 46189.87730727451, D-11) and the full suite + mypy --strict green.
