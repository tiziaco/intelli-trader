<objective>
Bound the live `OkxExchange` venue-correlation state so it no longer grows without limit
over a long session — resolving Phase 5 review warning **WR-05** for the common (fill) path,
with **zero backtest impact**.

Three locked requirements (from `05-SPEC.md`, narrow R1–R3 slice):
- **R1 — VenueCorrelationIndex encapsulation:** lift the four insert-only correlation
  structures (`_orders_by_venue_id`, `_venue_id_by_order_id`, `_orders_by_clOrdId`,
  `_seen_trade_ids`), the `_pending_fills_by_venue_id` late-fill buffer, and
  `_correlation_lock` out of `OkxExchange` into one cohesive, socket-free, unit-testable
  class exposing `register / resolve / adopt / release / mark_seen`; `OkxExchange` delegates.
- **R2 — release-on-terminal (fill-driven):** when a fill completes an order
  (cumulative filled reaches `order.quantity`), the index drops that order's
  venue-id / order-id / clOrdId entries and the now-empty pending buffer — draining any
  buffered late fill FIRST (per **WR05-D3**). The index self-releases via a per-`venue_id`
  cumulative-filled counter (per **WR05-D1**) — NOT coupled to `ReconcileManager`.
- **R3 — bounded trade-id dedup ring:** `_seen_trade_ids` becomes a `deque(maxlen=10000)`
  + companion `set` for O(1) membership (per **WR05-D2**), oldest id evicted past capacity;
  the durable `venue_trade_id` DB layer is the evicted-id backstop.

Purpose: WR-05 flagged the correlation maps + trade-id set as insert-only (nothing is ever
removed) → over a long live session every order and trade id is retained (unbounded memory).
This closes the fill-driven common path and bounds the ring.

Output: a new `VenueCorrelationIndex` class, a delegating `OkxExchange`, a symmetric
`release_venue_correlation` seam, and a socket-free unit-test suite — all live-only, oracle
byte-exact.

**Explicitly OUT of scope (→ future phase / R4):** non-fill terminals (cancel / expire /
reject-without-fill) mid-session and out-of-band venue changes have no mid-session terminal
signal today (`watch_orders` is log-only, `VenueReconciler` is startup-only) — their
correlation releases only at restart until R4. Do NOT plan a mid-session order-status signal.
</objective>

Existing seams to preserve / mirror (okx.py):

                                    return, writes the venue-id maps under lock, pops the pending buffer,
                                    then RE-DRAINS buffered trades OUTSIDE the lock via _handle_trade.
  adopt_venue_correlation(order)  — the INBOUND restart seam (VenueReconciler): repopulates the maps
                                    from a rehydrated Order + re-drains buffered fills outside the lock.
                                    `release_venue_correlation` is its symmetric OUTBOUND twin.

                                    then _emit_fill OUTSIDE the lock.

                                    (to_money(str(x))), commission abs()+None-guard, _ms_to_dt business
                                    time, and venue_trade_id; puts it on global_queue. Touches NO map,
                                    so it is the correct primitive to emit drained buffered fills.
