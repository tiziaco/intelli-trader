## Warnings

### WR-01: `_validate_transaction` classifies every transaction as a BUY (magnitude `> 0` is always true)

**✓ FIXED** (05.1 post-review follow-up, commit `26c9532f`, PR #76 / `b81d4384`) — `_validate_transaction` now gates the OPEN/GROW limit checks on `transaction.type == TransactionType.BUY` (`portfolio.py:757`), so the `max_positions` / `max_position_value` checks no longer run for closing SELLs and an exit at the position cap is not blocked. Original finding preserved below.

**File:** `itrader/portfolio_handler/portfolio.py:715-727` (esp. line 718)
**Issue:** `Transaction.quantity` is a positive magnitude everywhere (the fill
path builds it as `to_money(fill_event.quantity)` — never signed; direction is
carried by `transaction.type`). The guard `if transaction.quantity > 0:  # Buy
transaction` is therefore **always true**, so the buy-only limit checks
(`max_positions`, `max_position_value`) run for *every* transaction including
closing SELLs. When `config.validation.validate_transactions` is enabled, a
closing SELL issued while already at `max_positions` raises
`PortfolioError("Maximum positions limit reached")`, blocking an *exit* — and in
live mode that means `on_fill` raises, the fill never settles, and the engine
drifts (then halts). Currently oracle-dark only because `validate_transactions`
defaults off.
**Fix:**
```python
# Gate the OPEN/GROW limit checks on the transaction TYPE, not the magnitude.
if transaction.type == TransactionType.BUY:
    if self.n_open_positions >= self.config.limits.max_positions:
        raise PortfolioError(...)
    transaction_value = abs(transaction.quantity * transaction.price)
    if transaction_value > self.config.limits.max_position_value:
        raise PortfolioError(...)
```

### WR-02: spot `_stream_account` can clobber the derived position cache to flat and spuriously halt

**✓ FIXED** (05.1 post-review follow-up, commit `0f8011b5`, PR #76 / `b81d4384`) — new `_spot_positions_from_balance` (`venue.py:252-268`) returns `None` when the balance frame omits the base key, and `_write_balance_stream` writes only under the existing `if spot_positions is not None` guard (`venue.py:317-323`), so a partial/quote-only push leaves the prior derived holding intact and cannot trigger a spurious drift halt. Original finding preserved below.

**File:** `itrader/portfolio_handler/account/venue.py:215-239` (with `_extract_spot_position`, `190-211`)
**Issue:** On a spot leaf every `watch_balance` push recomputes positions via
`_extract_spot_position`, which returns `{}` whenever `total[BASE]` is missing
(or zero). Because that value is never `None` on the spot arm, line 238-239
**unconditionally overwrites** `self._venue_positions`. A balance frame that
omits the base currency (a partial/quote-only update) therefore resets the
derived holding to flat. The next engine-thread drift sweep (`_run_drift_sweep`,
no just-applied-fill band) then reads `engine_qty=0.5` vs `venue_qty=0` → beyond
band → unexplained → `halt("drift")`. This is mitigated *only* by ccxt.pro
internally merging deltas into a full balance object; it is not guaranteed by the
code under review, so a partial frame is a spurious-halt hazard.
**Fix:** Only replace the derived spot position when the frame actually carried
the base key; leave the prior cache intact otherwise:
```python
spot_positions = None
if self._market_type == "spot":
    total_map = update.get("total") if isinstance(update, dict) else None
    if isinstance(total_map, dict) and self._base in total_map:
        spot_positions = self._extract_spot_position(update)
# ... then keep the existing `if spot_positions is not None:` guard.
```

### WR-03: `_venue_available` is parsed and cached but never read; `available_balance` can overstate venue free funds

**✓ FIXED** (05.1 post-review follow-up, commit `196c1ee5`, PR #76 / `b81d4384`) — the reviewer's "drop the unused parse+field" option was taken: `_extract_balance` now returns only the settled `total` and documents "the venue `free` map is deliberately NOT parsed" (`venue.py:196-204`); the dead `_venue_available` field is gone. Admission (`available_balance`) nets solely against the settled `balance` minus the local pending overlay (Open Question 1 resolved as "ignore venue free"). Original finding preserved below.

**File:** `itrader/portfolio_handler/account/venue.py:128, 237, 302` (writes) vs `329-349` (`available_balance`)
**Issue:** `_extract_balance` parses the venue `free` map into `_venue_available`,
which is written in `__init__`, `_stream_account`, and `snapshot()` — but **no
reader exists**. `available_balance` computes `_venue_balance + _ledger_delta -
_pending_total()`, deliberately ignoring `_venue_available`. Consequently the
venue's own `used`/reserved funds (e.g. a pre-existing resting order at
startup, `used[USDC]=21000` in the spot fixture) are invisible to the admission
read: the engine reports full buying power the venue does not actually have free,
so an order can be admitted beyond venue-free cash. This is a dead-field code
smell *and* a real live buying-power over-statement. If the design intends to
ignore venue free (Open Question 1), the parse should be deleted; if not, the
admission read should account for venue-side reservations.
**Fix:** Either drop the unused parse+field (return only `balance` from
`_extract_balance`), or net venue `used` into `available_balance` at
reconcile/snapshot so admission cannot exceed venue-free funds.

### WR-04: session-start baseline guard halts with an off-vocabulary reason `'baseline-residual'`

**⚠ OPEN** — still `self.halt('baseline-residual')` at `live_trading_system.py:810`; the pinned vocabulary in `halt()`'s docstring (`:834-835`) and `core/enums/system.py:20-23` was NOT extended, so the off-vocabulary reason persists. Not tracked in `todos/pending/`. Original finding preserved below.

**File:** `itrader/trading_system/live_trading_system.py:626`
**Issue:** `halt()`'s contract (and `SystemStatus.HALTED`, `core/enums/system.py:20-23`)
pins the machine-readable halt reason to `{drift, reconciliation-unresolved,
connector-fatal, paused-on-disconnect}`. `_run_session_baseline_guard` calls
`self.halt('baseline-residual')`, a reason outside that set. Any operator/UI
consumer that switches on the documented vocabulary (surfaced via
`get_status()["halt_reason"]`) cannot classify this halt.
**Fix:** Use a documented reason (the residual is an unresolved reconcile
condition), or extend the pinned vocabulary in both `halt()`'s docstring and
`core/enums/system.py`:
```python
self.halt('reconciliation-unresolved')  # or add 'baseline-residual' to the pinned set
```

### WR-05: `_dispatch_live` drops risk-reducing CANCEL and protective bracket-child orders during pause/halt (tracked V17-11)

**✓ FIXED** (Phase 05.3 plan 05.3-06, D-14 / V17-11, commit `794ee75b`, PR #79 / `fe287f4b`) — `_dispatch_live` now branches by order KIND (`live_trading_system.py:1146-1178`): a CANCEL always passes the gate, a protective bracket-child (`parent_order_id` set) is deferred onto `_deferred_protective` and replayed on resume, and only fresh entries + SIGNALs stay suppressed. The RED gate `test_pause_defer_replay.py` is now green. Original finding preserved below.

**File:** `itrader/trading_system/live_trading_system.py:835-841`
**Issue:** While HALTED or paused-on-disconnect, `_dispatch_live` suppresses
**every** `SIGNAL`/`ORDER` event, including risk-*reducing* CANCELs and the
bracket-child (stop-loss / take-profit) orders a draining FILL generates. A
just-filled position is therefore left naked and stale resting orders cannot be
cancelled — a live-money risk. This is confirmed present and pinned by the RED
gate `tests/unit/trading_system/test_pause_defer_replay.py`, scheduled for the
D-14 defer+replay fix in Phase 05.3. Flagged for visibility: the current live
gate is unsafe until that lands.
**Fix (per D-14, 05.3):** defer protective orders onto a replay queue and replay
on resume; always let a CANCEL through immediately (it only reduces risk); keep
suppressing fresh entries.

### WR-06: VenueAccount push streams are bare `while True` loops with no supervisor (tracked V17-07)

**✓ FIXED** (Phase 05.3, D-11 / V17-07, PR #79 / `fe287f4b`) — both `_stream_account` and `_stream_positions` consume-loops are now wrapped in `_run_stream_supervisor` (`venue.py:356-413`), the bounded-retry reconnect supervisor mirroring the OKX donor: transients reconnect with backoff and an unclassified/fatal error (or exhausted ceiling) escalates via `_escalate_connector_halt` to a fail-safe `halt('connector-fatal')` instead of silently killing the writer. The RED gate `test_supervisor_catchall.py` is now green. Original finding preserved below.

**File:** `itrader/portfolio_handler/account/venue.py:215-239, 241-251`
**Issue:** `_stream_account` and `_stream_positions` are unguarded `while True`
loops. A single `ccxt.NetworkError` (or any exception) from `watch_balance` /
`watch_positions` kills the cache-writer coroutine outright and the venue
balance/position cache freezes forever with no reconnect and no halt — the engine
then trades / halts on stale venue truth without noticing. Confirmed present and
pinned by the RED gate `tests/unit/execution/test_supervisor_catchall.py`,
scheduled for the D-11 catch-all supervisor in Phase 05.3.
**Fix (per D-11, 05.3):** wrap both streams in the same bounded-retry
reconnect supervisor used by the OKX exchange/provider so a transient drop is
survived and an unclassified error escalates to a fail-safe halt.
