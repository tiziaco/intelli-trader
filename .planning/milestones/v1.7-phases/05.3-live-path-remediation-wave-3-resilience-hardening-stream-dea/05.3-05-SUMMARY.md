# Phase 05.3 Plan 05: Loop-Native Gap Backfill (D-17) Summary

**The mid-session candle-gap backfill is now loop-native: a gap detected on the connector loop thread hands its range to a dedicated coroutine spawned on that same running loop which `await`s the client `fetch_ohlcv` directly — killing the `connector.call(...).result()` self-deadlock (V17-15 / AUD-4c) that stalled the entire connector loop (and every stream sharing it) for 30s before livelocking.**

## Mechanism note (as requested by the plan output)

**Chosen: dedicated spawned coroutine on the loop** (not direct-await-in-place). `LiveBarFeed.update()` is synchronous and its candle-loop sink (`set_bar_sink(feed.update)`) is fixed at the composition root (outside this plan's files), so it cannot itself `await`. The loop-triggered gap branch therefore hands the range to `OkxDataProvider.spawn_gap_backfill(...)`, which schedules the backfill on the **running** connector loop via `loop.create_task(...)`. That coroutine `await`s the client fetch directly and, once resolved, replays the interior bars + delivers the trigger on the same loop thread. `connector.spawn` was deliberately NOT used (its cross-thread `ready.wait` would itself self-deadlock when driven from the loop thread), so **`connectors/okx.py` required no change** despite being listed in `files_modified`.

## Accomplishments
- `OkxDataProvider._fetch_ohlcv_backfill_async` — the byte-identical async twin of the synchronous `fetch_ohlcv_backfill`: same bounded/paginated shape (explicit `limit` per page, `since` advanced past the last bar), same Decimal edge (`to_money(str(...))`), same D-12 param-stamped routing keys — differing only in `await client.fetch_ohlcv(...)` in place of the blocking `connector.call(...)` bridge.
- `OkxDataProvider._rows_to_closed_bars` — extracted the raw-row→`ClosedBar` Decimal-edge builder so the sync and async backfills share one edge (no divergence).
- `OkxDataProvider.spawn_gap_backfill` + `_run_gap_backfill` + `_on_gap_backfill_done` — a supervised loop-native scheduler: `running-loop create_task` → await fetch → hand interior bars to the feed's replay callback; the done-callback observes the task's exception and escalates unknown failures to the injected halt entrypoint (D-11 shape, scrub-preserving: TYPE-only log, fixed `'connector-fatal'` reason).
- `LiveBarFeed._fill_gap_and_deliver` — the gap branch now dispatches: loop-triggered (running loop present + provider offers the seam) → `spawn_gap_backfill`; off-loop (engine-thread warmup-adjacent / reconnect-resume / offline `ReplayDataProvider`) → the unchanged synchronous `_backfill_gap` + `_deliver` (byte-identical to pre-D-17 for the socket-free unit matrix). Interior-then-trigger ordering + the CR-01 over-fetch clamp are preserved on both paths.
- **`warmup` untouched** (still the engine-thread synchronous `fetch_ohlcv_backfill`/`call()`); **`backfill_on_resume` stays dead/unwired**.

## Decisions Made
- Loop detection via `asyncio.get_running_loop()` is the branch key — it succeeds only on the connector loop thread (where `update()` runs synchronously inside the candle coroutine) and raises `RuntimeError` on the engine thread / offline synchronous providers.
- A `hasattr(provider, "spawn_gap_backfill")` capability check keeps any provider without the seam (e.g. the paper `ReplayDataProvider`, which also runs off-loop anyway) on the safe synchronous fallback.
