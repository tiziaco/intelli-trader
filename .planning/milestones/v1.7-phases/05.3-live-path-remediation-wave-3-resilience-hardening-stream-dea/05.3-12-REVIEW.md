## Summary

Reviewed only the plan-05.3-12 delta (git range `3a7a21d6..HEAD`): the `is_streaming_healthy()`
per-arm predicates on `OkxExchange` / `OkxDataProvider`, the `_all_venue_streams_healthy()`
compound resume gate wired into `_maybe_resume_after_reconnect`, and the loop-native gap-backfill
fail-loud + `_replaying_backfill` re-entrancy guard in `live_bar_feed.py`.

The resume-gate work (D-28/WR-03) is correct: the per-arm predicate is a GIL-atomic set-emptiness
read, the gate defaults absent arms to healthy, and the "do NOT re-set `_pending_stream_resume`"
decision is sound — I traced the re-fire path (`_on_stream_healthy` → `_on_stream_up` →
`_on_venue_stream_up` → `_pending_stream_resume.set()`) and confirmed a still-down arm re-drives the
gate on its next up-event, with the D-20 reconnect budget as the terminal escape (no silent
forever-pause). Error messages are secret-free (symbol + timeframe + integer ms only) and the
supervisor logs type-only — good.

The backfill fail-loud (D-29/WR-05), however, is **incomplete in exactly the direction the plan set
out to close**. The two new guards catch a page that starts late (wrong first bar) and a page with an
interior hole, but a page that is truncated at the **tail** (returns a contiguous prefix then stops
before `last_missing`) or returns **zero in-range bars** still silently swallows the gap through the
unconditional trailing `_deliver(t)`. That is a live-path data-loss defect — see CR-01.

## Critical Issues

### CR-01: Tail-truncated / empty under-returning backfill page silently swallows the gap

**Status:** RESOLVED (2026-07-06) — interior-completeness guard added after the replay loop and
before the trailing `_deliver(t)`: if `self._last_delivered.get((sym, tf_str)) != last_missing`, a
secret-free `MalformedDataError` (integer-ms last-delivered + last_ms) is raised so it escalates
through the same supervised-backfill error path. Driven by TDD — two RED tests added
(`test_tail_truncated_page_fails_loud_no_nested_spawn`, `test_empty_page_fails_loud_no_nested_spawn`)
confirmed failing first, then green after the fix; all 5 contiguity tests + 68 feed-suite tests pass.

**File:** `itrader/price_handler/feed/live_bar_feed.py:377-396`
**Issue:**
The new `_replay_and_deliver` fail-loud only fires in two shapes:
1. the first in-range bar's `ts != since_ms` (start-late page), and
2. `update()` re-enters the gap branch mid-replay because a bar exists *after* an interior hole
   (`_replaying_backfill` guard).

Both require a bar to be *present* to trip the check. But the terminal
`self._deliver(sym, tf_str, t, closed_bar)` (line 394) sets `L = t` **unconditionally**
(`_deliver` → `self._last_delivered[(sym, tf_str)] = t`, line 454) — it bypasses the gap check
entirely. So two under-return shapes escape fail-loud and silently drop interior bars:

- **Tail-truncated page.** Gap `t = L+3tf` (`first_missing=L+tf`, `last_missing=L+2tf`). Page returns
  only `[L+tf]` (omits the tail `L+2tf`). Replay: `update(L+tf)` delivers in-sequence (`L → L+tf`),
  loop ends with no gap raised (nothing after the hole to trip it), then `_deliver(t=L+3tf)` jumps
  `L` straight to `L+3tf`, silently skipping `L+2tf`.
- **Empty / no-in-range page.** `bars == []` (or all `< since_ms` / all `> last_ms`):
  `first_replayed` stays `False`, no `update()` runs, `_deliver(t)` jumps `L` across the whole
  interior. Silent swallow.

This is the same pagination-boundary mechanism the diff comment cites to justify the low clamp
("the real provider treats limit as a page size and can straddle the interior on either side") — a
page that can start *late* (Case A, explicitly handled) can equally end *early*. The head is
guarded; the symmetric tail is not. The 05.3-12 tests
(`test_live_bar_feed_backfill_contiguity.py`) cover only start-late (Case A) and interior-hole — no
tail-truncation or empty-page case — so the hole is untested. Net effect: on the live path an
under-returning venue page delivers the trigger bar with `L` advanced past never-backfilled bars,
feeding indicators a silently gap-swallowed series instead of failing loud to a connector halt —
defeating the stated D-29/WR-05 goal.

**Fix:** After the replay loop, before delivering `t`, assert the interior actually reached
`last_missing` (i.e. `L + tf == t`); raise otherwise so it escalates through the same supervised
backfill error path:
```python
            self.update(cb)
        # D-29 (WR-05): the interior must be fully contiguous up to last_missing
        # before t is delivered. A tail-truncated or empty page leaves L short;
        # _deliver would otherwise jump L past never-backfilled bars silently.
        if self._last_delivered.get((sym, tf_str)) != last_missing:
            raise MalformedDataError(
                f"gap-backfill:{sym}/{tf_str}",
                "under-returning backfill page "
                f"(interior incomplete: last-delivered={...}, expected last_missing={last_ms})")
        self._deliver(sym, tf_str, t, closed_bar)
    finally:
        self._replaying_backfill = False
```
(Use integer-ms coordinates in the message to keep it secret-free, matching the existing raise.)

## Warnings

### WR-01: `_replaying_backfill` is a non-atomic shared flag with no lock

**Status:** RESOLVED (2026-07-06) — documentation route only (per reviewer's fix option). The
single-loop-thread precondition is now pinned in two places: the flag declaration and the in-replay
guard in `update()`. Both note the flag encodes "THIS thread's replay is active" (lock-free-correct
only on the single connector-loop thread) and that the DEFERRED D-14 concurrent-bar path (05.3-06)
must scope re-entrancy to the replaying thread (thread-id / `threading.local`) BEFORE it is enabled.
No `threading.local` added now (deferred path not yet live) — comment only, no behavioral change.

**File:** `itrader/price_handler/feed/live_bar_feed.py:103, 377-396, 182-193`
**Issue:**
`_replaying_backfill` is a plain instance bool mutated on the connector-loop thread for the duration
of a replay and read by every `update()` call. The current loop-native path keeps replay and its
nested `update()`s on the *same* loop thread, so it is safe today. But `update()` is also reachable
from the **engine thread** (`warmup`, `backfill_on_resume`). The diff comment on `_fill_gap_and_deliver`
(lines 322-323) explicitly defers the concurrent-bar case to D-14 (05.3-06). If/when that path
lands, an engine-thread bar arriving mid-replay could observe `_replaying_backfill == True` and
spuriously raise `MalformedDataError` for a legitimate gap — escalating to a `connector-fatal` HALT.
The flag encodes "this thread's replay is active," not "any replay is active," yet it is process-wide
mutable state. Worth pinning now so the deferred concurrency work does not silently inherit a
false-positive halt.
**Fix:** Document the single-loop-thread precondition on the flag, or scope re-entrancy to the
replaying thread (e.g. `threading.local` / a thread-id stamp) before the D-14 concurrent-bar path is
enabled, so a concurrent engine-thread gap is not misclassified as an in-replay nested gap.

### WR-02: Interior-hole raise message omits the offending timestamp that the sibling raise includes

**Status:** RESOLVED (2026-07-06) — the in-progress-replay guard raise now appends the expected next
stamp `expected in-sequence L+tf=<int((last + tf) ms)>` alongside `got ts=<int>`, so both replay
raises carry {expected, got} integer-ms (secret-free) coordinates. Pure message change, no logic
change.

**File:** `itrader/price_handler/feed/live_bar_feed.py:189-193`
**Issue:**
The in-progress-replay guard raises `"gap detected during an in-progress backfill replay (expected
in-sequence L+tf, got ts=<t>)"` but does not include the last-delivered `L` (or `L+tf`), so an
operator triaging a `connector-fatal` halt sees the arriving `ts` without the expected next stamp —
they cannot tell *where* in the interior the hole is without cross-referencing logs. The sibling
under-returning-page raise (lines 388-391) includes both `since_ms` and `ts`. Minor diagnosability
asymmetry on an error that terminates the engine.
**Fix:** Include the expected next stamp in the message, e.g. append
`expected=<int(last.value // _NS_PER_MS) + tf_ms>`, keeping it integer-ms (secret-free). Both raises
should carry {expected, got} coordinates.
