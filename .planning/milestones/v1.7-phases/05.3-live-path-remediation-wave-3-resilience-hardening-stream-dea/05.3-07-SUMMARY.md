## Decisions Made
- **D-19 seam = option (a) shared-connection, NOT option (b) persist_fill.** The durable position write originates deep in `transact_shares → position_manager.process_position_update → set_position` (both `position_manager.py` and `sql_storage.py` are outside this plan's file scope). A `persist_fill(...)` method would have to receive the affected `Position` threaded back out of `transact_shares`; the shared-connection `fill_transaction()` context instead reuses the existing `set_position` call site — `set_position`/`save_account_state` transparently route onto `_fill_conn` when a fill txn is active. This keeps the change inside the three allowed files and preserves store-first ordering within the single transaction.
- Dedup mark placed after the atomic persist so a rolled-back fill records no key.
