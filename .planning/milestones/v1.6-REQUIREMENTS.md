# Requirements: iTrader v1.6 — N+3b Persistence Foundation

**Defined:** 2026-06-27
**Core Value:** A single backtest run of `SMA_MACD` on the golden BTCUSD CSV produces correct,
deterministic, cross-validated numbers. v1.6 adds a durable-storage + caching foundation **without
disturbing that** — the backtest path stays byte-exact and N+4 Live Trading inherits a persistent,
restart-safe system of record.

**Scope source:** `.planning/notes/persistence-milestone-design.md` (converged seed) + the four research
docs in `.planning/research/` (Q1–Q10 resolved, `e4ad7c9`) + the Owner Decisions locked 2026-06-27 (see
PROJECT.md → Current Milestone). Where the seed and Owner Decisions differ, the Owner Decisions win.

## v1 Requirements

Requirements for this milestone. Each maps to exactly one roadmap phase.

### SPINE — Swappable SQL interface (the spine)

- [x] **SPINE-01**: A single `SqlBackend` + `SqlSettings` selects the SQL driver by config, not code —
  SQLite for the research store, Postgres for the operational store; the interface is shaped so a
  Turso/libSQL driver can be slotted in later without changing storage code (driver NOT added this
  milestone).
- [x] **SPINE-02**: The three existing domain storage ABCs (`OrderStorage`, `PortfolioStateStorage`,
  strategy `SignalStore`) plus a new `ResultsStore` ABC are each implemented by exactly one
  `Sql<Concern>Storage` class that **composes** the shared `SqlBackend` — one class per concern, no
  cross-concern god base.
- [x] **SPINE-03**: UUIDv7 ids and business-time timestamps persist and round-trip **losslessly**
  through the SQL layer on both SQLite and Postgres — single UUIDv7 scheme and determinism preserved, no
  wall-clock writes, no second ID scheme.

### RESULT — Research/optimization results store (#1)

- [x] **RESULT-01**: Every backtest/optimization run persists a `runs` row (summary metrics as `Float`
  columns + a JSON settings column) so a full parameter sweep is never held in memory.
- [x] **RESULT-02**: Each run's equity-curve / trade-log frame persists as a `run_artifacts`
  JSON/gzip'd-text column (**no Parquet, no `pyarrow`**) and round-trips back to a pandas DataFrame.
- [x] **RESULT-03**: The results store runs on **SQLite** by default (schema via `create_all()`,
  ephemeral) and supports the cross-run query surface (e.g. top-N runs by a summary metric); the `runs`
  schema is Optuna-FK-ready (nullable study/trial ids) without building the sweep loop.
- [x] **RESULT-04**: Results-store reads/writes are validated by DB round-trip tests on an in-process
  SQLite database (write → read → assert equality).

### OPS — Operational store backends (#2, store layer)

- [x] **OPS-01**: `SqlOrderStorage` implements the `OrderStorage` ABC on Postgres — filling the
  `PostgreSQLOrderStorage` `NotImplementedError` stub — selectable via `OrderStorageFactory`
  (`in_memory` backtest / `postgresql` live; backtest backend unchanged).
- [x] **OPS-02**: A new `SqlPortfolioStateStorage` implements `PortfolioStateStorage` on Postgres
  (cash / position / transaction / metrics state), selectable via `PortfolioStateStorageFactory`.
- [x] **OPS-03**: A new `SqlSignalStorage` implements the strategy `SignalStore` ABC on Postgres
  (signal records + config snapshots), selectable via `SignalStorageFactory`.
- [x] **OPS-04**: Operational money persists as Postgres-native `Numeric` (Decimal end-to-end on the
  real-money path — no float-for-money, no `DecimalAsText` needed); validated by DB round-trip tests on
  a **testcontainers** Postgres.

### RETAIN — Two-knob write-through + retention model

- [x] **RETAIN-01**: A mode-aware write-through toggle — backtest = retain-all in-memory + optional
  end-of-run batch dump (**write-through OFF**: the backtest backend contains no per-tick serialization
  code); live = write-through ON to the Postgres system of record.
- [x] **RETAIN-02**: A bounded live working-set cache — open positions, working orders + brackets, the
  current account/portfolio snapshot, and running accumulators stay resident; terminal records (closed
  positions, terminal orders, full history) are purged on terminalize with a bounded read-through
  fallback, so the working set stays bounded in a long-running process.
- [x] **RETAIN-03**: Restart rehydration reconstructs the live working set (open positions + working
  orders + brackets) from the store — store is the system of record, cache is rebuildable —
  deterministically, without replaying terminal history; a bracket parent stays resident until its
  children terminalize. (Built + integration-tested here on testcontainers Postgres; driven by a real
  live feed in N+4.)

### CACHE — Cache inventory & classification (#3)

- [x] **CACHE-01**: Every ad-hoc cache / `lru_cache` / scattered in-memory lookup across `itrader/` is
  inventoried and classified into (a) hot-path data cache, (b) storage-index lookup already solved by
  v1.5 secondary indexes, or (c) legitimate pure-function memoization — delivered as a documented
  classification + routing, **not a rewrite or a unification**.
- [x] **CACHE-02**: The v1.5 hot path (stateful indicators / shared recent-bars feed) is left
  unchanged — any columnar/Arrow idea is rejected unless it lives strictly at the serialization boundary
  and does not regress the W1/W2 baseline (research Q7).

### MIG — Migrations

- [x] **MIG-01**: The live Postgres operational store has Alembic-managed schema migrations (one chain);
  the ephemeral research store uses `create_all()` (no Alembic).

### SEC — Security hardening (FL-06)

- [x] **SEC-01**: `SqlHandler` (`itrader/price_handler/store/sql_store.py`) uses parameterized queries +
  safe quoted identifiers and sources credentials from settings/secrets — no hardcoded credentials
  (L17), no f-string DDL / `DROP TABLE` injection (L35), no unsanitized symbol-as-table-name
  (L56/58/69).

### GATE — Correctness discipline (DB-gated milestone)

- [x] **GATE-01**: With write-through off, the SMA_MACD backtest oracle stays **byte-exact** (134 /
  `46189.87730727451`) and there is **no W1/W2 perf regression** vs the v1.5 frozen baseline
  (15.7 s / 152.8 MB) — proving the persistence layer is inert on the hot path.
- [x] **GATE-02**: The new persistence code is covered by DB round-trip + restart-rehydration tests on
  the right substrate (in-process SQLite for the results store; testcontainers Postgres for the
  operational store), with `mypy --strict` clean and `filterwarnings=["error"]` green.

## v2 Requirements

Deferred to future milestones. Tracked, not in this roadmap.

### Optimization loop (future milestone)

- **OPT-01**: Optuna sampler + search-space definition + trial orchestration that writes runs into the
  results store (this milestone ships only the substrate the loop writes to).

### Turso/libSQL (opt-in, future)

- **TURSO-01**: A `sqlalchemy-libsql` driver as an opt-in research-store backend, once the driver
  matures past beta and a measured perf advantage over plain SQLite is demonstrated for our workload.

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Optimization / parameter-sweep loop (Optuna sampler) | Substrate only — build the store now, the loop is a later milestone (v2 OPT-01) |
| Turso/libSQL driver this milestone | Perf premise didn't hold for our batch-dump workload + beta/stale driver; interface stays Turso-ready, driver deferred (v2 TURSO-01) |
| Parquet / `pyarrow` frame storage | Results store is all-`Float` (analytical) — JSON/gzip'd-text column is simpler, all-SQL, no extra dependency |
| `DecimalAsText` TypeDecorator | Unneeded — money never lands on a SQLite-family backend (results = `Float`, operational = Postgres-native `Numeric`) |
| SQLite operational backend | Operational store is Postgres-only; backtest keeps the in-memory backend |
| Exploded per-bar equity rows (row-per-bar tables) | Row-count blowup at sweep scale (the seed's stated objection) — blob-per-run + DuckDB-over-exports for the rare cross-run query |
| Live data feed / streaming / live execution loop / venue reconciliation | N+4 Live Trading (Backlog 999.3) — the operational store is built + tested here, driven live in N+4 |
| Float-for-money on the operational (real-money) path | Locked money policy — Decimal end-to-end stays on the live store (Postgres-native `Numeric`) |
