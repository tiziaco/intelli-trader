# Phase 05.1: Short Position Scale-In (Margin Increase) - Context

**Gathered:** 2026-06-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Let a strategy **INCREASE an open SHORT** (a same-side SELL add) exactly the way it
increases a long today ‚Äî by lifting the currently-unconditional short-increase admission
rejection (`admission_manager.py:537-556`) behind the strategy `allow_increase` flag,
mirroring the long INCREASE gate (`admission_manager.py:577-591`).

The admitted add reuses the EXISTING machinery end-to-end ‚Äî `SizingResolver.resolve_entry`
for sizing and the side-agnostic margin SCALE-IN branch (`portfolio.py:423-441`, recompute
the lock to `aggregate_notional / leverage`) for settlement. **No new sizing function, no
new settlement branch, no direction-specific logic.**

**Explicitly OUT of scope:** flip/split settlement (a single reducing fill that crosses
zero). The CR-02 over-cover crossing-fill guard (`portfolio.py:399-404`) stays a fail-loud
fence, locked by a regression test asserting it still raises.

This is an **owner-gated, result-changing** phase: it carries its OWN cross-validation
re-baseline (same discipline as Phases 2-4), sequenced after Phase 5 (trailing stops) and
BEFORE the Phase 6 capstone so the re-baselined numbers land before the flagship consumes
them.

**Scope note ‚Äî already done (NOT this phase):** WR-01 (settlement-side solvency,
`assert_lock_fits_buying_power`) and WR-04 (assert-BEFORE-release call order) landed in
Phase 4 and are present at `portfolio.py:439-441` / `:460-464`. The source todo lists them
as blockers, but they are resolved.

<decisions>
## Implementation Decisions

### Increase-flag semantics
- **D-01 ‚Äî shared `allow_increase` flag (no new flag).** A short ADD is gated by the SAME
  `signal_event.allow_increase` field that gates a long ADD ‚Äî exactly mirroring the long
  gate at `admission_manager.py:579`. Zero new API surface; matches the roadmap wording
  ("behind the strategy `allow_increase` flag"). A strategy that opts into scaling opts in
  for whichever side it is trading (direction is already scoped by
  `LONG_ONLY`/`SHORT_ONLY`/`LONG_SHORT` registration). NOT a dedicated `allow_short_increase`
  flag ‚Äî no requirement asks for independent long/short add control, and a separate
  default-off flag would be new surface to thread for no behavioral gain.

### Short-add sizing basis
- **D-02 ‚Äî direction-agnostic `resolve_entry`; NO short-specific sizing path.** A short add
  is a policy-declared entry resolved by the SAME `resolve_entry` dispatch
  (`sizing_resolver.py:75-145`) as any other entry. Direction is NOT a sizing input ‚Äî it is
  the settlement layer's concern. This preserves the D-01 one-resolver / `assert_never`
  exhaustive-dispatch invariant and the Phase-3 **D-05 netting principle** (order effect
  derived from side-vs-position; sizing resolved direction-agnostically).
- **D-03 ‚Äî the cash-vs-margin "basis" choice is already the strategy's, via policy.** The
  engine ALREADY separates direction from sizing basis: `FractionOfCash` sizes off live
  `available_cash`; `LeveredFraction` sizes notional off `total_equity()` and NEVER touches
  cash (the margin-aware arm, D-07/D-12/LEV-02). A spot/collateralized short declares
  `FractionOfCash`; a levered short declares `LeveredFraction`. The phase introduces no new
  basis ‚Äî it lets short adds reach the basis the strategy already declared. A bespoke
  "short notional/headroom" branch was REJECTED as strictly less correct (forks sizing by
  direction, breaks the one-resolver invariant, duplicates `LeveredFraction`).
- **D-04 ‚Äî `FractionOfCash` "fraction-of-remaining-cash" compounding is accepted as-is
  (criterion 4a confirmed).** Each `FractionOfCash` add takes its fraction of CURRENT
  remaining cash (compounding), a property of the arm reading live `available_cash`
  (`sizing_resolver.py:115`). This already governs LONG adds today; it is NOT short-specific
  and NOT introduced by this phase. A strategy wanting non-compounding margin-based sizing
  declares `LeveredFraction` instead.

### Unaffordable-add behavior
- **D-05 ‚Äî audited admission rejection, mirroring the long path.** An admitted short add
  falls through to `resolve_entry` and the check-and-reserve reservation gate; if the
  recomputed lock does not fit buying power, it is an AUDITED rejection at admission ‚Äî
  identical to an unaffordable long add (`admission_manager.py:588-591`, "the
  check-and-reserve gate covers the cash check"). `assert_lock_fits_buying_power`
  (`portfolio.py:439`) stays a fail-loud SETTLEMENT backstop that should never fire on this
  path. NOT clamp-to-fit (would diverge from the long path and hide the strategy's intended
  size).

### Reservation-side correctness (criterion 4b ‚Äî verify in planning, not a sizing choice)
- **D-06 ‚Äî confirm the SELL-add reserves the correct side/amount at admission.** Planning
  MUST verify the margin reservation for a SELL-add reserves `notional / leverage` on the
  reserve side at admission (the check-and-reserve gate), not only the settlement-side
  `assert_lock_fits_buying_power` invariant. This is a correctness check on the existing
  reservation path, not a new design decision.

### Flip/split ‚Äî out of scope, locked by regression
- **D-07 ‚Äî over-cover guard stays a fail-loud fence (criterion 3).** The CR-02 crossing-fill
  guard (`portfolio.py:399-404`) still raises on a reducing fill whose quantity exceeds the
  open position. A regression test asserts it raises. Single-order flips remain a deferred
  explicit close-then-open split (Phase-3 D-06 lineage).

### Proof / test breadth
- **D-08 ‚Äî focused scale-in suite + mandatory guards.**
  - **Unit:** with `allow_increase=True`, a same-side SELL against an open SHORT is ADMITTED
    and sized through `resolve_entry`; with `allow_increase=False` it remains an AUDITED
    `ADMISSION_INCREASE` rejection (today's behavior preserved).
  - **Regression (criterion 3):** the over-cover crossing fill still raises (`portfolio.py:399-404`).
  - **Parked & cross-validated for the re-baseline:** a short scale-in (‚â•2 SELL adds ‚Üí
    aggregate-notional re-lock through `portfolio.py:423-441`) and a
    scale-in-then-partial-cover round-trip. Mirrors the Phase-3 parked-scenario discipline.
  - NOT the minimal "gate units + flip-guard only" set (too thin for a result-changing
    owner-gated re-baseline); NOT the broad levered + 3+-stacked-add matrix (over-invests in
    oracle-dark paths the Phase-6 flagship will exercise for real).

### Claude's / Planner's Discretion
- Exact code placement of the short-increase `allow_increase` check ‚Äî whether to handle it
  inline at the WR-01 short-add site (`admission_manager.py:537-556`) or fold it into the
  side-dispatch block alongside the long gate (`:577-592`) so the two read symmetrically.
  The long path MUST stay byte-exact either way.
- The specific BTCUSD instrument parameters (leverage, borrow_rate) used for the parked
  short-scale-in scenarios (oracle-dark; realistic crypto defaults).
- Plan decomposition / sequencing of the gate-lift vs. the regression test vs. the
  cross-validation re-baseline.
- The cross-validation harness wiring for the scale-in scenario against backtesting.py
  0.6.5 / backtrader 1.9.78.123 (criterion 5).
- Indentation: tabs in `order_handler/`, `portfolio_handler/`; 4 spaces in `core/`,
  `config/` ‚Äî match the file.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **The margin SCALE-IN settlement branch is already side-agnostic** (`portfolio.py:423-441`):
  `is_increase` routes to recompute `new_lock = position.aggregate_notional / leverage`;
  nothing in it inspects long-vs-short. A short add settles through it with NO change.
- **`resolve_entry` is direction-agnostic and policy-dispatched** (`sizing_resolver.py:75-145`,
  `assert_never`-closed). The short add reaches it unchanged via the `allow_increase`
  fall-through.
- **`LeveredFraction`** (`sizing_resolver.py:129-137`) already provides margin/equity-based
  notional sizing for strategies that want it ‚Äî the "size off margin headroom" path exists.
- **The check-and-reserve reservation gate** at admission already produces audited rejections
  for unaffordable longs; the short add reuses it (D-05).

### Established Patterns
- **Netting / signed-Position model (Phase-3 D-05):** order effect = (order side vs position
  side); the cover-arm fix already generalized exit detection. The short-increase admission is
  the symmetric counterpart ‚Äî same `allow_increase` gate, no direction branch in sizing.
- **Byte-exact gate via mirroring:** the long INCREASE path must stay byte-exact; the short
  gate is added alongside it without altering the long arm.
- **Parked hand-verified scenarios, frozen only under owner sign-off (Phase-3 D-10):** D-08
  follows this ‚Äî park the scale-in scenarios, cross-validate, owner-gate the freeze.

### Integration Points
- `_enforce_*` admission flow in `admission_manager.py` ‚Äî the short-add site (`:537-556`)
  gains the `allow_increase` check; the SELL-add then falls through to the same
  `resolve_entry` + reservation gate the long add uses.
- `portfolio.py` SCALE-IN branch consumes the resulting fill unchanged (aggregate-notional
  re-lock).

## Specific Ideas

- The user pushed (again, as in Phase 3) on **architectural correctness over expedience** ‚Äî
  asking "what's the most correct thing to do here?" on sizing. This drove D-02/D-03: the
  answer is to add ZERO short-specific sizing and rely on the existing direction-agnostic
  policy dispatch, with `LeveredFraction` as the already-built margin-aware basis. The
  earlier "reuse cash vs. add a notional branch" framing was corrected as a false dichotomy.
