## Short-add sizing basis

| Option | Description | Selected |
|--------|-------------|----------|
| Direction-agnostic `resolve_entry`, no short-specific path | Short add resolved by the same policy dispatch as any entry; `FractionOfCash`/`LeveredFraction` already cover cash vs. margin basis; preserves one-resolver invariant + Phase-3 D-05 netting | ✓ |
| Short-specific notional/headroom branch | Bespoke short sizing arm; forks sizing by direction, breaks exhaustive dispatch, duplicates `LeveredFraction` | |

**User's choice:** Direction-agnostic `resolve_entry`, no short-specific path (Recommended)
**Notes:** User asked "architecturally speaking, what's the most correct thing to do here?" Tracing `sizing_resolver.py` revealed the original cash-vs-notional framing was a false dichotomy — the engine ALREADY separates direction from sizing basis. `FractionOfCash` (cash) and `LeveredFraction` (equity/notional, margin-aware) are both built; the strategy declares which. The most correct stance is to add zero short-specific sizing and let the short add reach the strategy's declared policy via the existing direction-agnostic dispatch. Criterion 4(a)'s "fraction-of-remaining-cash" wrinkle is a `FractionOfCash` property that already governs long adds, not a short concern. Criterion 4(b) reservation-side correctness is a planning verification, not a sizing-design choice.
